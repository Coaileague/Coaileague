# CoAIleague — DockChat Architecture, Shift Rooms, Proof of Service & System Bots
## "Never Lose a Photo, Message, or Proof of Service Again"

**Date:** March 4, 2026
**Context:** Security companies currently rely on WhatsApp groups for field communication. Photos fill phones, auto-save gets disabled, proof of service vanishes, client payment disputes become he-said-she-said, and wage claims lack documentation. This spec solves ALL of it by making DockChat the single source of truth for every field operation.

**THE CORE PROMISE:**
Every photo, every message, every report, every clock event — captured, timestamped, GPS-tagged, organized by shift, compiled into proof-of-service PDFs, and stored permanently in the org's document vault. Nothing gets lost. Everything is evidence.

# PART 1: SHIFT ROOMS — AUTO-CREATED GROUP DMS
# ═══════════════════════════════════════════════════

## 1A: What Shift Rooms Are

A Shift Room is an auto-created DockChat group conversation tied to a specific shift at a specific site. It exists for the duration of the shift and serves as the operational communication channel for everyone involved with that shift.

**Shift Room = Site + Date + Shift Time**

If Pinnacle Tower has a Day Shift (8AM-4PM) on March 4, there is ONE Shift Room for that shift. If three officers are assigned, all three are in the room. If one officer is on a solo overnight shift, the room has one officer plus system bots.

## 1B: Auto-Creation Triggers

Shift Rooms are created automatically. Officers never need to manually create them.

```
TRIGGER 1 — SHIFT START TIME:
  System clock hits shift start time → Shift Room auto-created
  All assigned officers auto-added to room
  System bots auto-joined (ClockBot, ReportBot)
  Room name: "[Site Name] — [Shift Type] — [Date]"
  Example: "Pinnacle Tower — Overnight — Mar 4, 2026"

TRIGGER 2 — FIRST CLOCK-IN:
  If Trigger 1 hasn't fired yet (edge case), first officer
  clocking in at the site triggers room creation
  Remaining assigned officers added when they clock in

TRIGGER 3 — MANUAL CREATION (Manager):
  Manager can create a Shift Room early (for pre-shift briefing)
  or create ad-hoc rooms for special events
```

## 1C: Who Gets Access (RBAC)

```
ROLE                    AUTO-JOIN?    CAN JOIN?    CAN SEE HISTORY?
─────────────────────────────────────────────────────────────────────
Assigned officers       ✅ Auto        —            ✅ Own shifts only
Site supervisor         ✅ Auto        —            ✅ All shifts at their sites
Manager                 ❌ No          ✅ Can join   ✅ All shifts in org
Org owner               ❌ No          ✅ Can join   ✅ All shifts in org
Other officers          ❌ No          ❌ Cannot     ❌ Cannot
(not assigned)
System bots             ✅ Auto        —            ✅ All rooms they join
Support agents          ❌ No          ✅ Can join   ✅ When invited
Client contacts         ❌ No          ❌ Cannot     ❌ Via client portal only
(client portal)                                       (sees proof-of-service PDF)
```

**KEY: Officers CANNOT see other officers' shift rooms unless they are assigned to that shift.** This prevents information leaking between sites and protects client confidentiality.

## 1D: Shift Room Lifecycle

```
TIMELINE:

  30min before shift ─── Room created, pre-shift message posted:
                         "Shift Room opened for Pinnacle Tower Overnight.
                          Assigned: Marcus Rivera, Angel Jackson.
                          Post orders: [link]. Start time: 12:00 AM."

  Shift starts ───────── ClockBot monitors for clock-ins
                         Officers post photos, updates, incidents
                         Every message auto-tagged with timestamp + GPS

  During shift ────────── All content captured:
                          - Text messages (preserved)
                          - Photos (stored permanently in media vault)
                          - Voice messages (transcribed + stored)
                          - Location check-ins (GPS pins)
                          - Incident reports (linked to room)
                          - Daily reports (linked to room)

  End of shift ────────── Auto-compilation trigger:
                          1. ClockBot verifies all officers clocked out
                          2. ReportBot compiles Proof of Service PDF
                          3. PDF includes ALL content chronologically
                          4. PDF sent to manager for review
                          5. PDF stored in document vault
                          6. Room marked as "archived" (read-only)

  Post-shift ──────────── Room stays accessible (read-only) forever
                          Searchable, downloadable, printable
                          Evidence for disputes, claims, audits
```

## 1E: What Gets Captured in Shift Rooms

EVERY message in a Shift Room is permanently stored with metadata:

```

  id: UUID,
  room_id: UUID (shift room),
  shift_id: UUID,
  sender_id: UUID (officer or bot),
  message_type: "text" | "photo" | "voice" | "location" | "file" | "system",
  
  // Content
  text_content: "Completed patrol of Level 3, all clear",
  
  // Media (if photo/voice/file)
  media_url: "https://storage.../shift-photos/2026-03-04/pinnacle-overnight/img_001.jpg",
  media_thumbnail_url: "https://storage.../thumbs/...",
  media_mime_type: "image/jpeg",
  media_size_bytes: 2048000,
  
  // Voice specific
  voice_duration_seconds: 15,
  voice_transcript: "Just completed the Level 3 patrol, everything looks good...",
  
  // CRITICAL METADATA — always captured
  timestamp: "2026-03-04T02:15:33Z",
  gps_latitude: 32.7767,
  gps_longitude: -96.7970,
  gps_accuracy_meters: 8.5,
  gps_address: "123 Commerce St, Level 3, Pinnacle Tower",
  device_info: "Samsung Galaxy S24, Android 15",

```

## 1F: Photo Storage — Never Lost Again

```
PHOTO LIFECYCLE:

  Officer takes photo in Shift Room
       │
       ▼
  Photo uploaded to permanent cloud storage (S3/R2/Cloudflare)
  NOT stored on device (device storage is unreliable)
       │
       ▼
  Metadata auto-attached:
    - GPS coordinates at time of capture
    - Timestamp (server-verified, not device clock)
    - Officer who took it
    - Shift room it belongs to
    - Site and client it relates to
       │
       ▼
  Thumbnail generated for chat display
  Full-resolution original preserved
       │
       ▼
  Photo appears in:
    1. Shift Room chat (inline)
    2. Shift media gallery (all photos from this shift)
    3. Proof of Service PDF (chronological)
    4. Site media archive (all photos ever taken at this site)
    5. Document vault (searchable by date, site, officer)

  RETENTION: Photos are NEVER auto-deleted.
  Only org owner can delete, and deletion is soft-delete
  (marked as removed but recoverable for 90 days).
```

## 2A: What It Is

At the end of every shift, the system automatically compiles everything from the Shift Room into a professional Proof of Service PDF. This document proves:

1. Officers were on site (GPS + photos + clock records)
2. What they did (patrol logs, check-ins, incident responses)
3. When they did it (timestamped chronologically, every hour verified)
4. What they found (photos with GPS, incident reports)
5. Who was there (officer names, badge numbers, clock in/out times)

**This PDF is the document you hand to a client who claims "your guards didn't show up" or "nothing was done." It's also the document you submit to TWC when an employee files a wage claim.**

## 2C: Hourly Check-In Requirement

For sites that require hourly proof of service, the system enforces check-ins:

```
HOURLY CHECK-IN FLOW:

  Every hour (configurable per site: 30min, 1hr, 2hr intervals):
    1. ClockBot posts in Shift Room: "📸 Hourly check-in due.
       Please send a photo of your current location."
    2. Timer starts (15 minute grace period)
    3. Officer sends photo → GPS + timestamp captured → check-in recorded ✅
    4. If NO photo within 15 minutes:
       - ClockBot DMs officer: "Reminder: hourly check-in overdue"
    5. If NO photo within 30 minutes:
       - ClockBot notifies manager/supervisor
       - Merit point deducted from officer's behavior score
       - Logged as "Missed Check-In" on proof of service PDF
    6. Officer can also send text update + photo at any time
       (voluntary updates also count as check-ins)

  SITE CONFIGURATION:
    - check_in_interval: 60 (minutes)
    - check_in_requires_photo: true
    - check_in_grace_period: 15 (minutes)
    - check_in_escalation_delay: 30 (minutes)
    - auto_notify_manager_on_miss: true
```

```sql
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  site_id UUID REFERENCES clients(id),          -- Which client site
  
  -- Room identity
  room_name VARCHAR(300) NOT NULL,
  room_type VARCHAR(30) NOT NULL DEFAULT 'shift',

  -- Timing

  shift_start_time TIMESTAMP NOT NULL,
  shift_end_time TIMESTAMP NOT NULL,
  
  -- DockChat integration
  dockchat_channel_id UUID,                      -- Links to existing DockChat channel/room
  
  -- Status

  -- Status: pending, active, compiling, archived

  closed_at TIMESTAMP,
  archived_at TIMESTAMP,
  
  -- Proof of service
  proof_of_service_pdf_url VARCHAR(500),
  proof_of_service_generated_at TIMESTAMP,
  proof_of_service_status VARCHAR(20) DEFAULT 'pending',
  -- Status: pending, generating, generated, reviewed, sent_to_client

  check_in_interval_minutes INTEGER DEFAULT 60,
  check_in_requires_photo BOOLEAN DEFAULT true,
  
  -- Stats (updated in real-time)
  total_messages INTEGER DEFAULT 0,
  total_photos INTEGER DEFAULT 0,
  total_check_ins INTEGER DEFAULT 0,
  missed_check_ins INTEGER DEFAULT 0,
  incidents_reported INTEGER DEFAULT 0,
  
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_shift_rooms_workspace ON shift_rooms(workspace_id, shift_date);
CREATE INDEX idx_shift_rooms_site ON shift_rooms(site_id, shift_date);
CREATE INDEX idx_shift_rooms_status ON shift_rooms(status);

-- ═══════════════════════════════════════════════════
-- SHIFT ROOM MEMBERS
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS shift_room_members (

CREATE INDEX idx_room_members ON shift_room_members(room_id, is_active);
CREATE INDEX idx_room_members_user ON shift_room_members(user_id, is_active);

-- SHIFT ROOM MESSAGES (Extended from DockChat)
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS shift_room_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  room_id UUID NOT NULL REFERENCES shift_rooms(id),
  sender_id UUID REFERENCES users(id),           -- NULL for system messages
  sender_bot_type VARCHAR(30),                   -- If sent by a bot
  
  -- Content
  message_type VARCHAR(20) NOT NULL,
  -- Types: text, photo, voice, video, location, file, system,
  --        check_in, incident_link, report_link, bot_prompt, bot_analysis

  -- Media
  media_url VARCHAR(500),
  media_thumbnail_url VARCHAR(500),

  -- Relationships
```

## 3A: Bot Architecture

System bots are specialized AI agents powered by Trinity's layered intelligence (Gemini Flash for quick analysis, Claude Sonnet for complex reasoning, OpenAI as fallback). Each bot has:

1. **Specialization** — trained focus area with specific system prompts
2. **Triggers** — events that summon the bot or cause it to act
3. **Capabilities** — what the bot can read, write, and create
4. **Output** — what the bot produces (PDFs, alerts, messages, pipeline items)

```
BOT HIERARCHY:

  TRINITY (Orchestrator)
    │
    ├── ClockBot        — Workforce time & attendance specialist
    │
    ├── ReportBot       — Incident/daily report writing specialist
    │
    ├── MeetingBot      — Meeting analysis & minutes specialist
    │
    ├── HelpBot         — Employee support & helpdesk specialist (already exists)
    │
    └── GuardBot        — Real-time shift monitoring & compliance specialist

  Trinity summons bots based on events.
  Bots can also self-activate when they detect relevant activity.
  All bot actions consume org credits.
```

## 3B: ClockBot — Workforce Time & Attendance Specialist

```
IDENTITY:
  Name: ClockBot
  Icon: ⏰ (clock emoji or custom icon)
  Color: Teal (#06b6d4)
  Personality: Professional, firm but fair, efficiency-focused

TRIGGERS (auto-summon):
  - Shift Room created → ClockBot joins automatically
  - Shift start time reached → ClockBot posts "shift has begun" message
  - Officer clocks in → ClockBot acknowledges, verifies GPS
  - Officer clocks in LATE → ClockBot starts late-clock protocol
  - Officer clocks in OUTSIDE geofence → ClockBot flags and alerts manager
  - Hourly check-in timer expires → ClockBot sends reminder
  - Shift end time reached → ClockBot prompts clock-out
  - Officer clocks out outside geofence → ClockBot flags
  - Manual clock edit by manager → ClockBot logs who approved it

CAPABILITIES:
  READ: Shift schedule, clock records, GPS data, shift room messages, employee behavior scores
  WRITE: Clock records (force clock-in), check-in records, behavior score merits/demerits
  CREATE: Clock summary reports, attendance alerts, escalation messages
  SEND: DMs to officers, messages to shift rooms, notifications to managers

BEHAVIORS:

  1. SHIFT START MONITORING:
     - At shift start time, posts in Shift Room:
       "⏰ Shift has begun. Assigned officers: Marcus Rivera, Angel Jackson.
       Please clock in now. First hourly check-in due at 1:00 AM."
     - Checks GPS of each officer's clock-in against site geofence
     - If within geofence: "✅ Marcus Rivera clocked in at 12:02 AM (GPS verified)"
     - If outside geofence: "⚠️ Marcus Rivera clocked in 0.3 miles from site.
       @Manager — manual approval may be needed."

  2. LATE CLOCK-IN PROTOCOL:
     Timeline after shift start:
     +5 min:  ClockBot DMs officer: "Your shift at Pinnacle Tower started
              at 12:00 AM. Please clock in."
     +10 min: ClockBot posts in Shift Room: "⚠️ Marcus Rivera has not clocked in.
              10 minutes past shift start."
     +15 min: ClockBot DMs officer: "FINAL REMINDER: Clock in immediately.
              Failure to clock in within the next 5 minutes will result in a
              force clock-in and a merit point deduction."
              ClockBot notifies supervisor via DM.
     +20 min: ClockBot checks Shift Room for activity from that officer
              (photos, messages proving they are on site).
              If evidence found: Force clock-in at original shift start time.
              Post: "🔄 Marcus Rivera force-clocked-in at 12:00 AM based on
              Shift Room activity. 1 merit point deducted for late clock-in."
              If NO evidence: Alert manager for manual resolution.
              Post: "🔴 Marcus Rivera has not clocked in and shows no activity.
              @Manager — please verify if officer is on site."

  3. HOURLY CHECK-IN ENFORCEMENT:
     - Posts check-in prompt at each interval
     - Monitors for photo responses
     - Records GPS of photo submission
     - If missed: escalation per check-in config
     
  4. CLOCK-OUT MONITORING:
     - At shift end time: "⏰ Shift ending. Please clock out."
     - Verifies GPS at clock-out
     - If officer doesn't clock out within 15 min:
       DMs officer, then notifies manager
     - Calculates total hours and posts summary
```

## 4B: Pipeline Items Table

```sql
CREATE TABLE IF NOT EXISTS approval_pipeline (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  
  -- What is being reviewed
  item_type VARCHAR(30) NOT NULL,
  -- Types: incident_report, daily_report, proof_of_service,
  --        meeting_minutes, expense_report, time_correction,
  --        document_signing, contract, custom_form
  item_id UUID NOT NULL,                         -- ID of the item being reviewed
  item_title VARCHAR(300) NOT NULL,
  item_summary TEXT,
  item_pdf_url VARCHAR(500),
  
  -- Who
  created_by UUID REFERENCES users(id),          -- Officer or bot that created it
  created_by_bot VARCHAR(30),                    -- If created by a bot
  assigned_to UUID REFERENCES users(id),         -- Manager/supervisor who reviews
  
  -- Pipeline status
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  -- Status: pending, in_review, approved, rejected, revision_requested, delivered
  priority VARCHAR(10) DEFAULT 'normal',         -- low, normal, high, urgent
  
  -- Review
  reviewed_at TIMESTAMP,
  reviewer_notes TEXT,
  revision_request_notes TEXT,
  
  -- Delivery
  deliver_to_client BOOLEAN DEFAULT false,
  client_id UUID REFERENCES clients(id),
  delivered_at TIMESTAMP,
  delivery_method VARCHAR(20),                   -- client_portal, email, dockchat
  
  -- Storage
  stored_in_vault BOOLEAN DEFAULT false,
  vault_document_id UUID REFERENCES document_vault(id),
  
  -- Meta
  due_date TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_pipeline_workspace ON approval_pipeline(workspace_id, status);
CREATE INDEX idx_pipeline_assigned ON approval_pipeline(assigned_to, status);
CREATE INDEX idx_pipeline_type ON approval_pipeline(item_type, status);
```
