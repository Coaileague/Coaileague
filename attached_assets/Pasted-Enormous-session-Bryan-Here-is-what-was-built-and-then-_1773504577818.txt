AUDIT CATEGORY 4 — SILENT FAILURE TRUTH AUDIT
Every catch block that only logs. Every unhandled promise. Every fire-and-forget.

Search the entire codebase for these patterns:

Pattern: catch(err) { console.error(...) } or catch(err) { logger.error(...) }
with no notification, no retry, no audit trail entry after the log.
For every match assess: if this fails at 2am on a Tuesday, does anyone find out?
If the answer is no, add at minimum: an audit trail entry with the error,
and if it is a user-facing operation add a notification to platform_staff.

Pattern: async function calls without await in a context where the result matters.
These create race conditions and silent failures.
Every async operation that affects data must be awaited.

Pattern: Promise chains without .catch() handlers.
Every unhandled promise rejection in production is a silent failure.

Pattern: email sends where failure is caught and swallowed.
If Resend fails the caller must know. The operation must be retried or queued.
Specifically check: every sendCanSpamCompliantEmail call,
every sendBrandedEmail call, every individual email function in emailCore.

Pattern: WebSocket broadcasts where failure is not handled.
If a broadcast fails the recipients never know an event occurred.
Add error handling to every broadcastToWorkspace call.

Pattern: document safe writes where failure is caught and swallowed.
If a shift report or compliance document fails to save to the document safe
the manager gets a notification that it was saved when it was not.
Every document safe write must either succeed with confirmation or fail with an alert.

Report total count of silent failure patterns found and fixed.

AUDIT CATEGORY 5 — API RESPONSE TRUTH AUDIT
Every API endpoint must return the correct status code and a useful response body.

Find every API endpoint in every route file.
For each endpoint verify:

On success: returns the correct 2xx status with a response body that contains
the actual data the frontend needs — not just {success: true}.

On validation failure: returns 400 with a specific message identifying
which field failed and why — not a generic error.

On auth failure: returns 401 with a message that does not expose internals.

On permission failure: returns 403 with a message that does not expose
the internal role name — use the sanitized message from the RBAC fix.

On not found: returns 404 with a message identifying what was not found.

On server error: returns 500 with a generic message that never includes
a stack trace, file path, or internal variable names.

Confirm the global error handler in production mode strips all stack traces.
Test this right now by triggering a deliberate server error and confirming
the response contains no stack trace, no file path, no internal detail.
