🛡️ I. Transactional Safety & Resilience (The "Must Haves")
Scheduling is a state-modifying process. Failures must be atomic and reversible.

1. Transactional Integrity and Rollback Logic
Action: Your application must treat the entire scheduling process as a transaction. If the generate_schedule() tool succeeds but the github_committer tool fails (e.g., network error prevents the final database write or API call), the system must execute a compensating transaction (Rollback).

Rollback Mechanism: If the final step fails, the system must trigger a specific tool function (revert_schedule_temp_state()) to clear the partially completed schedule data from all temporary files or database tables, ensuring a clean slate for the next run.

Why: Prevents "partial state" issues (e.g., the schedule is halfway assigned in the database, but the user doesn't see it, leading to corruption).

2. Handling Unexpected State Changes (Call-Outs)
Action: Implement a protocol for a real-time system change, like an employee calling out after the schedule is finalized.

Logic: The system must receive the real-time "Call-Out" signal and instantly trigger a Micro-Scheduling Run. Lomu should be instructed to:

Isolate the affected shift(s).

Query the available employee pool again (using get_employee_profiles()).

Find the best backup candidate based on the Reliability Score and Distance (the pre-defined Soft Constraints).

Generate an urgent notification for the best backup candidate, without re-running the entire week's schedule.

Why: Automation needs to handle real-world chaos, not just batch processing.
