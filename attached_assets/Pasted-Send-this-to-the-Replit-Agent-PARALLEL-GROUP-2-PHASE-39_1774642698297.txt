**STEP 2 — SUBSCRIPTION STATE MACHINE**
Verify and enforce the complete subscription state machine:

Valid states: `trialing → active → past_due → suspended → cancelled`

State transitions:
- `trialing → active`: trial period ends, first payment succeeds
- `trialing → cancelled`: trial ends, no payment method, or user cancels during trial
- `active → past_due`: payment fails — Stripe fires `invoice.payment_failed`
- `past_due → active`: payment succeeds during dunning — Stripe fires `invoice.payment_succeeded`
- `past_due → suspended`: dunning period exhausted (default 7 days) — workspace access restricted
- `suspended → active`: payment made — workspace access restored immediately
- `suspended → cancelled`: workspace owner explicitly cancels
- `cancelled → trialing`: not permitted — new workspace required

Every transition must:
- Update subscription status in database immediately
- Write an audit record with from_state, to_state, triggered_by (Stripe event or user action), timestamp
- Fire NDS notification to org_owner
