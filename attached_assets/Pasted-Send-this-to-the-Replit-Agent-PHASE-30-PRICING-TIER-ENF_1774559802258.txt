**STEP 2 — TIER DEFINITION AND FEATURE REGISTRY**

The tier system must have a single canonical source of truth:

- Each tier must define: tier_name, monthly_price, seat_limit, features[] (list of feature keys from featureRegistry.ts), and any tier-specific limits (API rate limits, storage limits, export limits)
- Tier hierarchy: Starter < Professional < Business < Enterprise < Strategic

**STEP 3 — SERVER-SIDE TIER ENFORCEMENT MIDDLEWARE**

Every tier-gated request must be blocked server-side:
- Create or verify `server/middleware/tierEnforcement.ts`
- Middleware accepts a `minimumTier` parameter: `checkTierAccess('professional')`
- Middleware reads workspace tier from DB (not from client-supplied header or body)
- Middleware compares workspace tier against minimum required tier using the tier hierarchy
- Below minimum tier: returns 403 with `{ error: 'TIER_UPGRADE_REQUIRED', currentTier, requiredTier, upgradeUrl }` — never a generic 403
- Above or equal minimum tier: passes through to the next handler
- Strategic and Enterprise tiers pass all tier checks
- Statewide Protective Services workspace passes all tier checks regardless of stored tier
- Tier check result cached per request — no duplicate DB queries on the same request
- Every tier enforcement block logged with: workspace_id, attempted_feature, current_tier, required_tier, timestamp

---

**STEP 4 — SEAT LIMIT ENFORCEMENT**

Seat limits must be enforced at officer activation time:
- Current active officer count queried at activation time — not cached
- If activating would exceed the workspace's tier seat limit: return 403 with `{ error: 'SEAT_LIMIT_REACHED', currentCount, seatLimit, upgradeUrl }`
- Seat limit check also runs on bulk import — cannot bulk import 50 officers into a 10-seat Starter plan
- Seat limit enforcement cannot be bypassed by any role including org_owner
- Deactivated officers do not count toward seat limit — only active officers
- Seat count visible to org_owner in workspace settings — shows current/limit in real time
- Approaching seat limit (at 80% capacity) triggers a proactive NDS notification to org_owner
- Enterprise and Strategic tiers enforce their custom seat limits — not unlimited

**STEP 6 — UI TIER GATE INTEGRATION**

The UI must reflect tier access correctly:
- Features not available at the current tier shown as locked in the UI — not hidden, but visibly locked with an upgrade prompt

- Upgrade prompt links to correct pricing page with the correct tier highlighted

- Downgrade behavior: if workspace downgrades and currently uses features above the new tier — grace period defined and communicated, features locked after grace period

**STEP 7 — TIER UPGRADE AND DOWNGRADE HANDLING**

Tier transitions must be seamless and correctly enforced:
- Upgrade: tier change takes effect immediately — features unlocked without server restart
- Upgrade: seat limit increases immediately — officers can be activated without waiting
- Downgrade: grace period of 30 days before feature lock — org_owner notified at downgrade, at 7 days remaining, and at lock date
- Downgrade: if seat count exceeds new tier limit at grace period end — org_owner must deactivate officers manually or upgrade again, platform does not auto-deactivate
- Tier change audit record written: workspace_id, from_tier, to_tier, changed_by, timestamp, reason
- Trinity notified of tier change — workspace context updated immediately
- Feature registry cache invalidated on tier change — no stale tier checks after upgrade
