**STEP 4 — HELPAI LIVE TRIAGE FLOW**

HelpAI must actively triage every user who needs help:
- HelpAI accessible via persistent button on every page
- On open: HelpAI greets user by name, identifies their role and workspace context
- HelpAI asks: "What can I help you with today?" — open-ended, not a menu
- HelpAI intent classification: bug report, billing question, how-to question, feature request, access issue, escalation needed
- For how-to questions: HelpAI searches FAQ database first, returns relevant answer with source
- For bug reports: HelpAI walks user through capturing reproduction steps, then creates ticket automatically
- For access issues: HelpAI checks user's current permissions, explains what they can't access and why, offers to submit an access request
- For billing questions: HelpAI retrieves relevant invoice or subscription data for the user's workspace and answers from context
- For escalation needed: HelpAI creates ticket, sets high priority, notifies platform staff immediately
- HelpAI follows up: if issue was resolved via FAQ, asks "Did this solve your problem?" — yes closes the interaction, no creates a ticket
- HelpAI session context: remembers what was discussed in the current session — does not ask the user to repeat themselves
- New user flow: HelpAI detects first-time users, offers a guided onboarding tour relevant to their role

**STEP 5 — FAQ DATABASE AND MANAGEMENT**

The FAQ system must be intelligent and self-improving:
- FAQ table schema: id, category, question, answer, related_ticket_ids, created_by, created_at, updated_at, view_count, helpful_count, not_helpful_count, status (draft/published/archived), auto_suggested (boolean)
- FAQ categories match ticket categories
- FAQ search: full-text search, returns most relevant result first
- FAQ management UI: platform staff can create, edit, publish, archive FAQ entries
- FAQ accessible to end users via help center page — searchable, categorized
- HelpAI reads from FAQ database on every relevant query — not hardcoded responses
- FAQ voting: users can mark an answer as helpful or not helpful — drives FAQ quality score

---

**STEP 6 — HELPAI LEARNING LOOP**

HelpAI must get smarter from every resolved ticket:
- When a ticket is resolved: platform staff marks it as `faq_candidate: true` if the resolution would help other users
- Trinity/HelpAI reviews `faq_candidate` tickets daily — extracts the question and resolution
- HelpAI drafts a FAQ entry: proposed question (cleaned and generalized), proposed answer (from resolution notes), suggested category
- Draft FAQ entry submitted for platform staff review — not auto-published
- Platform staff approves, edits, or rejects the draft
- Approved drafts published to FAQ database automatically
- HelpAI's response quality improves as FAQ database grows — it references new entries on next relevant query
- Learning loop audit: every auto-suggested FAQ entry tracks its source ticket ID

- Per-organization issue history accessible to platform staff on the ticket list — "View workspace history" shows all prior tickets for that workspace
- Issue history workspace-scoped — HelpAI cannot reference another workspace's issues
