# CoAIleague — Silent Failure Detection
## Run AFTER T001-T004 Complete

**Context:** 44 tables were missing workspace_id. If that many tables lacked basic org scoping, there are almost certainly workflows that fail silently — no error shown to user, no notification to manager, no record that anything went wrong. This audit exposes them.

---

## STEP 1: Find Every try/catch That Swallows Errors

```bash
# Find all catch blocks that only console.log/console.error (no user notification, no DB logging)
grep -rn "catch" server/ --include="*.ts" --include="*.js" -A 5 | grep -B 1 "console\.\(log\|error\|warn\)" | grep -v "node_modules"

# Find all catch blocks that are completely empty (swallow errors silently)
grep -rn "catch.*{" server/ --include="*.ts" --include="*.js" -A 1 | grep -B 1 "}" | grep -v "node_modules"

# Find all catch blocks that just "return" without handling
grep -rn "catch" server/ --include="*.ts" --include="*.js" -A 3 | grep "return\s*;" | grep -v "node_modules"

# Find all catch blocks that return null/undefined/false (silent failure disguised as "no result")
grep -rn "catch" server/ --include="*.ts" --include="*.js" -A 3 | grep "return\s*\(null\|undefined\|false\|\[\]\|{}\)" | grep -v "node_modules"
```

**What to look for:** Every catch block should do ONE of these:
1. Throw the error up (let the caller handle it)
2. Return a clear error response to the user
3. Log to the audit_log table (not just console)
4. Send a notification to the manager/owner

If a catch block does NONE of these — it's a silent failure point.

## STEP 2: Find Every Cron/Automation That Doesn't Report Failures

```bash
# Find all cron jobs / scheduled tasks
grep -rn "cron\|schedule\|setInterval\|setTimeout" server/ --include="*.ts" --include="*.js" | grep -v "node_modules" | grep -v "test"

# For each one, check: does it have error handling that notifies someone?
# Many cron jobs look like:
#   cron.schedule('0 0 * * *', async () => {
#     const result = await doSomething()  // If this fails... nobody knows
#   })
```

**Every cron job MUST have:**
```typescript
cron.schedule('...', async () => {
  try {
    const result = await doSomething();
    // Log success
    await auditLog.log({ action: 'cron.job_name.success', ... });
  } catch (error) {
    // Log failure WHERE SOMEONE WILL SEE IT
    await auditLog.log({ action: 'cron.job_name.failed', metadata: { error: error.message } });
    // Notify org owner or platform admin
    await notificationService.sendAlert({
      title: 'Automation Failed: [Job Name]',
      body: error.message,
      severity: 'high',
    });
  }
});
```

## STEP 4: Find Database Writes Without Error Handling

```bash
# Find all db.insert / db.update / db.delete calls
grep -rn "db\.\(insert\|update\|delete\)\|\.insert(\|\.update(\|\.delete(" server/ --include="*.ts" --include="*.js" | grep -v "node_modules" | grep -v "test"

# For each one, check: is it wrapped in a try/catch?
# Database writes can fail for many reasons:
# - Unique constraint violation (duplicate)
# - Foreign key violation (orphaned reference)
# - NOT NULL violation (missing required field)
# - Connection timeout
# If ANY of these happen without error handling, data is silently lost.
```

## STEP 6: Find Financial Operations Without Idempotency Guards

These are the ones that can LOSE MONEY:

```bash
# Find all credit deduction calls
grep -rn "deduct\|consume\|spend\|charge\|credit" server/ --include="*.ts" --include="*.js" | grep -v "node_modules" | grep -v "test" | grep -v "\.md"

# For each credit deduction:
# 1. Is there a check for existing transaction (idempotency)?
# 2. Is it inside a database transaction (BEGIN/COMMIT)?
# 3. Is there a SELECT FOR UPDATE lock on the balance?
# 4. What happens if it fails mid-deduction? (credits gone but action didn't complete)

# Find all invoice generation calls
grep -rn "invoice\|billing\|bill\|fee\|charge" server/ --include="*.ts" --include="*.js" | grep -v "node_modules" | grep -v "test" | grep -v "\.md"

# For each financial operation:
# 1. Is there duplicate prevention?
# 2. Is the amount calculated correctly (integers/cents, not floating point dollars)?
# 3. Is the operation logged?
```

## THE FIX PATTERN

For every silent failure found, apply this pattern:

```typescript
// BEFORE (silent failure):
try {
  const result = await someAction();
} catch (error) {
  console.error(error);  // Nobody sees this
}

// AFTER (visible failure):
try {
  const result = await someAction();
  
  // Log success
  await auditLog.log({
    workspaceId,
    action: 'action_name.success',
    actorType: context.type,
    entityType: 'table_name',
    entityId: result.id,
    changeType: 'create',
  });
  
} catch (error) {
  // Log failure to database (permanent, queryable)
  await auditLog.log({
    workspaceId,
    action: 'action_name.failed',
    actorType: context.type,
    entityType: 'table_name',
    changeType: 'action',
    metadata: { error: error.message, stack: error.stack },
  });
  
  // Notify someone who can act on it
  await notificationService.create({
    workspaceId,
    type: 'system_error',
    title: `Action Failed: ${actionName}`,
    body: `${error.message}. This may require manual intervention.`,
    targetRoles: ['owner', 'manager'],
    severity: 'high',
  });
  
  // Return clear error to caller
  throw new AppError(`Failed to ${actionName}: ${error.message}`, 500);
}
```

**RULE: If it can fail, it must VISIBLY fail. Console.log is NOT visibility.**
