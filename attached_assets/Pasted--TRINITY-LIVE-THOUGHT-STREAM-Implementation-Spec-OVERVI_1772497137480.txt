# TRINITY LIVE THOUGHT STREAM — Implementation Spec

## OVERVIEW

Trinity currently shows "Processing..." and "Analyzing staffing" in her status bar and workbox. This is boring and opaque. Users have no idea what she's actually doing, whether she's stuck, or what decisions she's making.

**The goal:** Trinity should show her real-time thinking, reasoning, and execution steps in natural language — like watching an AI agent work. This builds trust, shows transparency, and lets managers understand WHY Trinity made each decision.

## WHERE IT SHOWS

### 1. Status Bar (the blue bar at the top of schedule)
Currently shows: "Processing 0/0 — Starting auto-fill for 0 shifts..."
Should show: A single-line summary that updates as Trinity progresses through her cycle.

Examples of what it should say at each phase:
```
"Scanning 44 clients for staffing needs..."
"Found 12 clients needing coverage for Monday..."
"Generating 38 shifts across 12 clients..."
"Evaluating 115 employees for best fit..."
"Placing employees — 24 of 38 shifts filled..."
"Completed — 34 shifts filled, 4 unfilled (staffing shortage). Ready for review."
```

### 2. Trinity Workbox (the expandable panel showing execution steps)
Currently shows: "1 steps — Analyzing staffing" (same message repeated)
Should show: A scrollable feed of Trinity's actual thinking, organized by phase.

## HOW TO IMPLEMENT

### Backend: Trinity Thought Stream Service

Create a service that Trinity writes to as she works. Each step is a "thought" with a type, message, timestamp, and optional metadata.

```typescript
// Thought types
type TrinityThoughtType = 
  | 'phase_start'    // Starting a new phase
  | 'info'           // Normal progress update
  | 'success'        // Something completed successfully
  | 'warning'        // Issue detected but handled
  | 'error'          // Something failed
  | 'decision'       // Trinity made a choice (employee placement, skip, etc.)
  | 'recommendation' // Trinity has a suggestion for the org
  | 'summary'        // End-of-phase or end-of-cycle summary

interface TrinityThought {
  id: string;
  workspaceId: string;
  sessionId: string;        // Groups thoughts for one execution cycle
  type: TrinityThoughtType;
  phase: string;            // 'data_gathering' | 'demand_analysis' | 'employee_matching' | 'compliance_check' | 'summary'
  message: string;          // Natural language message
  metadata?: {              // Optional structured data for the UI
    shiftId?: string;
    employeeId?: string;
    clientId?: string;
    score?: number;
    reason?: string;
  };
  timestamp: Date;
}
```

### Where Trinity Writes Thoughts

In the scheduling pipeline, at each decision point, Trinity calls:

```typescript
// Phase 1: Data gathering
await trinity.think('info', 'data_gathering', `Fetching client data... ${clients.length} active clients found`);
await trinity.think('info', 'data_gathering', `Loading employee roster... ${employees.length} active employees`);

// Phase 3: Employee matching (for each shift)
await trinity.think('decision', 'employee_matching', 
  `Shift: ${client.name} — ${shiftType}. Best match: ${employee.name} (score: ${score}). ` +
  `Distance: ${distance}mi | Rate margin: $${margin}/hr | Assigned ✓`,
  { shiftId, employeeId, clientId, score }
);

// When skipping an employee
await trinity.think('warning', 'employee_matching',
  `Skipping ${employee.name} for ${client.name} — would push to ${projectedHours}hrs (OT threshold)`,
  { employeeId, reason: 'ot_threshold' }
);

// When a shift can't be filled
await trinity.think('error', 'employee_matching',
  `Cannot fill ${client.name} ${shiftType} — no eligible employees available. Escalating.`,
  { shiftId, clientId, reason: 'no_candidates' }
);
```

### Frontend: Real-Time Stream

The thoughts are broadcast via WebSocket as they happen. The Trinity workbox subscribes and renders them as a scrolling feed. Each thought type gets a visual treatment:

- ✓ success = green checkmark
- ⚠ warning = yellow/orange warning icon  
- ✗ error = red X
- → decision = blue arrow (Trinity made a choice)
- 💡 recommendation = lightbulb

The status bar shows the LATEST thought as a single line, updating in real-time.

### Persistence

Thoughts are stored in a `trinity_thought_log` table so managers can review past cycles. Each cycle gets a `sessionId` so you can replay what Trinity did and why.

This also feeds into the LEARNING pipeline — if a manager overrides one of Trinity's decisions, the system can look at the thought log to understand what Trinity's reasoning was and adjust scoring weights.

## WHY THIS MATTERS

1. **Trust** — Managers see exactly why Trinity placed each employee. No black box.
2. **Debugging** — When something goes wrong, you can read Trinity's thought process to find where the logic broke.
3. **Training** — Trinity's reasoning becomes visible, so managers can provide feedback on her decision quality.
4. **Sales** — Demo this to a prospect and it sells itself. Watching an AI reason through a schedule in real-time is impressive.
5. **Differentiation** — No competitor has this. GetSling, Deputy, When I Work — none of them show AI reasoning transparently.

## WHAT NOT TO DO

- Don't make the thoughts generic like "Analyzing staffing..." repeated 10 times
- Don't show internal technical details like database query times or API calls
- Don't show every single micro-step — group related actions into meaningful updates
- Don't block the UI waiting for thoughts — stream them as they arrive
- Don't lose thoughts on page refresh — they should persist for the current session at minimum

## PRIORITY

This is a post-launch enhancement, not a launch blocker. The current "Processing..." status works functionally — it just doesn't build trust or show transparency. Implement this after the core revenue cycle is verified in production with a real org.
