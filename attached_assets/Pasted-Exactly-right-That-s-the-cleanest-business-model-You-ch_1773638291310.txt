**Why Plaid over Gusto for your situation:**

Gusto is a full payroll platform — using their API means you're essentially reselling Gusto and they control the compliance, tax filings, W2s. You'd be dependent on their product roadmap and pricing forever.

Plaid Transfer is just the ACH rail — it moves money and gets out of the way. You keep full control of the payroll logic, the UI, the compliance, the tax documents. Plaid is infrastructure, not a competitor. That's the right relationship.

Plaid Transfer also has instant bank verification built in — when an org connects their bank account or an employee adds direct deposit, Plaid verifies it in seconds. That solves your encryption concern too because Plaid stores the sensitive bank credentials, not your DB.

**PHASE 1 — Plaid Client Setup**

Install `plaid` npm package if not present.

Create `server/services/partners/plaidService.ts` as the single canonical Plaid client. All Plaid API calls in the entire codebase must go through this file only — no direct Plaid SDK calls anywhere else.

- `PLAID_SECRET`
- `PLAID_ENV` (sandbox / development / production)

Add all three to `.env.example` with descriptions. Add startup validation — server must not start without these in production.

Export these methods from `plaidService.ts`:
```
createLinkToken(userId, workspaceId) → link_token for frontend Plaid Link
exchangePublicToken(publicToken) → access_token + item_id
getAccountDetails(accessToken) → routing number, account number, account type
verifyBankAccount(accessToken) → balance check and account status
initiateTransfer(params) → create ACH transfer
getTransferStatus(transferId) → check transfer status
cancelTransfer(transferId) → cancel if not yet settled
```

**PHASE 2 — Org Bank Account Connection**

Create `POST /api/payroll/bank/link-token` — calls `plaidService.createLinkToken()`, returns link token to frontend. Requires auth, workspace-scoped.

Create `POST /api/payroll/bank/connect` — receives `publicToken` from Plaid Link frontend callback. Calls `plaidService.exchangePublicToken()`, stores `plaidAccessToken` and `plaidItemId` encrypted in workspace settings. Never store routing/account numbers directly — Plaid holds them. Writes audit log entry with `isSensitiveData: true`. Broadcasts `workspace_bank_connected` via platformEventBus.

Create `GET /api/payroll/bank/status` — returns whether org has a connected bank account (boolean + last4 of account if connected). Never returns full account details.

Create `DELETE /api/payroll/bank/disconnect` — removes Plaid access token from workspace, writes audit log, broadcasts `workspace_bank_disconnected`.

**PHASE 3 — Employee Direct Deposit via Plaid**

**PHASE 4 — Payroll Disbursement via Plaid Transfer**

In `POST /api/payroll/runs/:id/process` — after payroll is approved and pay stubs are created, replace the NACHA file as the final step with Plaid Transfer execution:

For each employee in the payroll run who has a connected Plaid bank account:
1. Call `plaidService.verifyBankAccount()` to confirm account is valid and active
2. Call `plaidService.initiateTransfer()` with amount, employee account, org account as funding source
3. Store the returned `transferId` on the pay stub record
4. Set employee pay stub status to `transfer_initiated`
5. Emit `platformEventBus.publish({ type: 'payroll_transfer_initiated', workspaceId, employeeId, transferId, amount })`

For each employee WITHOUT a connected Plaid account — fall back to NACHA file generation for that employee only. Flag them in the payroll run summary as `requires_manual_disbursement`.

After all transfers initiated — update payroll run status to `disbursing`. Keep `processed` → `paid` transition for when all transfers confirm.

**PHASE 5 — Transfer Status Polling & Confirmation**

Create `payrollTransferMonitor.ts` service that runs every 2 hours via cron.

For every payroll run in `disbursing` status:
- Call `plaidService.getTransferStatus(transferId)` for each pending transfer
- If status is `settled` → update pay stub to `paid`, emit `payroll_transfer_settled`
- If status is `failed` → update pay stub to `failed`, emit `payroll_transfer_failed`, notify org owner with employee name and failure reason
- If ALL transfers for a run are settled → transition payroll run to `paid`, emit `payroll_run_paid`

Add `POST /api/payroll/runs/:id/retry-failed-transfers` — for any failed transfers, retry once with `plaidService.initiateTransfer()`. Requires manager role.
