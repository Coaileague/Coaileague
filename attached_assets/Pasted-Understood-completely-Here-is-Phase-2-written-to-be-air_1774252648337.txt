**ROOT CAUSE 2 — MISSING DATABASE TRANSACTIONS ON MULTI-TABLE FINANCIAL WRITES**

*The disease: operations that write to more than one database table as a single logical unit are executed as sequential independent writes with no transaction wrapping. Specifically search in these domains — these are the highest risk areas:

Payroll domain: any sequence touching more than one of payroll_runs, payroll_entries, pay_stubs, employee_payroll_info, employee_bank_accounts, payroll_deductions.

Invoice domain: any sequence touching more than one of invoices, invoice_line_items, invoice_payments, client billing records.

**Fix — apply completely:**

Wrap every identified sequence in `db.transaction(async (tx) => { })`. Every database call within the transaction must use `tx` as the database handle — never the global `db` inside a transaction block. The transaction commits only when every write succeeds. If any write throws, all writes roll back automatically.

The canonical pattern that every transaction in this codebase must follow:

```typescript
const result = await db.transaction(async (tx) => {
  // First write
  const [primaryRecord] = await tx
    .insert(primaryTable)
    .values(primaryData)
    .returning();

  // Second write depends on first
  await tx
    .insert(relatedTable)
    .values({ 
      ...relatedData, 
      primaryId: primaryRecord.id,
      workspaceId: primaryRecord.workspaceId 
    });

  // Return whatever the caller needs
  return primaryRecord;
});
```

Every transaction must: use `tx` not `db` for every operation inside it, include `workspaceId` on every inserted record, return a meaningful result to the caller, and handle only writes that are logically coupled — do not bundle unrelated writes into one transaction.

Wrapping the sequence in a try-catch and manually rolling back with a second database call on error. Manual rollback is not a transaction. If the manual rollback itself fails, you have compounded the inconsistency. Use `db.transaction()`.

Wrapping only the first and second writes when there are three writes in the sequence. If three writes are logically coupled, all three go inside the same transaction.

```typescript
// All outputs are strings (for DB storage or API response)
// The only time a financial value becomes a JS number is formatCurrency() for display

export function calculateGrossPay(
  hours: string, 
  rate: string, 
  payType: 'hourly' | 'salary' | 'contract'
): string {
  const h = new Decimal(hours);
  const r = new Decimal(rate);
  if (payType === 'salary') return r.toFixed(4);
  return h.mul(r).toFixed(4);
}

export function calculateOvertimePay(
  overtimeHours: string,
  rate: string,
  multiplier: string = '1.5'
): string {
  return new Decimal(overtimeHours)
    .mul(new Decimal(rate))
    .mul(new Decimal(multiplier))
    .toFixed(4);
}

export function calculateInvoiceLineItem(
  hours: string, 
  clientRate: string
): string {
  return new Decimal(hours)
    .mul(new Decimal(clientRate))
    .toFixed(4);
}

export function calculateInvoiceTotal(lineItems: string[]): string {
  return lineItems
    .reduce((sum, item) => sum.plus(new Decimal(item)), new Decimal(0))
    .toFixed(4);
}

export function applyDeduction(
  gross: string, 
  deductionAmount: string
): string {
  return new Decimal(gross)
    .minus(new Decimal(deductionAmount))
    .toFixed(4);
}

export function calculateNetPay(
  gross: string, 
  deductions: string[]
): string {
  const totalDeductions = deductions
    .reduce((sum, d) => sum.plus(new Decimal(d)), new Decimal(0));
  return new Decimal(gross)
    .minus(totalDeductions)
    .toFixed(4);
}

export function sumFinancialValues(values: string[]): string {
  return values
    .reduce((sum, v) => sum.plus(new Decimal(v)), new Decimal(0))
    .toFixed(4);
}

export function formatCurrency(amount: string): string {
  // ONLY use this at the display boundary — never for calculation inputs
  return new Decimal(amount).toDecimalPlaces(2, Decimal.ROUND_HALF_EVEN).toFixed(2);
}
```

**ROOT CAUSE 7 — UNVALIDATED API RESPONSE SHAPES ON FINANCIAL AND OPERATIONAL BOUNDARIES**

*The disease: TypeScript interfaces used to type API responses are compile-time assertions. They describe what the developer intended the API to return. They do not check what the API actually returns at runtime. When the API returns a different shape — a null where a number was expected, a string key where a different string key was expected, a missing nested object — TypeScript does not catch it at runtime. The value is undefined. The component renders blank or zero. No error is thrown. A payroll display shows $0.00 for an employee who should be paid $847.50. No one knows until the employee complains.*

**How this connects to Phase 1:** Phase 1 established explicit column mappings so the database layer returns exactly the right field names. Root Cause 7 establishes runtime validation at the API boundary so the frontend layer receives exactly the right field names and types and knows immediately when it does not.

**Search — find every instance:**

Scan every frontend service file and every React Query hook. For each API call that returns financial data, schedule data, employee data, or Trinity outputs: identify the TypeScript type used to type the response. Determine whether that type is a compile-time assertion only (`as ResponseType`, `<ResponseType>response.data`) or whether it includes runtime validation. List every API call that uses compile-time assertion only with no runtime validation.

Scan every API route that accepts financial or operational input in the request body. Identify every route that reads from `req.body` without first running the body through a validation function that checks types, required fields, and value constraints at runtime. List every route with no body validation or only partial body validation.

**Fix — apply completely:**

Install Zod as a project dependency if not already installed: `npm install zod`. Zod is the canonical runtime validation library for this codebase. No other validation library is introduced for this purpose.

Create `shared/schemas/` directory. This directory contains Zod schema files organized by domain — payroll, invoices, employees, shifts, clients, Trinity, subscriptions. These schemas are importable from both the frontend and the backend. One schema definition, used in both directions.

For every financial and operational API response: write a Zod schema that describes the exact shape the API returns. Add runtime validation to every React Query `queryFn` that calls a financial or operational endpoint:

```typescript
const { data } = useQuery({
  queryKey: ['payroll', workspaceId],
  queryFn: async () => {
    const response = await fetch(`/api/payroll?workspaceId=${workspaceId}`);
    const json = await response.json();
    // Runtime validation — catches shape mismatches at the boundary
    return PayrollResponseSchema.parse(json);
  }
});
```

If `PayrollResponseSchema.parse(json)` throws, it means the API returned something different from what was expected. The error surfaces immediately with the exact field that failed validation. This is diagnosed and fixed. Not silently rendered as blank.

For every API route that accepts financial or operational input: add request body validation middleware using the corresponding Zod schema:

```typescript
import { z } from 'zod';

const CreateInvoiceSchema = z.object({
  clientId: z.string().uuid(),
  lineItems: z.array(z.object({
    description: z.string().min(1),
    hours: z.string().regex(/^\d+\.?\d{0,4}$/),
    rate: z.string().regex(/^\d+\.?\d{0,4}$/)
  })).min(1),
  dueDate: z.string().datetime()
});

// In route handler:
const parseResult = CreateInvoiceSchema.safeParse(req.body);
if (!parseResult.success) {
  return res.status(400).json({
    error: 'Invalid request body',
    fields: parseResult.error.flatten().fieldErrors
  });
}
const validatedBody = parseResult.data; // fully typed, no any
```

Document the canonical financial value serialization contract in replit.md: financial values are stored as numeric strings in the database (Drizzle returns PostgreSQL numeric type as string). They travel as strings in API request and response bodies. They are parsed by `FinancialCalculator` functions (established in Root Cause 4) before arithmetic. They are formatted by `formatCurrency()` at the display boundary only. No financial value is a native JavaScript number at any layer except inside `formatCurrency()`.

**What is not acceptable:**

Adding Zod schemas to new endpoints only and leaving existing endpoints unvalidated. Every financial and operational API boundary gets Zod validation. The list from the search pass is the complete work order.

Using `z.any()` or `z.unknown()` in the Zod schema for financial fields. A schema that accepts anything is not a schema. Every financial field has a specific type constraint.

Adding Zod validation to the frontend only and leaving the backend routes with no request body validation. Both directions. Request bodies validated on the backend. Response bodies validated on the frontend.

**Verification — prove it works:**

Submit a request with a missing required field to every financial endpoint. Confirm every one returns 400 with the specific field name that failed validation. Submit a request with a pay rate of -5. Confirm 400 with a clear error message. Submit a request with a non-numeric string as an hours value. Confirm 400.

On the frontend: mock an API response that returns `grossPay: null` for a payroll endpoint. Confirm the Zod parse throws and the error is caught. Confirm the component does not render `$0.00` silently — it surfaces an error state.

Re-run all ten baseline operations. All ten must pass with correct data.

Add to replit.md Architecture Canons: "Zod is the canonical runtime validation library. Every financial and operational API boundary has a Zod schema enforced in both directions — request body validation on the backend, response validation on the frontend. TypeScript interface-only typing of API responses is permanently retired for financial and operational data."
