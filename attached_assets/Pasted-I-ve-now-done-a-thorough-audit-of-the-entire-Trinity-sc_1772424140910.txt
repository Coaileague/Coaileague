I've now done a thorough audit of the entire Trinity scheduling engine. Let me compile what I found:

Issues identified:

BUG: weekEnd calculation is off by 1 day (line 1059-1061) — weekStart is set to the beginning of the week (Sunday), then weekEnd = weekStart + 6 days at 23:59:59. But setDate mutates weekStart first, then creates weekEnd from it. Actually wait — line 1059 creates a new Date from shiftStart. Let me re-read that...
Actually the code at lines 1057-1061 looks correct: weekStart = new Date(shiftStart) → set to Sunday → weekEnd = new Date(weekStart) → add 6 days → set to 23:59:59.

BUG: dynamicMaxShifts can be too restrictive (line 328) — config.maxShiftsPerEmployee can be set to a minimum of 5 by the dynamicMaxPerWeek logic, but if there are many shifts and few employees, the + 3 buffer may not be enough. The formula also uses Math.min(config.maxShiftsPerEmployee, dynamicMaxPerWeek) which can reduce a manually-set higher limit.

BUG: 2-second cache TTL on scoring queries (line 1077) — During a run, the cached data for weekly assignments/hours becomes stale because runTracker compensates. BUT the cache is computed from DB state that doesn't include the current run's DB writes (the db.update at line 956 commits immediately). This means the next shift's cached query will include the just-assigned shift from DB, BUT only if the cache expired. With 2s TTL, multiple shifts processed within 2s will use stale DB data — but runTracker compensates. This is actually fine.

BUG: workloadBalance can go negative (line 1520-1523) — currentAssignments / maxShifts can exceed 1 if a contractor or employee already has more shifts than maxShifts from prior scheduling. This would make workloadBalance negative, which lowers their score but doesn't disqualify them. Not a hard bug but incorrect.

REAL BUG: Coverage filter silently drops shifts (lines 745-753) — If a client has coverageDays or coverageStartTime/EndTime configured, shifts outside those windows are filtered OUT entirely and never assigned. The console log says "skipped" but these aren't counted in skippedShifts — they just vanish from the total. This is invisible revenue loss.

REAL BUG: next_week mode includes current week too (line 568) — totalDays = 14 - dayOfWeek starts from TODAY, so "next_week" actually processes remainder of THIS week plus all of next week. Not necessarily wrong for coverage but misleading.

REAL BUG: maxShiftsPerEmployee being capped by dynamicMaxPerWeek when user sets it manually (line 329) — If a user/config explicitly sets maxShiftsPerEmployee, the code takes the MINIMUM of the user-set value and the dynamic calculation. But if the dynamic calculation is too low (because avgShiftsNeededPerEmployeePerWeek underestimates actual need), shifts go unfilled.
