PART 2 — STRIPE WEBHOOK HANDLER TRUTH AUDIT

Read stripeWebhooks.ts, stripeEventBridge.ts, and every file they import completely.

For every webhook event the platform handles verify this exact sequence:

SEQUENCE THAT MUST HAPPEN FOR EVERY WEBHOOK:
1. Stripe signature is verified using stripe.webhooks.constructEvent before ANY processing.
   If signature verification fails return 400 immediately — process nothing.
2. The event ID is checked against the idempotency_keys table before processing.
   If the event ID already exists return 200 immediately — do not process again.
3. The event ID is inserted into idempotency_keys before processing begins.
   This prevents duplicate processing even if the handler crashes mid-execution.
4. The business logic executes.
5. The idempotency key is marked complete after successful processing.
6. Return 200 to Stripe.

If any webhook handler skips step 1 that is a security vulnerability.
If any webhook handler skips steps 2-3 that is a duplicate-charge risk.

Now verify the specific business logic for each event:

Find the workspace from session.metadata.workspaceId.

Set the Stripe subscription ID on the workspace or subscription record.
Set the Stripe customer ID on the workspace record.
Allocate the correct first-month credits:
Starter: exactly 5000 credits, transaction_type subscription_allocation,
stripe_invoice_id from the session.
Professional: exactly 10000 credits, same fields.
Enterprise: exactly 22000 credits, same fields.
Send the subscription confirmation email via Resend.
Send the subscription confirmation in-platform notification.
Broadcast a WebSocket event to update the billing widget.
Log to the universal audit trail.
Verify every one of these 9 things actually happens for this event.
Report which ones do and which ones do not.
