BUG 1 — HARD CAP NOT ENFORCING ON STARTER AND TRIAL

checkCredits() must return a hard block for free_trial and starter tiers when balance is at or below zero. No exceptions. No bypass.

Correct logic for HARD_CAP tiers:
requested_cost = credit cost of the action being attempted
if (current_balance - requested_cost) is less than 0:
block action completely
do not deduct any credits
do not execute the action
return error: insufficient_credits with current balance and action cost
surface purchase prompt for Starter or upgrade prompt for Trial
else:
proceed with atomic deduction and execution

This check must be atomic — the balance check and deduction happen in a single database transaction. Two simultaneous requests cannot both pass the same balance check against the same insufficient balance.

BUG 2 — SOFT CAP HAS NO OVERAGE BILLING CODE

Professional and Enterprise tiers must NOT block at zero — they enter overage mode. The overage billing system must be built completely:

When balance goes below zero on SOFT_CAP tier:
Allow action to execute
Record deduction in main ledger with overage_flag: true
Record in overage_ledger: org_id, workspace_id, action_type, credits, dollar_cost at credits x 0.01, billing_cycle, timestamp
Update overage_accumulator for current cycle
If first overage transaction this cycle: notify org_owner via notification panel BILLING ALERT type, post to Org Operations Briefing Channel, send email via Resend with message: "You have entered overage mode. Additional credits cost $0.01 each. Current overage: X credits / $X.XX"

Overage accumulator visible in credits widget:
Show when org is in overage mode
Show: "Overage this cycle: X credits / $X.XX estimated charge"
Update in real time with every overage deduction

End of billing cycle overage charge fires on Stripe renewal webhook:
1. Sum all overage_ledger entries for closing cycle
2. If overage_credits greater than 0:
a. Calculate overage_dollars = overage_credits x 0.01
b. Create Stripe invoice item for overage_dollars
c. Charge org payment method
d. On Stripe payment success webhook: create overage_payment record in ledger with Stripe reference, reset overage accumulator to zero, allocate new cycle monthly credits

PART 6 — LEDGER INTEGRITY RULES — ENFORCE PERMANENTLY

Rule 1: Ledger is append-only. No record is ever edited or deleted. Corrections are made by adding new corrective entries with reason documented.

Rule 2: Every credit deduction must have a corresponding allocation transaction that preceded it in the same billing cycle. Deductions without allocations are orphaned transactions — flag immediately and investigate.

Rule 3: Every credit allocation must have either a Stripe payment reference or a documented system event reference. Allocations without references are unauthorized and must be flagged.

Rule 4: Ledger balance integrity check runs on every balance read:
calculated_balance = SUM(all allocation transactions) minus SUM(all standard deductions)
For SOFT_CAP tiers in overage: overage_balance = absolute value of negative balance, overage_dollars = overage_balance x 0.01
If calculated_balance does not match stored balance: log critical error immediately, alert platform_staff, lock credit deductions until discrepancy is resolved, never display incorrect balance to org_owner

Rule 5: Credits widget summary total must always equal sum of itemized usage breakdown. If these two numbers differ the widget is reading from two different sources — Single Source of Truth violation. Fix the data source. Never adjust the display to match.

Rule 6: Race condition prevention on all credit deductions. Credit check and deduction must be a single atomic database transaction. Two simultaneous requests against the same org balance must be serialized using database row-level locking on the credit ledger record for the org.
