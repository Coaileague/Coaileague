## PERMANENT SAFEGUARDS TO IMPLEMENT

### Safeguard 1: Workspace type classification
Add a `workspace_type` field to the workspaces table if it doesn't exist:
- `sandbox` — dev/test/demo workspaces, seed scripts allowed
- `production` — real business workspaces, seed scripts BLOCKED
- `platform` — CoAI support org, persistent, never reset

### Safeguard 2: Seed script workspace guard
Every seed script, training tool, and data reset function must check workspace type FIRST:

```typescript
// At the top of EVERY seed/reset/training function:
async function guardAgainstProduction(workspaceId: string) {
  const workspace = await getWorkspace(workspaceId);
  
  if (workspace.workspaceType === 'production') {
    throw new Error(
      `BLOCKED: Cannot run seed/reset/training operations on production workspace "${workspace.name}". ` +
      `This workspace contains real business data. Use a sandbox workspace instead.`
    );
  }
  
  if (workspace.workspaceType === 'platform') {
    throw new Error(
      `BLOCKED: Cannot run seed/reset/training operations on platform workspace "${workspace.name}". ` +
      `This is the CoAIleague support org and must remain persistent.`
    );
  }
  
  // Only sandbox workspaces reach this point
  console.log(`[Seed Guard] Workspace "${workspace.name}" is sandbox — proceeding.`);
}
```

### Safeguard 3: Production workspace creation flow
When a new org signs up in production:
- Their workspace is created with `workspace_type: 'production'`
- Zero seed data is inserted
- The workspace starts completely empty
- Bryan (or the org owner) adds their own clients, employees, and settings
- Trinity begins learning from scratch for that org

### Safeguard 4: Environment-level protection
```typescript
// In seed scripts:
if (process.env.NODE_ENV === 'production') {
  console.error('BLOCKED: Seed scripts cannot run in production environment.');
  process.exit(1);
}
```

### Safeguard 5: Seed script logging
Every time a seed script runs (even in dev), log:
- Which workspace it targeted
- What was deleted (table + count)
- What was created (table + count)
- Timestamp
- Who triggered it (user ID or 'system')

This creates an audit trail so if contamination happens again, you can trace exactly what script caused it and when.
