## SECTION 1 — FIX THE DESKTOP HEADER MENU (DO THIS FIRST)

### Problem
The desktop navigation/header menu is laggy, glitchy, and opens or closes at inappropriate times — often when the user didn't click it. This is the highest priority fix because it affects every page.

### Root Causes to Investigate and Fix

**1. Event listener conflicts**
- Check for duplicate `mouseenter`, `mouseleave`, `click`, and `blur` event listeners on the nav/dropdown components.
- Remove any redundant listeners. There should be ONE handler per event per element.

**2. Hover intent — replace with proper debounce**
- If the menu is triggered by hover, it likely lacks hover intent debounce.
- Add a minimum 150ms delay before opening on hover so accidental brushes don't trigger it.
- Add a 200ms grace period before closing so mouse movement between trigger and dropdown doesn't snap it shut.

**3. z-index and focus/blur issues**
- If the menu closes when clicking inside it, a `blur` event on the trigger is firing before the click registers on the child.
- Fix: use `mousedown` instead of `blur`, or add a `setTimeout(() => close(), 100)` with a cancel flag.

**4. Mobile touch events firing on desktop**
- Check for any `touchstart` or `touchend` listeners that may be bleeding into desktop mouse events via event bubbling.
- Separate touch and mouse handlers explicitly using media queries or `navigator.maxTouchPoints` checks.

**5. Animation jank**
- If the menu uses `display: none / block` toggling, replace with `opacity` + `pointer-events` + `transform: translateY(-4px)` transitions for GPU-accelerated smooth open/close.
- Use `will-change: transform, opacity` on the dropdown container.

**6. Re-render loops**
- If using React state for menu open/close, ensure the toggle function is not re-creating itself on every render. Wrap in `useCallback`.
- Check that the parent component is not re-rendering unnecessarily on scroll or resize, which would reset the menu state mid-interaction.

### Expected Result After Fix
- Menu opens only when intentionally clicked or hovered.
- Menu stays open when moving cursor from trigger to dropdown items.
- Menu closes cleanly when clicking outside or pressing Escape.
- Zero jank, zero flicker, smooth 150–200ms transition animation.
- Works correctly on both desktop and mobile without interference.

The current mobile dashboard is a plain grid of navigation tiles — functional but cold and informationless. The goal is a **living, breathing home screen** that greets the employee by name, shows them their money, their next shift, and their status — all without navigating anywhere.

## SECTION 4 — GLOBAL ANIMATION & POLISH

Apply across both mobile and desktop:

**Page load sequence (staggered):**
1. Header fades in (0ms)
2. Greeting banner slides up + fades in (100ms)
3. Earnings card slides up + fades in (200ms)
4. Quick actions slide up (300ms)
5. Grid tiles stagger in left-to-right, top-to-bottom (50ms each, starting at 400ms)
6. Activity feed fades in last (600ms)

**Micro-interactions:**
- Earnings amount: counter animation on load (count up from $0 to final value over 800ms)
- Progress bar: animate width from 0% to actual % over 600ms with easing
- Clock In button: pulse animation when employee is overdue to clock in (shift started > 10min ago)
- Notification badges: subtle bounce on first load if unread count > 0

**Performance requirements:**
- All animations use `transform` and `opacity` only (no layout-triggering properties)
- Use `will-change: transform` sparingly — only on animated elements
- Lazy-load activity feed (it's below the fold)
- Earnings data should load in parallel with page render, show skeleton loaders while fetching
- Skeleton loaders: dark gray shimmer animation on card placeholders

**Skeleton loader spec:**
```css
/* While earnings data is loading, show this */
background: linear-gradient(90deg, #161B22 25%, #21262D 50%, #161B22 75%);
background-size: 200% 100%;
animation: shimmer 1.5s infinite;
```

## SECTION 5 — DATA CONNECTIONS (READ ONLY)

All widgets read from existing tables. No new tables. No schema changes.

| Widget | Data Source | Query Logic |
|--------|------------|-------------|
| Earnings amount | timesheets + employee rate | SUM(hours_worked) × hourly_rate WHERE pay_period = current |
| Pay period dates | org settings / pay_periods table | Current active pay period for employee's org |
| Hours worked | timesheets | Approved entries this pay period |
| Progress bar | timesheets + schedules | hours_worked / total_scheduled_hours |
| Next shift | schedules | Next future shift for this employee |
| Current status | clock_events or time_entries | Latest clock in/out record |
| Activity feed | notifications or audit_log | Last 5–10 records for this employee |

If any of these tables don't exist by the exact name above, find the equivalent existing table — **do NOT create new ones**.
