**FIX 1 — CRITICAL — Invite acceptance race condition**

File: invite acceptance route

The current flow is read → check → write as three separate operations. Two simultaneous clicks on the same invite link both pass the check and both create user records.

Wrap the entire invite acceptance in a database transaction with a `SELECT FOR UPDATE` on the invite row as the first operation inside the transaction. If the invite status is already `accepted` when the lock is acquired, return 409 immediately without creating a user. The lock must be acquired before any user creation attempt. Only one request can hold the lock at a time — the second waits, then sees the already-accepted status and exits cleanly.

**FIX 2 — CRITICAL — Stripe webhook duplicate payment after restart**

File: `stripeWebhooks.ts`

The in-memory dedup cache resets on server restart. A Stripe webhook that fired just before restart and was processed will fire again after restart and pass the dedup check, writing a duplicate `payment_received` ledger entry.

Replace the in-memory dedup check with a DB-backed check using the existing `processed_stripe_events` table. Before processing any webhook event — check if the Stripe event ID exists in `processed_stripe_events`. If it does, return 200 immediately. If not, insert the event ID into `processed_stripe_events` BEFORE processing, inside a transaction. This makes dedup restart-safe. The in-memory LRU cache can remain as a fast-path layer on top but the DB check is the authoritative gate.
