Create: lib/ai-status/fallback-intelligence.ts

This module activates when all three AI models 
(GPT, Claude, Gemini) are unreachable. 
It does not simulate AI reasoning — 
it executes pre-defined deterministic 
logic for the highest-impact scenarios 
and serves cached responses for everything else.

Two-layer fallback architecture:

LAYER 1 — HARDCODED DECISION TREE
For CRITICAL and HIGH priority scenarios. 
These must work regardless of any model availability.

LAYER 2 — CACHED INTELLIGENCE
For NORMAL and LOW scenarios. 
Serves last-known-good responses and 
platform data lookups that don't require AI reasoning.

─────────────────────────────────────────────
LAYER 1 — HARDCODED DECISION TREE
─────────────────────────────────────────────

function handleFallbackRequest(
  input: string,
  context: ConversationContext,
  priority: Priority
): FallbackResponse

Emergency / Safety (always first check):
  Trigger: CRITICAL priority classification 
  OR safety keywords detected
  Action:
    1. Immediately respond with hardcoded 
       acknowledgment in session language:
       EN: "I hear you. Help is being 
           dispatched right now. 
           Stay with me. 
           [Emergency contact for workspace]"
       ES: "Te escucho. La ayuda está 
           en camino ahora mismo. 
           Quédate conmigo. 
           [Contacto de emergencia]"
    2. Fire notification to management 
       via existing notification system 
       (no AI required — direct DB write 
       + notification trigger)
    3. Create CRITICAL incident record 
       via ReportBot interface 
       (no AI required — direct API call)
    4. Log to trinity_helpai_command_bus 
       as ALERT type with status = pending
    5. Return hardcoded response immediately

Clock In / Clock Out:
  Trigger: clock-in or clock-out keywords 
  detected in either language
  Action:
    1. Route directly to ClockBot interface
    2. Execute clock action
    3. Return hardcoded confirmation:
       EN: "Got it — you're clocked [in/out] 
           at [timestamp]. Logged."
       ES: "Listo — registrado [entrada/salida] 
           a las [timestamp]. Confirmado."
    4. No AI reasoning required

Escalation Request:
  Trigger: "supervisor" "manager" 
  "escalate" "need help" + urgency signal
  Action:
    1. Fire notification to management 
       via existing notification system
    2. Log conversation context to 
       escalation queue
    3. Return hardcoded response:
       EN: "I've notified your supervisor. 
           Someone will be with you shortly. 
           Your request is logged."
       ES: "He notificado a tu supervisor. 
           Alguien estará contigo pronto. 
           Tu solicitud está registrada."

Incident Report Request:
  Trigger: "incident" "report" 
  "file" "document" keywords
  Action:
    1. Return hardcoded incident report 
       template in session language 
       with fill-in-the-blank fields
    2. Collect responses sequentially
    3. Compile into structured record
    4. File via existing Document Management 
       direct API (no AI required)
    5. Notify management via notification system

Platform Error / Technical Issue:
  Trigger: error keywords + 
  platform feature names
  Action:
    1. Log issue type to error queue
    2. Return hardcoded response:
       EN: "I've logged that issue. 
           Our team will review it. 
           If urgent, please contact 
           platform support directly."
    3. Alert platform support queue 
       via notification system

FALLBACK STATE MANAGER
─────────────────────────────────────────────

function detectModelAvailability(): 
  ModelStatus

Checks all three model API endpoints 
on a 30-second heartbeat.

Returns:
{
  gpt: 'online' | 'degraded' | 'offline',
  claude: 'online' | 'degraded' | 'offline',
  gemini: 'online' | 'degraded' | 'offline',
  overall: 'full' | 'partial' | 'fallback'
}

overall = 'full': all three reachable — 
  normal operation, no fallback
overall = 'partial': one or two models 
  reachable — Trinity routes away from 
  offline models, Graceful Degradation Mode 
  (built in prior implementation) active 
  for HelpAI
overall = 'fallback': all three unreachable — 
  Fallback Intelligence Layer activates 
  for both Trinity and HelpAI

State changes trigger:
- Trinity bar state update 
  (ACTIVE → FALLBACK or OFFLINE)
- HelpAI inline status update 
  (phrase library switches to FALLBACK set)
- Management notification via 
  existing notification system:
  "Trinity models offline — 
   fallback intelligence active. 
   Critical functions maintained."
- Log to system health record

On model restoration:
- Heartbeat detects recovery
- Queue of held requests processed 
  in priority order
- Trinity bar returns to ACTIVE state
- HelpAI phrase library returns to OPERATIONAL
- Management notification:
  "Trinity fully restored. 
   Processing queued requests now."
