**AGENT 2 — RAW SQL MIGRATION TO ORM**

*Mission: Replace raw SQL query strings in route and service files with type-safe Drizzle ORM calls, using the schema definitions Agent 1 is producing.*

This agent begins its work only after Agent 1 completes Step 5 and the new schema definitions are exported and importable. Agent 2 depends on those definitions to replace the raw SQL they describe.

**Step 1 — Raw SQL inventory.**

Scan every route file and every service file in the codebase. Produce a complete list of every raw SQL query string — every `db.execute(sql\`...\`)`, every `pool.query('...')`, every template literal SQL string, and every other raw SQL pattern. For each, record: the file and line number, the tables it touches, the columns it references, and whether it is a SELECT, INSERT, UPDATE, or DELETE operation.

**Step 2 — Classify each query.**

For each raw SQL query, classify it into one of three categories:

*Category A — Direct replacement:* The query is a simple SELECT, INSERT, UPDATE, or DELETE against a single table or a simple join. It can be directly replaced with an equivalent Drizzle query using the new schema definitions. This is the majority of queries.

*Category B — Complex query requiring careful migration:* The query uses advanced SQL features — window functions, complex aggregations, CTEs, or multi-level subqueries — that are difficult to express in Drizzle's query builder. These require careful migration with exact result verification.

*Category C — Keep as raw SQL with typed wrapper:* The query is so complex or performance-critical that replacing it with the ORM query builder would be counterproductive. These queries are kept as raw SQL but wrapped in a typed function that declares explicit input and output types, uses the canonical column names from the Name Registry, and is documented with a comment explaining why ORM replacement was not done.

**Step 3 — Replace Category A queries.**

For every Category A query: replace the raw SQL string with the equivalent Drizzle ORM call using the imported schema table and column definitions. The replacement must produce identical results to the raw SQL it replaces. Before removing the raw SQL, run both the original and the replacement against the ACME sandbox and confirm the results are identical. Then remove the raw SQL.

The canonical replacement patterns are:

```typescript
// SELECT with filter
const results = await db
  .select()
  .from(schemaTable)
  .where(eq(schemaTable.workspaceId, workspaceId));

// INSERT with return
const [created] = await db
  .insert(schemaTable)
  .values(data)
  .returning();

// UPDATE with filter
await db
  .update(schemaTable)
  .set(updateData)
  .where(eq(schemaTable.id, id));

// SELECT with relation join using with clause
const results = await db.query.schemaTable.findMany({
  where: eq(schemaTable.workspaceId, workspaceId),
  with: { relatedTable: true }
});
```

**Step 4 — Migrate Category B queries.**

For every Category B query: write the Drizzle equivalent carefully. Run both versions against the ACME sandbox with identical inputs. Compare outputs row by row and field by field. If the outputs match exactly, replace the raw SQL. If they do not match exactly after three migration attempts, escalate to Category C and wrap the raw SQL with a typed function.

**Step 5 — Wrap Category C queries.**

For every Category C query that is kept as raw SQL: wrap it in a typed function with explicit TypeScript input and output types. The function name must describe what the query does. Every column name referenced in the raw SQL string must match the canonical Name Registry established in Phase 1. Add a comment block explaining why this query was not migrated to the ORM. These typed wrappers are the permanent home for complex raw SQL — not inline in route handlers.

Example pattern:

```typescript
// COMPLEX QUERY — kept as raw SQL due to window function requirement
// Column names verified against canonical Name Registry 2026-03-22
// Input and output types are explicit — no any types permitted
async function getEmployeeOvertimeSummary(
  workspaceId: string,
  periodStart: Date,
  periodEnd: Date
): Promise<OvertimeSummaryRow[]> {
  const result = await db.execute(sql`
    SELECT 
      employee_id,
      SUM(regular_hours) as regular_hours,
      SUM(overtime_hours) as overtime_hours
    FROM shifts
    WHERE workspace_id = ${workspaceId}
    AND shift_date BETWEEN ${periodStart} AND ${periodEnd}
    GROUP BY employee_id
  `);
  return result.rows as OvertimeSummaryRow[];
}
```

**Step 6 — Verify zero raw SQL strings remain inline in route handlers.**

After all replacements and wrappers are applied: scan every route file and every service file. Confirm zero inline raw SQL strings remain. Every database operation is either a Drizzle ORM call or a call to a named typed wrapper function. Inline SQL strings in route handlers are permanently retired.
