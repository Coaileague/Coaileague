**PART 2 — CREDIT LEDGER SYSTEM**

The credit ledger is a financial record. It must be mathematically perfect and immutable.

**Ledger Architecture:**
- Every org has exactly one credit ledger — single source of truth
- Ledger operates as append-only transaction log — no record is ever edited or deleted
- Every credit event creates a new ledger entry — debit or credit, never a balance update alone
- Current balance always calculated from ledger transaction history — never stored as a mutable number that can drift
- Ledger entry contains — org ID, workspace ID, transaction type, credit amount, action that triggered it, timestamp, executing user or system, resulting balance after transaction
- Balance displayed in UI always reads from calculated ledger total — never from a cached or separate balance field that can desync

**Credit Allocation Events — all must create ledger entries:**
- Monthly credit reset on billing cycle — logs as credit allocation with period reference
- Credit purchase — logs as credit purchase with Stripe payment reference
- Plan upgrade bonus credits — logs as allocation with upgrade reference
- Manual credit adjustment by platform staff — logs with staff ID and reason

**Credit Deduction Events — all must create ledger entries:**
- Smart scheduling execution — deduct correct amount, log action type
- Scheduling session fee — deduct correct amount, log session reference
- AI contract negotiation — 30 credits per contract, log contract ID
- Payroll analysis — log credit amount and analysis scope
- Schedule analysis — log credit amount and analysis scope
- Notification sends — log per batch
- Any Trinity autonomous action — log action type and credit cost
- Any future premium action — framework must support new deduction types without architectural changes

**Ledger Integrity Rules:**
- Balance can never go below zero through deduction — system must check balance before executing any credit action
- If insufficient credits — action blocked, org notified immediately with current balance and purchase option
- Race condition prevention — credit check and deduction must be atomic transaction, two simultaneous actions cannot both pass a balance check and overdraw the ledger
- Monthly reset never overwrites purchased credits — only resets allocated monthly credits, purchased credits carry forward
- Purchased credits consumed after monthly allocation — monthly credits spent first, purchased credits are the overflow reserve

**PART 3 — CREDIT GATING AND ENFORCEMENT**

Every feature that costs credits must enforce the gate server side:

- Before executing any credit-costing action — check current balance atomically
- If balance sufficient — deduct and execute in same atomic transaction
- If balance insufficient — block execution immediately, return clear error with current balance and credit cost of attempted action
- UI must reflect insufficient credits state — button disabled, tooltip explaining credits needed, direct link to purchase more
- Trinity must never attempt a credit-costing action without first verifying balance
- Trinity must notify org owner proactively when balance drops below configurable threshold — default 20% remaining
- Low credit warning appears in notification panel, Org Operations Briefing Channel, and email
- Critical credit warning at 10% remaining — escalated notification type
- Zero credit state — all credit-gated automations pause, org notified, clear path to purchase and resume

**Credit Purchase Flow:**
- Org selects credit package
- Stripe payment intent created server side
- Payment confirmed by Stripe webhook — never by client confirmation alone
- On webhook confirmation — ledger entry created, balance updates, UI refreshes
- Failed payment — clear error, no ledger entry, no credit addition
- Purchase receipt generated and emailed via Resend
- Purchase history accessible to org owner in billing section

**PART 7 — STRIPE INTEGRATION VERIFICATION**

Stripe is the payment processor. Every integration point must be verified:

- Stripe webhook endpoint receiving all required event types — payment_intent.succeeded, payment_intent.failed, customer.subscription.updated, customer.subscription.deleted, invoice.payment_succeeded, invoice.payment_failed
- Webhook signature verification — every incoming webhook verified with Stripe signature before processing
- Webhook idempotency — same webhook event delivered twice must not create duplicate records or double charge
- Webhook failure handling — if processing fails, retry logic in place, alert to platform staff
- Test mode and production mode keys correctly separated — test keys never in production environment
- Stripe customer ID stored per org — every org has exactly one Stripe customer record
- Payment method storage — org can save payment method for automatic billing
- Failed subscription payment — retry logic, grace period, then correct access gating
- Stripe dashboard reconciliation — amounts in CoAIleague ledger match Stripe dashboard exactly

**PART 8 — WORKSPACE AND SESSION ISOLATION FOR BILLING**

Billing data is among the most sensitive data on the platform. Isolation must be absolute.

- Org A can never see, access, or affect Org B's credit ledger under any circumstance
- Billing API endpoints verify workspace ownership on every request — not just authentication, but authorization to that specific org's billing data
- Credit deductions only post to the workspace that triggered the action — no cross-workspace credit bleeding
- Stripe customer records mapped to org ID with verified ownership check on every billing operation
- Session tokens cannot be used to access billing endpoints of a different org
- Admin role inside an org cannot access billing — org owner only
- Platform staff billing access is completely separate system — never accessible through org-facing routes
- All billing endpoints rate limited independently from general API

**PART 9 — ABUSE, FRAUD, AND ATTACK PREVENTION**

The credit system and billing system are attack surfaces. Protect them:

**Excessive Token and Credit Usage Prevention:**
- Rate limit all AI action endpoints — maximum requests per minute per org
- Maximum credit spend per hour per org — configurable threshold, alert if exceeded
- Anomalous usage detection — if an org suddenly spends 10x their normal credit rate Trinity flags it to platform staff
- Automated credit farming prevention — detect and block patterns of triggering actions purely to consume credits without legitimate workflow purpose
- Per-action rate limits — scheduling analysis cannot be triggered more than X times per hour per org

**API and Endpoint Protection:**
- All billing and credit endpoints require authentication — no public access
- All billing and credit endpoints require correct org authorization — authenticated user must belong to the org whose billing they are accessing
- CSRF protection on all billing form submissions
- Input validation on all credit and billing inputs — no negative credit purchases, no zero amount invoices
- SQL injection prevention on all billing queries — parameterized queries only

**DDoS and Spam Prevention:**
- Rate limiting on all endpoints — global, per IP, per org, per user
- Exponential backoff on repeated failed payment attempts
- Repeated failed authentication on billing endpoints triggers lockout and alert
- Webhook endpoint rate limited and signature verified — cannot be spammed
- Credit purchase endpoint rate limited — cannot be hammered to create multiple Stripe payment intents

**Fraud Prevention:**
- Duplicate payment detection — same amount, same org, within short time window flagged for review
- Chargeback handling process — if Stripe chargeback received, access gated pending resolution
- Unusual billing pattern alerts — org purchases large credit package then immediately triggers maximum credit actions
