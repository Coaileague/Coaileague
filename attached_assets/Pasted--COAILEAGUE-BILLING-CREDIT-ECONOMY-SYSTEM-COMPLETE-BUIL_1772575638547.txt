# ═══════════════════════════════════════════════════
# PART 1: THE BUSINESS MODEL
# ═══════════════════════════════════════════════════

CoAIleague operates a **dual-revenue billing model** — two completely separate billing streams that appear on every org's monthly platform invoice.

## Stream A: Subscription + Credits (Platform Access + AI Actions)

- Org pays a fixed monthly subscription tier fee
- Subscription includes a monthly credit allotment
- Credits are consumed when Trinity or any AI feature TAKES A VALUABLE ACTION
- Watching/monitoring/scanning is ALWAYS FREE — credits only burn when Trinity DOES something
- When credits deplete, Trinity degrades gracefully to free database-only mode (Tier 1)
- Org can buy credit packs at any time to restore full capability

## Stream B: Financial Processing Fees (Real Dollar Charges)

- Charged as real dollar amounts, NOT credits
- Per-transaction fees for every invoice generated and every payroll employee processed
- Per-event fees for every QuickBooks auto-sync
- These accumulate throughout the month as line items on the org's platform bill
- This is how CoAIleague replaces QuickBooks/Gusto — by being the middleware processor/automator
- Priced competitively against QB ($6.50/emp) and Gusto ($6/emp) while providing MORE automation

## Stream C: Monthly Platform Bill (What The Org Actually Pays)

At the end of each billing cycle, the org receives ONE itemized invoice from CoAIleague:
1. Subscription tier fee (fixed monthly)
2. Per-employee overage charges (if they exceed tier's included employee count)
3. Invoice processing fees (count × per-invoice rate for their tier)
4. Payroll processing fees (employee-count × per-employee rate × number of runs)
5. QuickBooks sync fees (event count × per-sync rate)
6. Credit pack purchases made during the month
7. Add-on module fees (Financial Intelligence, Forecasting, Multi-Location)

**THE CARDINAL RULE: CoAIleague takes zero loss on any operation. Every action is either included in the subscription (with hard caps) or charged per occurrence. Every AI token cost is covered 100× or more by credit revenue. The platform profit margin on every tier exceeds 50% of the customer's savings.**

# ═══════════════════════════════════════════════════
# PART 2: WHY THE OLD PRICING IS BEING REPLACED
# ═══════════════════════════════════════════════════

## Problem 1: Credit Allocations Were Massively Oversaturated

Trinity scheduled a full month for an enterprise org and consumed only ~1,700 credits. Old tiers gave:
- Starter: 2,500 credits/mo → org uses ~500 = 5× surplus (zero upgrade pressure)
- Professional: 10,000 credits/mo → org uses ~3,000 = 3.3× surplus (never buy packs)
- Enterprise: 50,000 credits/mo → org uses ~8,000 = 6.25× surplus (absurd waste)

Credits were decoration, not a monetization lever. NEW allocations are right-sized based on real usage modeling.

## Problem 2: "Trinity Thought Stream powered by Claude" Listed as Separate Feature

Trinity IS the AI triad (Gemini + Claude + OpenAI) in orchestrated chain-of-command. There is no separate "Claude feature." Remove "Trinity Thought" from everywhere. All three models work together as Trinity. Period.

## Problem 3: "Premium Features Use 2× Credits" — Confusing and Wrong

Each feature should have its own credit cost based on: task difficulty + estimated token usage + convenience premium. A contract review (30 credits) already costs more than a GPS scan (1 credit). The multiplier is built into the per-action cost. Remove the "2× premium" concept entirely.

## Problem 4: Financial Processing Was Charged as Credits Instead of Dollars

Invoicing and payroll were listed as credit costs (6 credits per invoice, 2 credits per employee). But these are the platform's highest-value operations — they replace QuickBooks ($6.50/employee) and Gusto ($6/employee). Charging credits instead of real dollars for financial processing left massive revenue on the table. Financial processing is now a dollar-denominated transaction fee.

## Problem 5: Enterprise Was Underpriced at $5,500

Enterprise serves multi-state operators with 150-5,000+ guards (Securitas, Allied Universal, regional multi-state owners). These companies currently spend $35,000-120,000/month on the fragmented software stack CoAIleague replaces. Charging $5,500 captures less than 15% of their savings — leaving money on the table and undervaluing the platform. The correct price captures ~50% of savings while remaining an obvious ROI win.

## Problem 6: No Hard Caps Created Zero Upgrade Pressure

Without caps, a Starter org could process unlimited operations for a flat monthly fee. Hard caps on AI-powered features create natural tier boundaries — when an org consistently hits caps, upgrading becomes the obvious economic choice.

## 4B: Credit Lifecycle

```
MONTH START
  ├── Subscription credits deposited (tier allotment)
  ├── Carried-over credits from last month (max 10% of tier allotment)
  ├── Any purchased credit pack balances remaining (never expire)
  │
  ├── Throughout month: credits consumed per action
  │   DRAIN ORDER: subscription → carryover → purchased
  │
  ├── At 75% consumed: in-app notification to org owner
  ├── At 90% consumed: warning + "Buy Credits" upsell prompt
  ├── At 100% consumed: Trinity degrades to Tier 1 (free DB mode)
  │   ├── Org can still manually trigger actions (warned, auto-buys pack if card on file)
  │   └── Org can buy a credit pack at any time to restore full capability
  │
MONTH END
  ├── Calculate unused subscription + carryover credits
  ├── Carry over: min(unused_sub + unused_carryover, 10% of tier allotment)
  │   Example: Professional tier allotment = 3,500
  │            Max carryover = 350 credits
  │            If 800 unused subscription + carryover → only 350 carry over
  ├── Expired credits (above carryover cap) → Platform Credit Pool
  ├── Purchased pack credits: UNCHANGED (never expire, carry forward forever)
  └── New subscription credits deposited for next month
```

## 4C: Platform Credit Pool

Expired credits from all orgs flow into the Platform Credit Pool. This pool costs CoAIleague nothing (orgs already paid through subscriptions) and funds:

- **Bug compensation**: Refund credits when platform issues affect an org
- **Loyalty bonuses**: Anniversary rewards, long-term customer appreciation
- **Support operations**: Credits used when support agents troubleshoot org issues
- **Promotional credits**: New feature launch incentives
- **Platform maintenance**: Operations that benefit all orgs

This is a self-funding goodwill fund. CoAIleague never dips into revenue to compensate customers.

## 5A: How Financial Processing Fees Work

Financial processing fees are real dollar charges — NOT credits. They accumulate throughout the month and appear as line items on the org's CoAIleague platform bill.

**When an invoice is generated:**
1. Trinity or the org creates an invoice from approved time entries
2. Invoice is committed to database with line items, tax, totals
3. A `financial_processing_fee` record is created with the tier's per-invoice rate
4. Fee accumulates until the monthly platform bill

**When payroll is processed:**
1. Manager approves the payroll gate
2. System processes payroll entries for N employees
3. A `financial_processing_fee` record is created: tier rate × N employees
4. Fee accumulates until the monthly platform bill

**When QuickBooks syncs (Professional+ only):**
1. Auto-sync fires (invoice sync, payment sync, account sync)
2. A `financial_processing_fee` record is created with the tier's per-sync rate
3. Fee accumulates until the monthly platform bill

## 6A: The Concept

Enterprise customers operate across multiple states. A company like "National Guard Services" might have:
- **Mother Org**: National Guard Services (headquarters — manages billing, credits, company-wide settings)
- **Sub-Org: Texas Division** — 80 employees, 8 clients, TX labor laws, TX tax withholding
- **Sub-Org: California Division** — 60 employees, 5 clients, CA labor laws (daily OT!), CA tax
- **Sub-Org: Florida Division** — 50 employees, 6 clients, FL labor laws, FL tax
- **Sub-Org: New York Division** — 40 employees, 4 clients, NY labor laws, NY tax
- **Sub-Org: Illinois Division** — 30 employees, 3 clients, IL labor laws, IL tax
- **Total: 260 employees across 5 states, 26 clients**

```

  12,000 base + (260 × 10) = 14,600 credits/month

  New York payroll: 2 runs × 40 emp × $3.50 = $280
```

## 7A: The Rule

Client portals and state auditor portals are **FREE to the client/auditor**. The org that provisions their access pays for any AI-powered actions those external users trigger. The cost is charged against the org's credit pool.

This is a sound business model: the org provides the portal as a value-add to their clients and regulators. It increases client retention (clients love real-time visibility) and reduces audit burden (auditors can self-serve). The cost to the org is minimal — most portal activity is just reading data (zero credits).

## 9A: Credit Service

```typescript
// server/services/creditService.ts

class CreditService {

  /**
   * Get the correct credit balance workspace.
   * If this workspace is a sub-org, return the mother org's workspace_id.
   * If standalone or mother org, return this workspace_id.
   */
  private async getCreditPoolWorkspaceId(workspaceId: string): Promise<string> {
    const workspace = await getWorkspace(workspaceId);
    if (workspace.parentWorkspaceId) {
      return workspace.parentWorkspaceId; // Sub-org → use mother's pool
    }
    return workspaceId; // Standalone or mother org → use own pool
  }

  /**
   * Calculate total monthly credits for a workspace (including per-employee scaling)
   */
  async calculateMonthlyCredits(workspaceId: string): Promise<number> {
    const sub = await getSubscription(workspaceId);
    const tier = await getTier(sub.tierId);
    const empCount = await countAllEmployees(workspaceId); // Includes sub-orgs for mother
    return tier.monthlyCreditsBase + (tier.creditsPerEmployee * empCount);
  }

  /**
   * Consume credits for an action.
   * DRAIN ORDER: subscription → carryover → purchased
   * Uses SELECT FOR UPDATE to prevent race conditions.
   */
  async consumeCredits(
    workspaceId: string,      // The workspace performing the action
    amount: number,
    featureKey: string,
    referenceId?: string,
    referenceType?: string,
    triggeredBy: string = 'system'
  ): Promise<{ success: boolean; creditsRemaining: number; transactionId: string }> {

    const poolWorkspaceId = await this.getCreditPoolWorkspaceId(workspaceId);
    const isSubOrg = poolWorkspaceId !== workspaceId;

    return await db.transaction(async (tx) => {
      // Lock the balance row
      const [bal] = await tx.select()
        .from(creditBalances)
        .where(eq(creditBalances.workspaceId, poolWorkspaceId))
        .for('update');

      if (!bal) throw new Error(`No credit balance for workspace ${poolWorkspaceId}`);

      const total = bal.subscriptionCredits + bal.carryoverCredits + bal.purchasedCredits;

      if (total < amount) {
        return { success: false, creditsRemaining: total, transactionId: '' };
      }

      // Drain: subscription → carryover → purchased
      let remaining = amount;

      const newSub = Math.max(0, bal.subscriptionCredits - remaining);
      remaining = Math.max(0, remaining - bal.subscriptionCredits);

      const newCarry = remaining > 0 ? Math.max(0, bal.carryoverCredits - remaining) : bal.carryoverCredits;
      remaining = remaining > 0 ? Math.max(0, remaining - bal.carryoverCredits) : 0;

      const newPurch = remaining > 0 ? Math.max(0, bal.purchasedCredits - remaining) : bal.purchasedCredits;

      // Update balance
      await tx.update(creditBalances).set({
        subscriptionCredits: newSub,
        carryoverCredits: newCarry,
        purchasedCredits: newPurch,
        updatedAt: new Date()
      }).where(eq(creditBalances.workspaceId, poolWorkspaceId));

      const newTotal = newSub + newCarry + newPurch;

      // Record transaction
      const [txRecord] = await tx.insert(creditTransactions).values({
        workspaceId: poolWorkspaceId,
        subOrgWorkspaceId: isSubOrg ? workspaceId : null,
        transactionType: 'action_debit',
        amount: -amount,
        balanceAfter: newTotal,
        sourceType: newSub < bal.subscriptionCredits ? 'subscription' :
                    newCarry < bal.carryoverCredits ? 'carryover' : 'purchased',
        featureKey,
        referenceId,
        referenceType,
        triggeredBy,
        description: `${featureKey}: ${amount} credits consumed${isSubOrg ? ` (sub-org: ${workspaceId})` : ''}`
      }).returning();

      // Check degradation thresholds
      const monthlyCredits = await this.calculateMonthlyCredits(poolWorkspaceId);
      const pctRemaining = newTotal / monthlyCredits;

      if (pctRemaining <= 0.25) {
        const level = pctRemaining <= 0 ? 'depleted' :
                      pctRemaining <= 0.10 ? 'very_low' : 'low';
        await this.sendCreditWarning(poolWorkspaceId, level, newTotal, monthlyCredits);
      }

      return { success: true, creditsRemaining: newTotal, transactionId: txRecord.id };
    });
  }

  /**
   * Monthly credit cycle: deposit new credits, calculate carryover, expire excess
   */
  async processMonthlyReset(workspaceId: string): Promise<void> {
    await db.transaction(async (tx) => {
      const [bal] = await tx.select()
        .from(creditBalances)
        .where(eq(creditBalances.workspaceId, workspaceId))
        .for('update');

      const sub = await this.getSubscription(workspaceId);
      const tier = await this.getTier(sub.tierId);
      const monthlyCredits = await this.calculateMonthlyCredits(workspaceId);
      const maxCarryover = Math.floor(monthlyCredits * Number(tier.maxCreditCarryoverPct));

      const unusedSubAndCarry = bal.subscriptionCredits + bal.carryoverCredits;
      const carryover = Math.min(unusedSubAndCarry, maxCarryover);
      const expired = Math.max(0, unusedSubAndCarry - carryover);

      // Expire excess to platform pool
      if (expired > 0) {
        // Get current pool balance
        const [poolBal] = await tx.select({ total: sql`COALESCE(SUM(amount), 0)` })
          .from(platformCreditPool);
        const poolBalance = Number(poolBal?.total || 0);

        await tx.insert(platformCreditPool).values({
          transactionType: 'expired_credits_deposit',
          amount: expired,
          sourceWorkspaceId: workspaceId,
          balanceAfter: poolBalance + expired,
          description: `${expired} credits expired from ${workspaceId}`
        });

        await tx.insert(creditTransactions).values({
          workspaceId,
          transactionType: 'expiration_debit',
          amount: -expired,
          balanceAfter: monthlyCredits + carryover + bal.purchasedCredits,
          sourceType: 'subscription',
          description: `${expired} unused credits expired to platform pool`
        });
      }

      // Deposit new credits
      await tx.update(creditBalances).set({
        subscriptionCredits: monthlyCredits,
        carryoverCredits: carryover,
        lastRefillAt: new Date(),
        updatedAt: new Date()
      }).where(eq(creditBalances.workspaceId, workspaceId));

      await tx.insert(creditTransactions).values({
        workspaceId,
        transactionType: 'subscription_deposit',
        amount: monthlyCredits,
        balanceAfter: monthlyCredits + carryover + bal.purchasedCredits,
        sourceType: 'subscription',
        description: `Monthly deposit: ${monthlyCredits} credits`
      });

      // Reset usage caps
      const newCycle = format(new Date(), 'yyyy-MM');
      await tx.insert(usageCaps).values({
        workspaceId,
        billingCycle: newCycle
      }).onConflictDoUpdate({
        target: [usageCaps.workspaceId, usageCaps.billingCycle],
        set: { aiScheduledShiftsUsed: 0, gpsVerificationsUsed: 0,
               analyticsReportsUsed: 0, contractReviewsUsed: 0,
               docchatSessionsUsed: 0, botInteractionsToday: 0,
               updatedAt: new Date() }
      });
    });
  }
}
```

```typescript
  /**
   * Check if an action is within the org's tier cap.
   * Within cap → free (no credits). Over cap → costs credits.
   * Cap = -1 → unlimited (always free). Cap = 0 → feature unavailable.
   */
  async checkAndIncrement(
    workspaceId: string,
    capType: string,           // e.g., 'ai_scheduled_shifts'
    creditCostIfOverCap: number
  ): Promise<{ allowed: boolean; withinCap: boolean; creditCost: number }> {
    // For sub-orgs, get the mother org's tier
    const poolWorkspaceId = await creditService.getCreditPoolWorkspaceId(workspaceId);
    const sub = await getSubscription(poolWorkspaceId);
    const tier = await getTier(sub.tierId);

    const cap = tier[`cap_${capType}`];

    if (cap === 0) return { allowed: false, withinCap: false, creditCost: 0 };
    if (cap === -1) {
      await this.incrementUsage(poolWorkspaceId, capType);
      return { allowed: true, withinCap: true, creditCost: 0 };
    }

    const cycle = format(new Date(), 'yyyy-MM');
    const usage = await this.getUsage(poolWorkspaceId, cycle);
    const currentUsed = usage[`${capType}_used`] || 0;

    if (currentUsed < cap) {
      await this.incrementUsage(poolWorkspaceId, capType);
      return { allowed: true, withinCap: true, creditCost: 0 };
    }

    // Over cap — check credits
    const balance = await getCreditBalance(poolWorkspaceId);
    if (balance.totalCredits < creditCostIfOverCap) {
      return { allowed: false, withinCap: false, creditCost: creditCostIfOverCap };
    }

    await this.incrementUsage(poolWorkspaceId, capType);
    return { allowed: true, withinCap: false, creditCost: creditCostIfOverCap };
  }
}
```

## 9C: Financial Processing Fee Service

```typescript
// server/services/financialProcessingService.ts

class FinancialProcessingService {
  async recordFee(
    workspaceId: string,
    feeType: 'invoice_generation' | 'payroll_processing' | 'quickbooks_sync',
    referenceId: string,
    referenceType: string,
    employeeCount?: number
  ): Promise<void> {
    // Get the billing workspace (mother org or standalone)
    const workspace = await getWorkspace(workspaceId);
    const billingWorkspaceId = workspace.parentWorkspaceId || workspaceId;

    const sub = await getSubscription(billingWorkspaceId);
    const tier = await getTier(sub.tierId);

    // Idempotency check
    const existing = await db.select().from(financialProcessingFees)
      .where(and(
        eq(financialProcessingFees.referenceId, referenceId),
        eq(financialProcessingFees.feeType, feeType),
        eq(financialProcessingFees.billingCycle, format(new Date(), 'yyyy-MM'))
      )).limit(1);

    if (existing.length > 0) return; // Already recorded

    let amountCents: number;
    let perUnitRate: number;

    switch (feeType) {
      case 'invoice_generation':
        amountCents = tier.invoiceFeeCents;
        perUnitRate = tier.invoiceFeeCents;
        break;
      case 'payroll_processing':
        amountCents = tier.payrollFeeCents * (employeeCount || 1);
        perUnitRate = tier.payrollFeeCents;
        break;
      case 'quickbooks_sync':
        amountCents = tier.qbSyncFeeCents;
        perUnitRate = tier.qbSyncFeeCents;
        break;
    }

    await db.insert(financialProcessingFees).values({
      workspaceId,
      motherOrgWorkspaceId: workspace.parentWorkspaceId || null,
      feeType,
      amountCents,
      referenceId,
      referenceType,
      employeeCount: employeeCount || null,
      perUnitRateCents: perUnitRate,
      billingCycle: format(new Date(), 'yyyy-MM')
    });
  }
}
```

# ═══════════════════════════════════════════════════
# PART 12: RACE CONDITION & INTEGRITY PROTECTIONS
# ═══════════════════════════════════════════════════

## 12A: Credit Balance Mutations
ALL credit balance changes MUST use `SELECT ... FOR UPDATE` within a database transaction. No optimistic updates. No concurrent reads without locks.

## 12B: Double-Charge Prevention
Every credit deduction records a `reference_id` + `feature_key`. Before deducting, check if a transaction exists for this combination. If so, skip (idempotent).

## 12C: Financial Fee Deduplication
Before recording a fee, check for existing `reference_id` + `fee_type` + `billing_cycle`. If exists, skip.

## 12D: Platform Bill Idempotency
Before generating, check for existing `workspace_id` + `billing_cycle`. If draft exists, update it. If sent/paid exists, skip entirely.

## 12E: Credit Pack Purchase Atomicity
Stripe webhook fires → check if `payment_intent_id` already processed → if not, add credits + record transaction → if yes, skip.
