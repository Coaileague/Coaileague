**Phase 49 — Notification Preferences and Communication Template Management**

Users must control how they are notified. A security officer who is asleep after a night shift does not want SMS alerts for low-priority events.

**Notification preferences per user:**
Each user configures: preferred channel per notification category (email, SMS, push, in-app, or none), quiet hours (no push or SMS between defined times), language preference (English or Spanish — feeds Phase 32), digest preference (receive individual notifications vs daily digest for low-priority types). Preferences stored per user per workspace — the same user in different workspaces can have different preferences.

**Channel fallback chain:**
When a user's preferred channel is unavailable NDS falls back to the next configured channel. If all channels are disabled for a notification category the notification is still stored in-app — never completely dropped for critical types. Critical notifications (duress alert, payroll run failure, compliance violation) override quiet hours — always delivered immediately regardless of preferences.

**Communication template management:**
Platform staff and org_owner can customize notification templates: subject lines, email body content, SMS message content, push notification title and body. Templates support merge fields. Template changes versioned — previous versions retained for audit. Default templates in both English and Spanish. Custom templates per workspace override defaults.

**Template management UI:**
Clean interface showing all notification types, current template per channel, preview function, edit history, restore to default option. Role-gated to org_owner and platform staff.

**Phase 50 — Outbound Webhook System and Integration Framework**

Every B2B SaaS platform needs a general webhook system. Security companies use dozens of tools — scheduling software, HR systems, accounting packages beyond QuickBooks, insurance portals. CoAIleague must be able to push events to any external system.

**Webhook configuration:**
Org_owner can register webhook endpoints: URL, secret for HMAC signature, event types to subscribe to, active/inactive status. Multiple webhooks per workspace supported. Webhook test button — sends a sample payload to verify the endpoint is receiving correctly.

**Event catalog:**
Every platform event that can trigger a webhook: officer_activated, officer_deactivated, shift_created, shift_assigned, clock_in, clock_out, calloff_submitted, incident_created, invoice_generated, invoice_paid, payroll_run_completed, document_executed, compliance_violation_detected, license_expiring. Payload per event is structured JSON with: event_type, workspace_id, timestamp, entity_id, entity_type, and relevant data fields.

**Delivery:**
Webhooks delivered with HMAC-SHA256 signature header. Retry with exponential backoff on failure. Dead letter after max retries — org_owner notified. Delivery log per webhook endpoint — org_owner can see recent deliveries, status, and response codes. Webhook delivery tracked in notification_deliveries with channel: webhook.

**Trinity integration:**
Trinity can query webhook delivery health and manually trigger a retry for a failed webhook via registered action.
