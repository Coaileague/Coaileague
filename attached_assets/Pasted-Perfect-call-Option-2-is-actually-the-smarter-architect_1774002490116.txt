SECTION 3 — TRINITY LICENSE VERIFICATION (Option 2 Architecture)

When an applicant submits with has_license = true and uploads a screenshot:

Trigger a Trinity action called: VERIFY_LICENSE_SCREENSHOT

Trinity does the following using AI image analysis:
1. Extract: license number, full name, license type, expiration date, 
   issuing state, and status text visible on the document
2. Cross-check extracted name against applicant submitted name 
   (flag mismatch)
3. Cross-check extracted license number against submitted license number 
   (flag mismatch)
4. Check expiration date — if expired, auto-flag and note expiration date
5. Check for visual anomalies: pixelation inconsistencies, mismatched 
   fonts, suspicious alterations (flag if detected)
6. Produce a verification_result object:
   {
     extracted_name, extracted_license_number, extracted_expiration,
     extracted_status, name_match (boolean), number_match (boolean),
     is_expired (boolean), anomaly_detected (boolean),
     confidence_score (0-100), verification_notes (text summary),
     verified (boolean — true only if all checks pass and 
     confidence >= 80)
   }
7. Write result back to applicant record fields: license_verified, 
   license_verification_notes
8. If verified = false for any reason, move applicant to 
   pipeline_stage = 'pre_screened' with a flag for management review 
   rather than auto-disqualifying — let management make final call
9. If verified = true, advance applicant to 
   pipeline_stage = 'interview_scheduled' and trigger ChatDock 
   interview invite

SECTION 4 — TRINITY INTERVIEW VIA CHATDOCK

When pipeline_stage = 'interview_scheduled':

1. Trinity sends applicant an email (via Resend) with a unique 
   ChatDock invite link for their interview session
2. Interview session opens in ChatDock as a dedicated thread under 
   Trinity's orchestration — tagged as interview_session type
3. Trinity uses the role-appropriate question_set for that job posting
4. For each question:
   - Trinity asks the question in conversational tone
   - Waits for applicant response
   - Scores the response against the weight and category defined 
     in the question set
   - Asks natural follow-up if answer is vague or incomplete 
     (max 1 follow-up per question)
5. After all questions complete:
   - Trinity calculates interview_score (weighted average 0-100)
   - Calculates liability_score based on:
     * License verified status (pass/fail)
     * Scenario question responses (use of force, chain of command)
     * Prior termination or incident disclosures
     * Document anomaly flags
   - Generates trinity_summary: 2-3 paragraph written assessment 
     of the applicant in plain English for management
   - Updates applicant record with all scores and summary
   - Moves pipeline_stage to 'interview_complete'
   - Notifies management via platform notification

Default question sets to seed on install (one per role type):
Each set should have 8-10 questions covering:
availability/scheduling, prior experience, reason for leaving last 
employer, situational judgment (2-3 scenario questions), 
physical requirements acknowledgment, license/compliance awareness

SECTION 6 — TRINITY DEMAND TRIGGER (auto job posting)

Trinity monitors three signals on a nightly basis per workspace:

1. SCHEDULING_GAP: Any shift in the next 14 days that has been 
   posted for open coverage more than 48 hours without being filled
2. CONTRACT_START: Any new client contract with a start date within 
   30 days that requires headcount not yet assigned
3. DISCIPLINARY_TERMINATION: Any employee whose status was changed 
   to 'terminated' with reason containing 'disciplinary' in the 
   last 7 days

When any signal fires:
- Trinity drafts a job_posting record with auto_generated = true 
  and the appropriate demand_trigger value
- Status is set to 'draft' — NOT published automatically
- Management receives a platform notification: 
  "Trinity has drafted a new job posting for [position] based on 
  [reason]. Review and publish when ready."
- Management approves with one click → status flips to 'active' 
  and posting goes live on the public board
