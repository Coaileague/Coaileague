## AUDIT A: NEW ORG SIGNUP → FIRST VALUE — THE CRITICAL PATH

A new security company owner finds CoAIleague, signs up, and tries to get value. Trace EVERY step of this journey in the actual code and find where it breaks.

### A.1 — Marketing Site → Signup

```
VERIFY:
[ ] Landing page loads on mobile AND desktop (not just desktop)
[ ] All CTAs (buttons, links) actually navigate to signup
[ ] SEO meta tags present (title, description, og:image) — important for Google/social sharing
[ ] Page loads fast (not waiting on massive JS bundle before anything renders)
[ ] Site is server-side rendered or has pre-rendered HTML for SEO crawlers
    → If it's pure client-side React with no SSR, Google may not index it properly
    → This is a MAJOR issue for discoverability
[ ] No console errors on landing page
[ ] Pricing page loads and displays all tiers correctly
[ ] "Start Free Trial" or "Sign Up" button works from every page
```

### A.2 — Account Creation

```
VERIFY THE SIGNUP FLOW:
[ ] Email + password registration works
[ ] Email validation (proper format, not disposable emails if you filter those)
[ ] Password strength requirements enforced
[ ] Duplicate email handled gracefully (not a server error)
[ ] Confirmation email sent (via Resend)
[ ] Confirmation email actually arrives (check Resend deliverability)
[ ] Email confirmation link works and activates account
[ ] OAuth signup works (Google, Microsoft) if offered
[ ] After signup → user lands on onboarding, NOT a blank dashboard

EDGE CASES THAT WILL HAPPEN:
[ ] User signs up → closes browser → comes back later → can they still confirm?
[ ] User signs up with wrong email → can they change it?
[ ] Confirmation link expires → is there a resend option?
[ ] User hits signup twice quickly → duplicate account created? (race condition)
[ ] User signs up on mobile → entire flow works on small screen?
[ ] What if Resend is down? Does signup silently fail or queue the email?
```

### A.3 — Onboarding Flow (First 10 Minutes = Make or Break)

```
THIS IS WHERE MOST SAAS PLATFORMS LOSE CUSTOMERS.

[ ] Onboarding wizard/steps exist and guide the user
[ ] Step 1: Company info (name, address, state, license number)
[ ] Step 2: Invite first employees OR add them manually
[ ] Step 3: Add first client/site
[ ] Step 4: Create first schedule or shift
[ ] Step 5: Connect integrations (QuickBooks, Stripe billing)

FOR EACH STEP VERIFY:
[ ] Can be completed on mobile
[ ] Can be skipped and returned to later
[ ] Progress is saved if user leaves mid-step
[ ] Validation errors are clear (not generic "something went wrong")
[ ] Back button doesn't lose data
[ ] Step completion triggers the correct database records

AFTER ONBOARDING:
[ ] Dashboard shows relevant data (not empty state with no guidance)
[ ] Empty states have helpful prompts ("Add your first employee to get started")
[ ] Trinity chat is accessible and responsive
[ ] User knows what to do next (not staring at blank screen)
```

### A.4 — Subscription & Payment (Stripe)

```
VERIFY THE MONEY PATH:
[ ] Free trial starts automatically (if offered)
[ ] Trial duration is correct and enforced
[ ] Trial expiration warning emails sent at right intervals
[ ] Plan selection page shows all tiers with correct pricing
[ ] Stripe checkout flow works (card entry, processing, confirmation)
[ ] Successful payment → account upgraded to paid tier immediately
[ ] Failed payment → clear error message, account not upgraded
[ ] Webhook from Stripe received and processed correctly
[ ] Subscription status stored in database AND checked on feature access
[ ] Downgrade flow works (features restricted appropriately)
[ ] Cancellation flow works (what happens to their data?)
[ ] Reactivation after cancellation works

STRIPE EDGE CASES:
[ ] Card declined → user can try another card
[ ] 3D Secure (SCA) challenge required → flow handles it
[ ] Webhook arrives before redirect completes (race condition)
[ ] Customer disputes charge → chargeback webhook handled
[ ] Proration on mid-cycle plan change calculated correctly
[ ] Multiple workspaces under same Stripe customer?
```

## AUDIT B: MULTI-TENANT ISOLATION — THE #1 PRODUCTION KILLER

If Org A can see Org B's data, the platform is dead on arrival.

### B.1 — Data Isolation Verification

```
FOR EVERY API ENDPOINT, VERIFY:
[ ] Query includes workspace_id / org_id filter
[ ] User can only access data from their own workspace
[ ] A user in Org A cannot craft a request to access Org B's data
[ ] Even if they guess/know another org's ID

TEST THIS:
1. Create two test orgs with different data
2. Log in as user from Org A
3. Try to access Org B's:
   - Employees (GET /api/employees?workspaceId=orgB_id)
   - Clients
   - Shifts/schedules  
   - Invoices
   - Time entries
   - Chat messages
   - Documents
   
If ANY of these return Org B's data → CRITICAL security vulnerability
```

### B.2 — Cross-Tenant Bleed Points

```
CHECK THESE SPECIFIC RISK AREAS:
[ ] WebSocket rooms — can a user join another org's chat room?
[ ] Real-time updates — do push events leak to wrong org?
[ ] File uploads — are files stored with org isolation (not shared bucket)?
[ ] Reports — does report generation scope to requesting org?
[ ] Search — does search index scope to user's org?
[ ] Trinity AI — does Trinity's context include ONLY the requesting org's data?
[ ] Admin panels — does platform admin view properly isolate org data?
[ ] Email notifications — do emails reference correct org branding/info?
```

## AUDIT E: ROLE-BASED ACCESS CONTROL (RBAC) — CAN THE WRONG PERSON SEE THE WRONG THING?

```
ROLE HIERARCHY:
Platform Admin > Owner > Co-Owner > Manager > Supervisor > Employee

FOR EACH ROLE, VERIFY THEY CAN ONLY ACCESS WHAT THEY SHOULD:

PLATFORM ADMIN:
[ ] Can access all orgs (for support)
[ ] Can NOT modify org data without audit trail
[ ] Admin routes are completely separate from user routes

OWNER:
[ ] Full access to their org
[ ] Can manage billing/subscription
[ ] Can manage all employees, clients, schedules
[ ] Can access all reports and analytics
[ ] Can configure org settings

CO-OWNER:
[ ] Same as owner EXCEPT billing? (define boundary)
[ ] Can they invite other admins?

MANAGER:
[ ] Can manage schedules and shifts
[ ] Can approve timesheets
[ ] Can view (but not create?) invoices
[ ] Cannot access billing/subscription
[ ] Cannot delete employees

SUPERVISOR:
[ ] Can view team schedules
[ ] Can approve time entries for their team
[ ] Cannot access financial data
[ ] Cannot modify client contracts

EMPLOYEE:
[ ] Can ONLY see their own schedule, time entries, pay stubs
[ ] Cannot see other employees' data
[ ] Cannot access admin features
[ ] Cannot see client billing information
[ ] Cannot modify schedules

TEST: Log in as each role and try to access pages/endpoints they shouldn't.
Any access that succeeds where it shouldn't → CRITICAL bug.
```

## AUDIT F: WHAT HAPPENS WHEN THINGS GO WRONG?

Production isn't about the happy path. It's about when things break.

### F.1 — Database Goes Down

```
[ ] App shows maintenance page (not stack trace)
[ ] Active users see graceful error (not white screen)
[ ] Data entered just before crash → was it saved? (transaction integrity)
[ ] When DB comes back → app recovers without manual intervention
[ ] No orphaned records from partial writes during crash
```

### F.2 — Third-Party Service Goes Down

```
STRIPE DOWN:
[ ] Signup still works (just delays payment)
[ ] Existing users unaffected
[ ] Webhook queue doesn't lose events

RESEND DOWN:
[ ] Email-dependent flows gracefully degrade
[ ] Emails queued for retry (not lost)
[ ] User not told "email sent" when it wasn't

QUICKBOOKS DOWN:
[ ] Sync failures queued for retry
[ ] Data entry still works (just doesn't sync)
[ ] No duplicate sync when QB comes back

AI MODELS DOWN (Gemini/Claude/OpenAI):
[ ] Fallback chain activates
[ ] If ALL models down → graceful message (not error)
[ ] Non-AI features still work normally
[ ] Queued AI requests don't pile up and explode when service returns
```

### F.3 — User Errors

```
[ ] User deletes something important → undo or soft delete?
[ ] User enters clearly wrong data (negative hours, $0 pay rate) → validation catches it?
[ ] User navigates to invalid URL → 404 page (not blank screen)
[ ] User's session expires while filling long form → data preserved on re-login?
[ ] Two admins edit same record simultaneously → conflict resolution?
```

## AUDIT G: SCALABILITY — WILL IT WORK WITH 10 ORGS? 100? 1000?

```
PERFORMANCE CHECKS:
[ ] Dashboard load time with 100 employees → under 2 seconds?
[ ] Schedule view with 500 shifts in a month → renders without lag?
[ ] Chat with 1000+ messages in history → virtualized/paginated?
[ ] Employee list with 200+ employees → paginated, searchable?
[ ] Report generation for 3 months of data → doesn't timeout?
[ ] Concurrent users: 50 users hitting API simultaneously → no 500 errors?

DATABASE:
[ ] Indexes on all frequently queried columns (workspace_id, employee_id, date ranges)
[ ] N+1 query problems → eager loading where needed
[ ] Large table queries use LIMIT/OFFSET or cursor pagination
[ ] Connection pooling configured (not opening new connection per request)

WEBSOCKETS:
[ ] Socket server handles 100+ simultaneous connections
[ ] Room join/leave doesn't leak memory
[ ] Heartbeat/ping configured for stale connection cleanup
```
