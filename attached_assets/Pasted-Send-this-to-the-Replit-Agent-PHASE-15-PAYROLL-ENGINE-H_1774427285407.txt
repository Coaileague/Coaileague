**STEP 2 — PAY PERIOD BOUNDARY VERIFICATION**

Pay period boundaries must be mathematically precise:
- Weekly, bi-weekly, and semi-monthly period types all calculate correctly
- Bi-weekly UTC fix confirmed still intact from Phase 11 — verify with grep
- No gap between consecutive periods — every hour belongs to exactly one period
- No overlap between consecutive periods — same hour cannot appear in two periods
- `assertNoPeriodOverlap` DB guard confirmed active on every payroll run creation
- Period boundaries displayed correctly in UI — officers and managers see the same period dates
- Late-approved time entries correctly assigned to the period they belong to — not silently dropped

---

**STEP 3 — HOURS AGGREGATION VERIFICATION**

Every confirmed hour must flow into payroll correctly:
- Only supervisor-approved time entries are included in payroll runs — pending or disputed entries excluded
- Clock-in and clock-out timestamps used for actual hours — not scheduled shift hours
- Break deductions applied correctly per workspace configuration
- Split shifts (same officer, same day, multiple entries) aggregated correctly as one continuous block for OT calculation purposes
- Officers who worked across midnight have their hours correctly attributed to the correct calendar day
- Verify no time entry can be included in two different payroll runs — `payrolledAt` lock confirmed
- Time entries modified after payroll lock are flagged — not silently re-processed

---

**STEP 4 — RATE CALCULATION VERIFICATION**

Every dollar must be calculated correctly:
- Base rate per officer applied from the rate snapshot at clock-in time — not the current rate at run time
- FLSA overtime calculated correctly: hours over 40 in a workweek at 1.5x base rate
- Weighted average overtime rate applied correctly for officers who worked multiple posts at different rates in the same week
- Split shift premium applied where applicable
- Holiday pay applied only on qualifying dates — verify against the holiday calendar
- Night shift, weekend, and hazard differentials applied as distinct line items in the earnings breakdown
- All differential types confirmed tracked as distinct fields in `earningsBreakdown` (confirmed in Phase 11 — regression check)
- Post-specific rate overrides take precedence over base rates — verify the precedence chain
- Zero-rate hard block confirmed still active (confirmed in Phase 11 — regression check)

---

**STEP 5 — DEDUCTION PIPELINE VERIFICATION**

Every deduction must apply in the correct legal order:
- Deduction order enforced: pre-tax deductions → 401k → statutory (federal/state/FICA) → post-tax deductions → garnishments
- Each deduction type has a corresponding line item in the pay stub
- Deductions cannot reduce net pay below zero — hard floor enforced
- Garnishment limits respected — cannot exceed federal maximums
- Deduction configuration changes effective-date correctly — mid-period rate changes do not retroactively alter locked periods
- Missing deduction configuration produces an explicit error — not a silent zero deduction

**STEP 6 — PAYROLL RUN LIFECYCLE VERIFICATION**

Trace the complete payroll run state machine:
- Valid states: `draft → pending_approval → approved → processing → processed → paid → amended`
- No invalid state transitions permitted — jumping from draft to processed must be blocked
- Four-eyes approval confirmed — the officer who creates the run cannot approve it
- Triple idempotency confirmed still active: request header key + `assertNoPeriodOverlap` ledger guard + DB unique constraint (regression check from Phase 11)
- Run locked after `processed` state — no modifications to included time entries after this point
- `paid` state transition records payment method, reference number, and timestamp
- Every state transition writes an audit record

---

**STEP 7 — PAYROLL CORRECTION AND AMENDMENT FLOW**

Off-cycle corrections must be airtight:
- Amendment route blocks on `paid` runs — cannot amend a disbursed payroll (confirmed Phase 11 — regression check)
- Off-cycle run creates a new run record — does not modify the original
- Off-cycle run clearly flagged as `amended` with reference to the original run_id
- Officers receive a notification when an amendment affects their pay
- Amendment audit trail: who initiated, what changed, original values, corrected values, timestamp
- Negative adjustment handling — what happens when a correction results in a net reduction

**STEP 8 — PAY STUB GENERATION AND ACCESS**

Every officer must receive an accurate pay stub:
- Pay stub generated for every officer in the run with confirmed hours
- Pay stub contains: officer name, pay period, regular hours and rate, overtime hours and rate, all deduction line items, gross pay, net pay, employer name, pay stub number
- Pay stub stored in private object storage with correct workspace and officer scoping
- Pay stub accessible only to the officer it belongs to and authorized managers — not visible to other officers
- Officer notified via NDS when pay stub is available
- Pay stub audit record written on generation (confirmed Phase 11 — regression check)
- Historical pay stubs retrievable — officer can access all prior periods
