An audit of the 750+ registered Trinity action definitions.
Identification of duplicate action names pointing to the same behavior.

Removal of stubs that have no real implementation.

and the handler functions that those registry entries point to.

STEP 4 — AFTER APPROVAL: CONSOLIDATION RULES

When I approve consolidation proceed using these exact rules.
Do not deviate from these rules.

RULE 1 — DUPLICATE CONSOLIDATION:
For each duplicate group keep exactly one action.
The action to keep is the one that:
Has the most complete handler implementation.
Has the correct credit cost.
Has RBAC enforcement.
Has an audit trail entry.
Has the most correct naming convention.
If no action in the group meets all criteria keep the best one and improve it.
Remove the duplicates from the registry only — do not delete handler files.
Handler files may be used by other parts of the system outside the registry.

RULE 2 — STUB REMOVAL:
For each stub action make a decision:
If the feature this stub represents is on the pricing page or in a user-facing flow
it must be built for real before being kept in the registry.
Build the real implementation first. Then keep it in the registry.
If the feature this stub represents is not in any user-facing flow or pricing page
remove the stub from the registry entirely.
A stub in the registry is worse than no action — it charges credits for nothing.

RULE 3 — CREDIT COST CORRECTION:
For every credit cost mismatch update the registered cost to match the canonical table.
The canonical table is the source of truth.
Never update the canonical table to match the wrong registry value.

RULE 4 — RBAC ADDITION:
For every action missing RBAC add the correct role check.
The role required for each action is determined by what the action does:
Actions that read data: employee and above.
Actions that modify schedules: manager and above.
Actions that touch financial data: org_owner and co_owner only.
Actions that touch billing or Stripe: org_owner and co_owner only.
Actions that generate compliance reports: manager and above.
Actions that execute payroll: org_owner only.
Actions that manage documents: manager and above.
Platform health actions: platform_staff only.

RULE 5 — AUDIT TRAIL ADDITION:
For every action missing an audit trail entry add this minimum record:
action_type: the action name.
actor_id: the user or system that triggered the action.
workspace_id: the workspace the action executed in.
timestamp: exact execution time.
credits_deducted: amount deducted for this action.
result: success or failure with reason.

RULE 6 — NAMING STANDARDIZATION:
Rename every action that does not follow the domain.verb convention.
Update every reference to the old action name across the entire codebase.
Search for every string that contains the old action name and update it.
Do not leave any orphaned references to renamed actions.
