# SPS DOCUMENT MANAGEMENT SYSTEM — REPLIT AGENT BUILD PROMPT
## CoAIleague Integration | Statewide Protective Services
---

## ROLE
You are building a full-stack DocuSign-style document execution system embedded inside the CoAIleague platform for Statewide Protective Services (SPS). This system handles two primary workflows: (1) New Employee/Officer Onboarding Packets and (2) New Client Contract Proposals with AI-assisted negotiation.

Read every instruction fully before writing a single line of code. Do not declare victory until every section is working end-to-end with real data flow.

---

## TECH STACK
- **Frontend**: React + Tailwind CSS (existing CoAIleague stack)
- **Backend**: Node.js / Express (existing CoAIleague API)
- **Database**: PostgreSQL via Supabase (existing CoAIleague DB)
- **File Storage**: Supabase Storage buckets
- **Email**: Resend (existing pipeline — use calloffs@, incidents@, docs@, support@ routing)
- **AI**: Trinity (Claude primary, GPT fallback, Gemini fallback) via existing orchestration layer
- **Signing**: Canvas-based signature capture (NO external DocuSign dependency)
- **PDF Generation**: react-pdf or pdfmake for final sealed documents

CREATE TABLE sps_document_templates (

  -- Parties
  org_signer_name TEXT,
  org_signer_email TEXT,
  recipient_name TEXT NOT NULL,
  recipient_email TEXT NOT NULL,
  
  -- Employee-specific fields
  employee_dob DATE,

  employee_ssn_last4 TEXT, -- last 4 only stored
  employee_address TEXT,
  employee_phone TEXT,
  guard_license_number TEXT,
  guard_license_expiry DATE,
  guard_license_type TEXT CHECK (guard_license_type IN ('commissioned','non_commissioned','armed','unarmed')),
  assignment_site TEXT,
  assignment_address TEXT,
  hire_date DATE,
  position TEXT,
  pay_rate DECIMAL(8,2),
  uniform_size TEXT,
  
  -- Client-specific fields
  client_company_name TEXT,
  client_address TEXT,
  client_contact_name TEXT,
  client_ein TEXT,
  service_type TEXT CHECK (service_type IN ('armed','unarmed','ppo','patrol','mixed')),
  rate_primary DECIMAL(8,2),
  rate_additional DECIMAL(8,2),
  service_location TEXT,
  service_hours TEXT,
  contract_start_date DATE,
  contract_term TEXT,
  officers_required INTEGER,
  
  -- Document data (all form fields as JSON)
  form_data JSONB DEFAULT '{}',
  
  -- Signature data
  signatures JSONB DEFAULT '{}', -- {field_id: {svg_data, timestamp, ip_address, user_agent}}
  initials JSONB DEFAULT '{}',
  
  -- Audit trail
  audit_log JSONB DEFAULT '[]',

  id_upload_url TEXT,

  -- Timestamps
  sent_at TIMESTAMPTZ,
  viewed_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- Document Negotiation Thread (client ↔ org messages before contract)
CREATE TABLE sps_negotiation_threads (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  document_id UUID REFERENCES sps_documents(id),
  proposal_number TEXT UNIQUE NOT NULL,
  client_name TEXT NOT NULL,
  client_email TEXT NOT NULL,
  status TEXT DEFAULT 'active' CHECK (status IN ('active','agreed','declined','converted_to_contract','expired')),
  agreed_terms JSONB DEFAULT '{}', -- final agreed pricing/terms snapshot
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()

CREATE TABLE sps_negotiation_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

CREATE TABLE sps_document_safe (

  - Personal Protection Officer (PPO) ($__ /hr)
  - Mobile Patrol Service ($__ /hr)
  - Event Security ($__ /hr)
- Rate per type — NUMBER input per selected service type

- Additional officer rate — NUMBER (beyond first, lower rate)

**Advertising Compliance Block (Texas DPS requirement)**
- Company name in full: Statewide Protective Services / Statewide Public Safety Services LLC
- Full address
- License number prominently displayed
- Rates displayed per service type
- "Services regulated under Chapter 1702 of the Texas Occupations Code" statement

### 2B. Negotiation Thread (Trinity-Assisted)

Once proposal is sent to client email:

1. **Client receives email** with secure link to view proposal
2. **Client can reply** via embedded message form on proposal page (no login required for client)
3. **Org receives notification** in CoAIleague dashboard
4. **Before org sends ANY message**, Trinity offers an AI-enhanced version:
   - Shows original typed message
   - Shows Trinity-polished version
   - "Use Trinity's version" / "Use my version" / "Edit suggestion" buttons
   - Trinity prompt: "You are a professional security company sales consultant for Statewide Protective Services. Polish this message to be professional, clear, legally appropriate, and persuasive while maintaining the core intent. Keep it concise. Do not change pricing unless instructed."
5. **Terms extraction**: Trinity scans incoming client messages and extracts:
   - Proposed rate changes
   - Schedule modifications
   - Officer count changes
   - Special requests
   - Concerns/objections
   Displays extracted terms as tags: "Client proposes: $25/hr unarmed | Fri-Sun only | 2 officers"
6. **Agreement detection**: When Trinity detects agreement language ("that works", "we accept", "agreed", "let's move forward"), it flags the thread:
   - "Agreement detected — ready to convert to contract?"
   - Shows snapshot of agreed terms
   - [Convert to Contract] button

### 2C. Contract Generation from Agreed Proposal

When org clicks [Convert to Contract]:
1. Pre-fills all contract fields from agreed terms in negotiation thread
2. Generates Contract Number: CON-{YYYY}-{####}
3. Links back to proposal number
4. Both parties must sign:
   - Client signs: signature + initials on liability clauses + date
   - SPS signs: org signature auto-filled or org manually signs in dashboard
5. ID verification required for client (upload driver's license or state ID)

### 2D. Full Contract Sections (from uploaded document)

10. Warranty — initials required

### Submission & Sealing

When all required fields complete:
1. [Complete & Submit] button activates
2. Show summary: "You are about to sign [Document Name]. This is legally binding."
3. On confirm:
   - Set document.status = 'completed'
   - Seal signatures (no further editing)
   - Generate PDF via pdfmake with all data embedded
   - Upload sealed PDF to Supabase Storage under: `/{workspace_id}/documents/{document_id}/sealed_{document_number}.pdf`
   - Save record in sps_document_safe
   - Send completion emails to all parties via Resend (docs@ routing)
   - Add final audit log entry

// 1. Accept image (jpg/png) or PDF

const trinityIDScan = async (imageBase64, documentType) => {
  const prompt = `
You are an ID verification assistant for a licensed Texas security company. 
Analyze this ${documentType} image and extract the following information.
Return ONLY valid JSON, no markdown.

Extract:
{
  "document_type": "drivers_license|state_id|passport|guard_card",
  "issuing_state": "string",
  "full_name": "string",
  "date_of_birth": "YYYY-MM-DD",
  "id_number": "string",

  "is_expired": boolean,

    "has_state_seal": boolean,
    "has_hologram_indicators": boolean,
    "text_consistent": boolean,
    "format_matches_state": boolean
  },
  "verification_confidence": "high|medium|low",
  "flags": ["any concerns or anomalies"],
  "license_type_if_guard_card": "non_commissioned|commissioned|armed",
  "license_number_if_guard_card": "string",
  "guard_card_expiry_if_applicable": "YYYY-MM-DD"
}

If this is a guard card, also confirm it appears to be an official Texas DPS Private Security Bureau card.

  // Send to Trinity (Claude primary)

    model: 'claude',

  return JSON.parse(result);
