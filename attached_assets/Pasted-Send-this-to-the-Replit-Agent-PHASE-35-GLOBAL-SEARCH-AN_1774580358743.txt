**STEP 2 — SEARCH ARCHITECTURE**

Design the search system before building:

**PostgreSQL full-text search approach:**
- Use PostgreSQL `tsvector` and `tsquery` for full-text search — no external search engine required for initial implementation
- Create `search_indexes` materialized view or use GIN indexes on text columns across all searchable tables
- Weighted search: exact matches rank higher than partial matches, recent records rank higher than old ones
- Fuzzy matching for names — handle typos and partial names using `pg_trgm` extension

**Searchable entity types and fields:**

| Entity | Searchable Fields | Minimum Role |
|--------|------------------|--------------|
| Officers | Name, badge number, email, phone, license number | Supervisor |
| Clients | Company name, contact name, email, phone | Manager |
| Shifts | Post name, date, officer name, status | Supervisor |
| Invoices | Invoice number, client name, amount | Manager |
| Documents | Document name, document type, entity name | Manager |
| Incidents | Description, location, officer name, type | Supervisor |
| Compliance violations | Description, officer name, violation type | Compliance officer |
| Support tickets | Subject, description, category | Platform staff |
| Pay stubs | Officer name, period | Org owner only |
| Work orders | Title, client name, type | Manager |
| Audit log | Action type, entity name | Org owner |

**Search result structure:**
Every result returns: entity_type, entity_id, display_name, subtitle, relevance_score, created_at, a deep_link to the correct page

**STEP 3 — SEARCH API**

Build a unified search endpoint:

**`GET /api/search?q={query}&types={comma-separated}&limit={n}&offset={n}`**

- `q` — search query string, minimum 2 characters, maximum 100 characters
- `types` — optional filter by entity type, defaults to all types the requesting user can access
- `limit` — default 20, maximum 50
- `offset` — for pagination
- Returns: `{ results: SearchResult[], total: number, took_ms: number, query: string }`
- Every result includes a `deep_link` — the exact URL to navigate to that record
- Results grouped by entity type in the response
- Results ordered by relevance score then by created_at descending within each type
- Workspace isolation enforced on every sub-query — results only from requesting workspace
- Role filtering enforced — pay stubs only returned for org_owner, audit log only for authorized roles
- Search query logged for analytics (not personally identifiable — just the query string, entity types, result count, response time)
- Rate limited — maximum 30 search requests per minute per user

---

**STEP 4 — SEARCH INDEXING**

Every searchable entity must have proper indexes:

- Enable `pg_trgm` extension for fuzzy matching
- Create GIN indexes on text columns for all searchable entities:
  - `officers`: GIN index on (first_name || ' ' || last_name), email, license_number
  - `clients`: GIN index on company_name, contact_name, email
  - `shifts`: GIN index on post_name, notes
  - `invoices`: GIN index on invoice_number, client name (via join)
  - `documents`: GIN index on document_name, document_type
  - `incident_reports`: GIN index on description, location
  - `compliance_violations`: GIN index on description, violation_type
- Index creation must be CONCURRENT — non-blocking, does not lock tables during creation
- Index creation logged and verified before search goes live
- Search performance baseline: simple name query must return in under 100ms

**STEP 5 — GLOBAL SEARCH UI**

Search must be accessible from everywhere:

**Command palette (desktop):**
- Keyboard shortcut `Cmd+K` (Mac) / `Ctrl+K` (Windows/Linux) opens global search
- Full-screen overlay with search input prominently centered
- Recent searches shown before typing (stored in localStorage per user)
- As user types (debounced 300ms) — live results appear below input
- Results grouped by entity type with entity type label
- Each result shows: icon for entity type, display name, subtitle, and the deep_link page label
- Keyboard navigable — up/down arrows, Enter to navigate, Escape to close
- Loading state: subtle spinner while fetching, not a full skeleton
- Empty state: helpful message "No results for '[query]' — try a different term"
- Result count shown: "12 results for 'Martinez'"

**Search bar (mobile):**
- Persistent search icon in the mobile top bar — tap to open search
- Full-screen search view on mobile — search input at top, results below
- Same result structure as desktop but optimized for touch — larger tap targets
- Recent searches shown as chips below the input before typing
- Voice search option — tap microphone to speak the search query (uses Phase 22 voice infrastructure)

**Inline search within pages:**
- Officers list, Clients list, Invoices list — existing search/filter inputs enhanced to use the global search API where appropriate
- Results in inline search do not navigate away from the page — they filter the current view

**STEP 7 — TRINITY SEARCH INTEGRATION**

Trinity must be able to search on behalf of users:

- `search.query` Trinity action — accepts natural language query, translates to structured search, returns results
- Trinity can answer: "Find all invoices for Securitas from last month" or "Show me Officer Martinez's recent incidents"
- Trinity search results respect the requesting user's role — Trinity cannot return records the user cannot access directly
- Trinity can refine search results conversationally: "Show me only the overdue ones" after an initial search
- Trinity search uses the same `/api/search` endpoint — not a separate implementation
- Trinity can navigate the user to a result: "Here is the invoice — shall I open it for you?"
