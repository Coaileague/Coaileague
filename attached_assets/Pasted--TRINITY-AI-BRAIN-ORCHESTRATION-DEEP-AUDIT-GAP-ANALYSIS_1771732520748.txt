You are performing a comprehensive audit of Trinity — our multi-model AI orchestration system. Trinity is the AI brain of CoAIleague, a B2B SaaS workforce management platform for security companies. Trinity uses 3 AI models (Gemini as primary, Claude as validator/judge, OpenAI/GPT as workhorse/backup) working in governance and tandem with a meta-cognition executive function layer above them.

### 1.1 — Model Router & Selection Logic

```
VERIFY:
[ ] Request comes in → Router determines which model handles it
[ ] Task classification exists (simple/operational/strategic/complex)
[ ] Model selection is based on task type, not random or hardcoded
[ ] Selection considers: cost, speed, accuracy needed, task complexity
[ ] Router logs WHY it chose a specific model

GAPS TO FIND:
- Is there actually a router, or does everything just go to one model?
- Is task classification real or just a placeholder?
- Can the router dynamically adjust based on model performance history?
- Is there a "model health" check before routing?
```

### 1.2 — Fallback Chain Integrity

```
VERIFY THE CHAIN:
Primary (Gemini/Trinity) → fail → Claude → fail → OpenAI/GPT → fail → Human Escalation

[ ] Primary model attempt has proper timeout (not hanging forever)
[ ] Timeout value is configurable, not hardcoded
[ ] On primary failure, error is captured with full context
[ ] Context is passed to fallback model (not starting from scratch)
[ ] Fallback model receives: original prompt + primary failure reason + any partial results
[ ] If fallback also fails, third model gets cumulative context
[ ] If ALL models fail, human escalation triggers with full audit trail
[ ] Each attempt is logged to meta_cognition_logs
[ ] Token usage tracked per attempt per model per org
[ ] Cost attributed to the correct subscriber organization (NOT to CoAIleague platform)

GAPS TO FIND:
- Does the fallback actually work or is it just defined but never tested?
- Does context degrade through the chain (losing info at each hop)?
- Is there retry logic WITH backoff, or just immediate retry?
- What happens if the fallback model is also rate-limited?
- Are partial results from failed attempts preserved or thrown away?
- Is there a circuit breaker to stop hammering a model that's down?
```

### 1.3 — Meta-Cognition Executive Function Layer

```
THIS IS THE BRAIN ABOVE THE BRAIN. VERIFY:

[ ] Meta-cognition layer exists as actual code (not just a concept)
[ ] It triggers when:
    - Task classified as 'complex' or 'high-stakes'
    - Any model returns confidence below threshold (0.75)
    - Two models return conflicting answers
    - Task involves money/payroll/billing
    - Task involves compliance/legal
[ ] It has access to:
    - Original task/prompt
    - ALL model responses
    - Individual confidence scores
    - Reasoning chains from each model
    - Execution time per model
    - Historical performance data
[ ] It can:
    - SYNTHESIZE: Merge best parts of multiple model responses
    - ARBITRATE: Choose the best response when they conflict
    - ESCALATE: Flag for human review when confidence is too low
    - LEARN: Log the decision for future pattern improvement

GAPS TO FIND:
- Does this layer actually exist in code or is it theoretical?
- Is the confidence threshold configurable per task type?
- Does it log its own reasoning (metacognition of metacognition)?
- Can it override a model that returned high confidence but wrong answer?
- Is there a feedback loop where human corrections train the meta-layer?
```

### 2.3 — Tool Execution Framework

```
EVERY TOOL MUST FOLLOW THE 7-STEP PIPELINE:

1. TRIGGER → What event/request activates this tool?
2. FETCH → What data does the tool need to gather?
3. VALIDATE → Are all required inputs present and authorized?
4. PROCESS → Execute the core logic
5. MUTATE → Save results to database
6. CONFIRM → Return success/failure with details
7. NOTIFY → Alert relevant parties of the outcome

FOR EACH TOOL, VERIFY:
[ ] All 7 steps are implemented
[ ] Step 3 (VALIDATE) includes org-scoping / tenant isolation
[ ] Step 4 (PROCESS) has try/catch with meaningful error messages
[ ] Step 5 (MUTATE) uses transactions for multi-table writes
[ ] Step 6 (CONFIRM) returns structured response Trinity can interpret
[ ] Step 7 (NOTIFY) actually sends notifications (not just logs)

COMMON GAPS:
- Tools that fetch but don't validate permissions
- Tools that process but don't save results
- Tools that save but don't confirm back to Trinity
- Tools that work but don't notify anyone
- Tools that skip straight from trigger to process (no fetch/validate)
```

## PHASE 3: TRINITY'S INTELLIGENT LOOP & ITERATION

This is what makes Trinity smart vs. just a chatbot wrapper. She needs to THINK iteratively.

### 3.1 — Logic Loop Framework

```
VERIFY TRINITY'S REASONING LOOP EXISTS:

UNDERSTAND → GATHER → ANALYZE → PROPOSE → VALIDATE → EXECUTE → CONFIRM
    ↑                                                              │
    └──────────────────── ITERATE IF NEEDED ───────────────────────┘

[ ] Trinity can decompose complex requests into sub-tasks
[ ] Each sub-task follows the 7-step pipeline independently
[ ] Results from sub-tasks feed back into the main task
[ ] If a sub-task fails, Trinity re-plans (not just fails entirely)
[ ] Trinity can ask clarifying questions before executing
[ ] Trinity explains her reasoning before taking action on high-stakes tasks
[ ] Trinity can handle multi-turn conversations maintaining full context

EXAMPLE TEST CASE — Give Trinity this prompt and verify she handles it:

"We just lost 3 guards at the Walmart account. Two quit and one's 
license expired. I need coverage for all their shifts this week, 
make sure replacements are qualified, notify the client, and give 
me a cost impact analysis."

TRINITY SHOULD:
1. UNDERSTAND: 3 guards gone, need immediate replacement + client comms + cost analysis
2. GATHER: Query the 3 guards' shift schedules, Walmart site requirements, available qualified staff
3. ANALYZE: Which shifts are open? Who's qualified AND available? Distance? Cost difference?
4. PROPOSE: "Here are my recommended replacements for each shift with reasoning..."
5. VALIDATE: Check each replacement for: certifications, overtime impact, availability conflicts
6. EXECUTE: Assign shifts, send notifications, generate client email, calculate cost delta
7. CONFIRM: Summary of all actions taken with links/references

IF SHE CAN'T DO THIS END-TO-END, DOCUMENT EXACTLY WHERE SHE BREAKS.
```
