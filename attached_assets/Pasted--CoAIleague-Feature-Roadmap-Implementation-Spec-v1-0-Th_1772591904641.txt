# ═══════════════════════════════════════════════════

## 1A: What Security Companies Sign

Every security company generates the same document types repeatedly. CoAIleague should handle all of them natively:

- Employment agreements (merit-based pay, NDAs, at-will acknowledgment, policy acceptance)
- Client service contracts (guard services agreements, SLAs, pricing schedules, scope of work)
- Post orders (site-specific instructions signed by each officer assigned to that site)
- Policy acknowledgments (use of force, uniform policy, attendance policy, vehicle policy)
- Incident report sign-offs (officer signs, supervisor countersigns, client acknowledges)
- Equipment issue/return forms (radio, badge, uniform, vehicle keys, body camera)
- Vehicle inspection sign-offs (pre/post shift vehicle condition reports)
- Training completion certificates (CPR, first aid, de-escalation, firearms qualification)
- Non-compete / non-solicitation agreements
- Background check authorization forms
- Direct deposit authorization forms

## 1C: Database Schema

```sql
-- ═══════════════════════════════════════════════════
-- DOCUMENT TEMPLATES
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS document_templates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  name VARCHAR(200) NOT NULL,
  description TEXT,
  category VARCHAR(50) NOT NULL,
  -- Categories: employment, client_contract, post_order, policy,
  --             incident, equipment, vehicle, training, compliance, custom
  content_type VARCHAR(20) NOT NULL DEFAULT 'rich_text',
  -- Types: rich_text (built in editor), uploaded_pdf, html_template
  content_body TEXT,                          -- Rich text / HTML template content
  uploaded_pdf_url VARCHAR(500),              -- If type = uploaded_pdf
  merge_fields JSONB DEFAULT '[]',           -- Array of {key, label, source_table, source_column}
  signature_fields JSONB DEFAULT '[]',       -- Array of {id, label, signer_role, page, x, y, width, height}
  -- signer_role: employee, supervisor, manager, client, witness
  requires_countersign BOOLEAN DEFAULT false,
  countersign_roles JSONB DEFAULT '[]',      -- e.g., ["supervisor", "client"]
  auto_send_on_event VARCHAR(50),            -- Event trigger: new_hire, site_assignment, shift_complete, contract_renewal
  expiration_days INTEGER,                   -- Days until signature request expires
  version INTEGER NOT NULL DEFAULT 1,
  is_active BOOLEAN DEFAULT true,
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_doc_templates_workspace ON document_templates(workspace_id, category);
CREATE INDEX idx_doc_templates_event ON document_templates(auto_send_on_event) WHERE auto_send_on_event IS NOT NULL;

-- ═══════════════════════════════════════════════════
-- DOCUMENT INSTANCES (Generated Documents)
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS document_instances (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  template_id UUID REFERENCES document_templates(id),
  title VARCHAR(300) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'draft',
  -- Status: draft, pending_signature, partially_signed, completed, expired, voided
  generated_pdf_url VARCHAR(500),
  signed_pdf_url VARCHAR(500),
  merge_data JSONB DEFAULT '{}',             -- Snapshot of data used to fill template
  related_entity_type VARCHAR(30),           -- employee, client, site, shift, incident
  related_entity_id UUID,
  created_by UUID REFERENCES users(id),
  completed_at TIMESTAMP,
  expires_at TIMESTAMP,
  voided_at TIMESTAMP,
  voided_reason TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_doc_instances_workspace ON document_instances(workspace_id, status);
CREATE INDEX idx_doc_instances_entity ON document_instances(related_entity_type, related_entity_id);

-- ═══════════════════════════════════════════════════
-- SIGNATURE REQUESTS (One per signer per document)
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS signature_requests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id UUID NOT NULL REFERENCES document_instances(id),
  signer_role VARCHAR(30) NOT NULL,          -- employee, supervisor, manager, client, witness
  signer_user_id UUID REFERENCES users(id),  -- Internal user (employee/manager)
  signer_email VARCHAR(255),                 -- External signer (client) or fallback
  signer_name VARCHAR(200),
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  -- Status: pending, viewed, signed, declined, expired
  signing_token VARCHAR(100) UNIQUE,         -- JWT or unique token for signing link
  signing_order INTEGER DEFAULT 1,           -- 1 = first signer, 2 = countersigner, etc.
  signature_fields JSONB DEFAULT '[]',       -- Which fields this signer is responsible for
  reminder_sent_at TIMESTAMP,
  viewed_at TIMESTAMP,
  signed_at TIMESTAMP,
  declined_at TIMESTAMP,
  declined_reason TEXT,
  ip_address VARCHAR(45),
  user_agent TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_sig_requests_document ON signature_requests(document_id);
CREATE INDEX idx_sig_requests_signer ON signature_requests(signer_user_id, status);
CREATE INDEX idx_sig_requests_token ON signature_requests(signing_token);

-- ═══════════════════════════════════════════════════
-- SIGNATURES (Actual signature data)
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS signatures (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  request_id UUID NOT NULL REFERENCES signature_requests(id),
  field_id VARCHAR(50) NOT NULL,             -- Matches signature_fields[].id from template
  signature_type VARCHAR(20) NOT NULL,       -- draw, type, upload
  signature_data TEXT NOT NULL,              -- Base64 PNG (draw), typed name (type), image URL (upload)
  signed_at TIMESTAMP NOT NULL DEFAULT NOW(),
  ip_address VARCHAR(45) NOT NULL,
  user_agent TEXT NOT NULL,
  geolocation JSONB                          -- {lat, lng, accuracy} if available
);

CREATE INDEX idx_signatures_request ON signatures(request_id);

-- ═══════════════════════════════════════════════════
-- DOCUMENT VAULT (Completed + Uploaded Documents)
-- ═══════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS document_vault (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  document_instance_id UUID REFERENCES document_instances(id),  -- NULL if manually uploaded
  title VARCHAR(300) NOT NULL,
  category VARCHAR(50) NOT NULL,
  file_url VARCHAR(500) NOT NULL,
  file_size_bytes INTEGER,
  mime_type VARCHAR(50),
  tags JSONB DEFAULT '[]',
  related_entity_type VARCHAR(30),
  related_entity_id UUID,
  uploaded_by UUID REFERENCES users(id),
  is_signed BOOLEAN DEFAULT false,
  retention_until TIMESTAMP,                 -- For compliance: when this can be deleted
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_vault_workspace ON document_vault(workspace_id, category);
CREATE INDEX idx_vault_entity ON document_vault(related_entity_type, related_entity_id);
```

## 1D: Service Layer

```
DocumentTemplateService:
  - createTemplate(workspaceId, templateData) → template
  - listTemplates(workspaceId, category?) → templates[]
  - getTemplate(templateId) → template with merge field definitions
  - updateTemplate(templateId, updates) → template (creates new version)
  - deleteTemplate(templateId) → soft delete (is_active = false)

DocumentGenerationService:
  - generateFromTemplate(templateId, mergeData, signerConfig[]) → documentInstance
    1. Fetch template
    2. Resolve merge fields from database (employee name, pay rate, site address, etc.)
    3. Generate PDF using pdf-lib (fill merge fields into template)
    4. Store generated PDF
    5. Create document_instance record
    6. Create signature_request records for each signer
    7. Generate signing tokens (JWT, expires per template.expiration_days)
    8. Send notification to first signer (respecting signing_order)
    CREDIT COST: 5 credits (Trinity merge + PDF generation)

  - generateBulk(templateId, entityIds[], signerConfig) → documentInstances[]
    For sending the same template to many employees (policy acknowledgments, etc.)
    CREDIT COST: 3 credits each (bulk discount)

SigningService:
  - getSigningSession(token) → document + fields for this signer (no auth required — token IS auth)
  - submitSignature(token, signatures[]) → signed document
    1. Validate token (not expired, not already signed)
    2. Store each signature image/data
    3. Embed signatures into PDF at defined coordinates using pdf-lib
    4. Update signature_request status = 'signed'
    5. If all signers complete → update document_instance status = 'completed'
    6. Store signed PDF in document_vault
    7. Notify next signer if countersign required
    8. Notify document creator that signing is complete
    CREDIT COST: 0 credits (signing is free — generation costs credits)

  - declineSignature(token, reason) → declined
  - sendReminder(requestId) → reminder notification sent

DocumentVaultService:
  - listDocuments(workspaceId, filters) → documents[] with pagination
  - getDocument(documentId) → document with download URL
  - uploadManual(workspaceId, file, metadata) → vault entry
  - searchDocuments(workspaceId, query) → full-text search results
```

## 2B: Report Definitions

### REPORT 1: Client Profitability (P&L by Client)
**Data source:** Invoices (revenue) + Time entries × pay rates (labor cost)
**Why it matters:** Tells the owner which contracts make money and which are underwater

```
TRINITY INSIGHT (auto-generated):
"North Star Data Center has your highest margin at 26.5%. Consider
proposing expanded coverage. Retail Solutions Group is losing $386/mo
on only 8 hours of coverage — recommend contract renegotiation or

```

**Calculation:**
- Revenue = Sum of invoices for client in period
- Labor Cost = Sum of (hours worked × employee pay rate) for shifts at client sites
- Margin = Revenue - Labor Cost
- Margin % = Margin / Revenue × 100
- Color coding: Green ≥ 25%, Amber 15-25%, Red < 15%

## 2C: Technical Architecture

```
ReportingService:
  - generateReport(workspaceId, reportType, dateRange, filters) → reportData
  - exportReport(reportId, format) → PDF or CSV download
  - getReportInsight(reportType, reportData) → Trinity AI analysis
  
  Report types: client_profitability, payroll_summary, ar_aging,
                revenue_trend, labor_cost, tax_liability, cash_flow,
                workers_comp

  All reports query existing tables:
    - invoices (revenue)
    - time_entries / clock_records (hours)
    - employees (pay rates, classification)
    - shifts (site assignment)
    - clients (contract rates)
    - payroll_runs (tax withholdings)
    - expenses (outflows)
```

No new database tables needed — reports are read-only aggregations of existing data. The only new table is for saved/scheduled reports:


```sql

CREATE TABLE IF NOT EXISTS incident_reports (

  gps_longitude DECIMAL(11, 8),
```

## 4A: The Problem

Mobile forms throughout the app (incident reports, daily reports, safety checklists, expense submissions, time-off requests, equipment forms) all use the same plain pattern: dark card with basic HTML inputs. They feel like developer prototypes, not professional tools that field officers use at 2am in poor lighting conditions.

## 4B: Universal Form Component System

Build a reusable form component library that ALL forms in the app use. This ensures consistency and makes future form creation trivial.

### Form Field Components:

```
TextInput:         Standard text field with floating label, clear button
TextArea:          Multi-line with character count, auto-expand, voice-to-text button
NumberInput:       Numeric with +/- steppers (for counts, quantities)
SelectChips:       Horizontal scrollable colored chips (replaces dropdowns for <8 options)
SelectDropdown:    For >8 options, searchable dropdown with categories
DateTimePicker:    Native date/time picker with quick presets ("Now", "Start of shift", "End of shift")
PhotoCapture:      Camera + gallery with thumbnail preview strip, up to 10 photos
VoiceInput:        Hold-to-record button → auto-transcribe → fills text field
SignatureField:    Canvas for drawing signature (from Feature 1)
ChecklistGroup:    Styled checkboxes with tap-entire-row, completion counter
LocationCapture:   Auto GPS with address display, tap to view on map
ToggleSwitch:      On/off with clear visual state
RatingScale:       1-5 stars or 1-10 slider for severity/satisfaction ratings
FileUpload:        Document/PDF upload with preview
```

### Form Container Patterns:

```

  - Slides up from bottom, handles ~40% of screen
  - Pull-down handle to dismiss
  - Content scrolls within the sheet
  - Primary action button fixed at bottom of sheet
```

# ═══════════════════════════════════════════════════

## 5A: Architecture

DockChat already handles internal team messaging. The goal is to bridge external messaging platforms so ALL communication flows through one unified inbox — DockChat.

```
EXTERNAL MESSAGE FLOW:

  Client sends WhatsApp msg ──► Twilio/360dialog webhook ──► DockChat middleware ──► Appears in
                                                                                     DockChat as
  Client sends SMS ───────────► Twilio webhook ────────────► DockChat middleware ──► a unified
                                                                                     conversation
  Client sends email ─────────► Resend inbound webhook ───► DockChat middleware ──►
                                                                                     
  Client sends FB Messenger ──► Meta webhook ──────────────► DockChat middleware ──►

OUTBOUND (reply from DockChat):

  Manager replies in DockChat ──► Middleware detects channel ──► Routes back via same channel
                                  (WhatsApp/SMS/Email/Messenger)   (client receives reply on
                                                                    their original platform)
```

## 5B: Middleware Service

```
MessageBridgeService:
  - receiveInbound(channel, from, message, metadata) → creates/appends to DockChat conversation
    1. Identify sender: match phone/email to client or employee record
    2. Find or create DockChat conversation thread
    3. Store message with channel metadata (so replies go back same channel)
    4. If sender unknown: create "Unknown Contact" thread, notify manager
    
  - sendOutbound(conversationId, message, replyTo) → delivers via original channel
    1. Look up conversation's channel metadata
    2. Route through appropriate provider (Twilio for SMS/WhatsApp, Resend for email, Meta API for Messenger)
    3. Store delivery status
    
  - getUnifiedInbox(workspaceId) → all conversations across all channels, merged
```

## 6A: The Pipeline

Security companies win new business through a consistent flow: Lead → RFP Response → Proposal → Contract → Onboarding. Most companies do this in Word docs and email. CoAIleague automates the entire pipeline.

```
PIPELINE STAGES:

  1. LEAD           New business opportunity entered (manual or from inbound email)
  2. SITE SURVEY    Schedule and document site assessment (photos, notes, requirements)
  3. RFP RESPONSE   Trinity generates professional response from templates + site data
  4. PROPOSAL       Pricing proposal with coverage plan, rates, terms
  5. NEGOTIATION    Track revisions, counter-offers, communications
  6. CONTRACT       Generate contract from proposal terms (→ Document Signing flow)
  7. ONBOARDING     New client setup: create site, assign officers, generate post orders
```

## Recommended Build Order

Each feature is independent — they can be built in parallel by the Replit agent. But if doing them sequentially, this order maximizes value:

```
PHASE 1 — Revenue Generators (Weeks 1-3):
  Feature 2: QB Reports         → Helps close Enterprise deals (owners want reports)
  Feature 6: Pipeline/RFP       → Directly drives new client acquisition
  Feature 3: Incident Pipeline   → Biggest differentiator vs all competitors

PHASE 2 — Platform Completeness (Weeks 3-5):
  Feature 1: Document Signing    → Eliminates DocuSign dependency, enables auto-onboarding
  Feature 4: Form Builder/UX     → Every form in the app gets better simultaneously

PHASE 3 — Communication & Scale (Weeks 5-7):
  Feature 5: DockChat Bridges    → Requires Twilio/Meta accounts, more complex integration
  Feature 7: Desktop Polish      → Large surface area, benefits from all other features existing first
```
