# PHASE 1: DATABASE SCHEMA & ENUM INTEGRITY AUDIT

**Purpose:** Eliminate all missing tables, misnamed tables, missing columns, missing enums, missing identifiers, improperly named fields, and workspace_id/UUID issues that break workflows and cause silent failures.

### Step 1.1 — Full Schema Inventory
- Pull the complete database schema (all tables, all columns, all types, all enums, all constraints, all foreign keys, all indexes).
- Output a complete list of every table and its columns with data types.
- Confirm this matches what the codebase expects.

### Step 1.2 — Enum Validation
- List every enum defined in the database.
- List every enum referenced in the codebase (TypeScript types, Zod schemas, form selectors, API validators).
- Cross-reference: Flag every mismatch where a code enum value does not exist in the database enum, or a database enum value is not handled in code.
- Fix every mismatch found. Do not skip any.

### Step 1.3 — Table Name Consistency
- Compare every table name referenced in queries, ORM models, API routes, and migration files against actual database tables.
- Flag every instance where code references a table name that does not exist or is misspelled.
- Fix every instance found.

### Step 1.4 — Column Name & Type Consistency
- For every table, compare every column reference in the codebase against the actual schema.
- Flag missing columns, misspelled column names, and type mismatches (e.g., code expects UUID but column is TEXT, code expects integer but column is varchar).
- Fix every instance found.

### Step 1.5 — workspace_id / org_id / UUID Propagation Audit
- Trace every workflow, API call, mutation, and query that requires a workspace_id, org_id, or user UUID.
- Verify that these identifiers are properly passed through every layer: frontend → API → service → database.
- Flag every instance where a workspace_id/org_id/UUID is undefined, null, missing from the query, or not included in the request context.
- Fix every instance found.

### Step 1.6 — Foreign Key & Relationship Integrity
- Verify all foreign key relationships are properly defined.
- Check for orphaned references (foreign keys pointing to non-existent tables/columns).
- Verify cascade rules are appropriate (ON DELETE, ON UPDATE).
- Fix every issue found.

### Step 1.7 — Migration File Consistency
- Review all migration files in order.
- Verify they produce the current expected schema when run sequentially.
- Flag any migrations that reference tables/columns that don't exist at that point in the migration chain.
- Flag any missing migrations for tables/columns that exist in the schema but have no migration.

### 📋 PHASE 1 SUMMARY
Output: Total tables checked, total enums validated, total mismatches found, total fixes applied, pass/fail status.

# PHASE 2: RACE CONDITIONS & WORKFLOW PIPELINE INTEGRITY

**Purpose:** Eliminate all race conditions, timing issues, and workflow pipeline failures that cause silent execution failures.

### Step 2.1 — Identify All Async Operations
- List every async operation in the platform: API calls, database transactions, real-time subscriptions, webhook handlers, cron jobs, AI orchestration calls, QuickBooks sync operations.
- For each, document the expected execution order and dependencies.

### Step 2.2 — Race Condition Scan
- For every pair of operations that share state (same table, same row, same cache key), verify that proper locking, transactions, or serialization is in place.
- Flag every instance where two operations can read/write the same data without protection.
- Pay special attention to:
  - Trinity AI orchestration (multi-model calls with fallback chains — are responses handled atomically?)
  - QuickBooks sync (are sync operations serialized per org? Can two syncs overlap?)
  - Real-time subscriptions (are mutations and subscription updates properly ordered?)
  - Scheduling operations (can two scheduling requests for the same employee conflict?)
  - Credit/token tracking (are credit deductions atomic? Can double-spend occur?)
  - Clock in/out operations (GPS + time verification — can race conditions create duplicate entries?)
- Fix every race condition found with appropriate locking/transaction mechanisms.

### Step 2.3 — Silent Failure Audit
- Search the entire codebase for:
  - `catch` blocks that are empty or only log without re-throwing or returning error state
  - `.then()` chains without `.catch()`
  - `async` functions called without `await`
  - Database operations without error handling
  - API calls that don't check response status
  - Webhook handlers that return 200 regardless of processing outcome
- Every silent failure point must be converted to explicit error handling with:
  - User-visible error messaging where appropriate
  - Logging with context (workspace_id, user_id, operation type, timestamp)
  - Proper error propagation to calling functions

### Step 2.4 — AI Token & Credit Tracking Verification
- Trace the complete flow of AI token usage from request → execution → cost calculation → credit deduction → org attribution.
- Verify that EVERY AI call (Gemini, Claude, OpenAI) properly:
  - Records input/output token counts
  - Calculates cost accurately per model
  - Attributes cost to the correct organization (never to the platform)
  - Deducts from the org's credit balance atomically
  - Handles failed/partial responses (no phantom charges)
  - Logs the transaction with full audit trail
- Verify the fallback chain (primary model fails → secondary → tertiary) properly tracks costs for whichever model actually responds.
- Fix every gap found.

### Step 2.5 — Execution Pipeline End-to-End Test
- Trace every major workflow from trigger to completion:
  - User signup → org creation → onboarding pipeline
  - Employee invite → acceptance → workspace join
  - Schedule creation → Trinity AI processing → assignment
  - Clock in → GPS verification → time entry → payroll calculation
  - QB sync trigger → data transformation → API call → confirmation
  - AI chat request → model selection → execution → response → token tracking
  - Broadcast creation → recipient resolution → delivery → confirmation
- For each, verify every step has proper error handling, state tracking, and rollback capability.
- Fix every gap found.

### 📋 PHASE 2 SUMMARY
Output: Total async operations audited, race conditions found/fixed, silent failures found/fixed, token tracking issues found/fixed, pipeline gaps found/fixed, pass/fail status.

# PHASE 3: AUTHENTICATION & AUTHORIZATION AUDIT

**Purpose:** Ensure all auth flows are bulletproof, all routes are protected, and all role-based access is enforced.

### Step 3.1 — Auth Flow Verification
- Trace the complete authentication flow:
  - Org signup → email verification → session creation → token issuance
  - Employee invite flow → invite validation → account creation → workspace join
  - Login → credential verification → session/token creation → refresh flow
  - Password reset → token generation → validation → update
  - Session expiry → refresh → re-authentication
- Verify each step handles errors properly and cannot be bypassed.

### Step 3.2 — Route Protection Audit
- List EVERY API route and page route in the application.
- For each route, verify:
  - Authentication middleware is applied (no unprotected routes that should be protected)
  - Authorization checks verify the user has permission for the specific workspace/org
  - workspace_id scoping is enforced (users cannot access data from other orgs)
  - Role-based access control is checked where required (admin vs manager vs officer)
- Flag and fix every unprotected or improperly scoped route.

### Step 3.3 — Invite-Only Enforcement
- Verify that users CANNOT sign up as standalone users — they must either:
  - Create an org (org signup flow), OR
  - Accept a valid invite from an existing org
- Verify that expired/revoked invites are properly rejected.
- Verify that invite tokens cannot be reused after acceptance.
- Verify seat limits are enforced at invite creation time.

### Step 3.4 — Session & Token Security
- Verify tokens have appropriate expiry times.
- Verify refresh token rotation is implemented.
- Verify sessions are invalidated on password change/account deactivation.
- Verify CSRF protection is in place for state-changing operations.

Output: Total routes audited, unprotected routes found/fixed, auth flow gaps found/fixed, invite enforcement status, pass/fail.

# PHASE 4: FRONTEND VISUAL & UI AUDIT — DESKTOP

**Purpose:** Eliminate ALL visual issues on desktop: text bleeding, improper sizing, card rendering, badge rendering, text overlap, button overlap, and ensure Fortune-500 visual polish.

### Step 4.1 — Global CSS & Layout Foundation Check
- Review the global CSS/Tailwind configuration:
  - Verify consistent spacing scale is used across the platform
  - Verify consistent font sizing scale (no arbitrary pixel values)
  - Verify consistent color palette usage
  - Verify consistent border-radius, shadow, and transition values
  - Verify responsive breakpoints are properly defined
  - Verify z-index layering scale (modals > sheets > menus > tooltips > content)

### Step 4.2 — Typography & Text Rendering Audit
- Check EVERY page/component for:
  - Text overflow/bleeding outside containers (missing `overflow-hidden`, `truncate`, `line-clamp`, or `text-ellipsis`)
  - Text overlapping other text (z-index issues, absolute positioning conflicts)
  - Text overlapping buttons/action elements (insufficient spacing, missing padding)
  - Font sizes that are inconsistent with the design system
  - Line heights that cause text to appear cramped or overly spaced
  - Long text strings (names, descriptions, addresses) that break layouts
- Apply proper text containment: `overflow-hidden`, `text-ellipsis`, `whitespace-nowrap`, or `line-clamp-N` as appropriate for each context.
- Fix every instance found.

### Step 4.3 — Card & Container Rendering Audit
- Check EVERY card component for:
  - Cards that are too wide (exceeding their grid column)
  - Cards that are too narrow (text wrapping excessively or being cut off)
  - Inconsistent card heights in grid layouts (missing equal-height handling)
  - Card content overflowing card boundaries
  - Inconsistent internal padding between cards
  - Missing or inconsistent borders/shadows
  - Grid layouts that break at certain content lengths
- Ensure all cards use proper `min-width`, `max-width`, and `flex`/`grid` properties for responsive containment.
- Fix every instance found.

### Step 4.4 — Badge & Icon Audit
- Check EVERY badge component for:
  - Badges that are oversized relative to their container
  - Badges that are squashed or distorted (aspect ratio issues)
  - Badge stacking that is misaligned or overlapping incorrectly
  - Badge text that overflows the badge container
  - Inconsistent badge sizing across different contexts
  - Hero badges that are disproportionate to their section
- Check EVERY icon for:
  - Inconsistent sizing (icons should follow a consistent size scale)
  - Icons that are too large or too small for their context
  - Icons that are misaligned with adjacent text
  - Missing icons or broken icon references
- Fix every instance found.

### Step 4.5 — ChatDock Desktop Audit (Facebook Messenger Style)
- Review the ChatDock component structure and verify it matches Facebook Messenger patterns:
  - Left sidebar: conversation list with search, filters (groups, condos, DMs), unread indicators, last message preview, timestamp, user avatars
  - Main panel: active conversation with message bubbles (sent vs received styling), timestamps, read receipts, typing indicators
  - Right panel (if applicable): group/condo info, member list, shared media
  - Message input: text area with expand, emoji picker, attachment button, send button
  - Proper conversation switching without flicker or layout shift
- Verify visual polish:
  - Consistent bubble styling (border-radius, padding, max-width)
  - Proper avatar rendering and alignment
  - Smooth scroll behavior with virtualized list for performance
  - Unread message indicators and notification badges
  - Online/offline status indicators
  - Group chat visual differentiation from DMs
  - Condo channel organization and visual hierarchy
- Fix every issue found.

### Step 4.6 — Header, Menu & Navigation Audit
- Check all navigation elements:
  - Top header bar: proper height, alignment, spacing, branding
  - Side navigation menu: proper width, item spacing, active state highlighting, collapse/expand behavior
  - Breadcrumbs: proper formatting, clickable links, overflow handling
  - Tab bars: proper alignment, active state, content switching without layout shift
- Fix every issue found.

### Step 4.7 — Modal, Dialog, Sheet & Drawer Audit
- For EVERY modal, dialog, sheet, bottom sheet, and drawer in the platform:
  - Verify proper backdrop/overlay (click-outside-to-close, proper z-index)
  - Verify proper padding (content should not touch edges)
  - Verify proper header (title, close button, consistent styling)
  - Verify proper borders and separation between header/body/footer
  - Verify proper max-height with scroll for long content
  - Verify proper width constraints (not too wide, not too narrow)
  - Verify proper animation/transition (smooth open/close)
  - Verify proper focus trapping (tab key stays within modal)
  - Verify consistent styling across ALL modals/dialogs/sheets (no one-offs)
  - Verify nested modals stack properly with increasing z-index
  - Verify body scroll lock when modal is open
- Fix every issue found.

### Step 4.8 — Button & Interactive Element Audit
- Check all buttons and interactive elements:
  - Consistent sizing across the platform (small, medium, large variants)
  - Proper hover, active, disabled, and loading states
  - Text not overflowing button boundaries
  - Icon + text alignment within buttons
  - Sufficient touch/click target size (minimum 44x44px for accessibility)
  - Proper spacing between adjacent buttons
  - Consistent color usage (primary, secondary, danger, ghost variants)
- Fix every issue found.
