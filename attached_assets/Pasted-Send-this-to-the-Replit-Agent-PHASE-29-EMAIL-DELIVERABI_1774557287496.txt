**STEP 2 — SPF RECORD VERIFICATION AND CONFIGURATION**

SPF (Sender Policy Framework) authorizes which servers can send email on behalf of your domain:
- Check current SPF record for the sending domain: `dig TXT [domain]`
- Confirm Resend's sending servers are included in the SPF record
- SPF record syntax validated — no errors, no redundant includes
- SPF record does not exceed the 10 DNS lookup limit — count all includes
- SPF record uses `-all` (hard fail) not `~all` (soft fail) — hard fail is stronger
- SPF record documented in replit.md with the exact record value
- If SPF record does not exist or is incorrect — provide the exact DNS record to add and document it

---

**STEP 3 — DKIM SIGNATURE VERIFICATION AND CONFIGURATION**

DKIM (DomainKeys Identified Mail) signs outbound emails to prove they originated from your domain:
- Check if DKIM is configured for the sending domain in Resend dashboard
- DKIM keys generated and DNS CNAME or TXT records published
- DKIM selector confirmed — document the selector name
- DKIM key length confirmed minimum 2048 bits — not legacy 1024 bit
- DKIM signature verified on a test email — header shows `dkim=pass`
- DKIM records documented in replit.md with the exact record values
- If DKIM is not configured — provide the exact DNS records to add and document them

---

**STEP 4 — DMARC POLICY VERIFICATION AND CONFIGURATION**

DMARC (Domain-based Message Authentication Reporting and Conformance) ties SPF and DKIM together and defines what happens to unauthenticated emails:
- Check current DMARC record: `dig TXT _dmarc.[domain]`
- DMARC policy set to minimum `p=quarantine` — ideally `p=reject` for maximum protection
- DMARC alignment mode confirmed — `aspf=s` (strict) or `aspf=r` (relaxed)
- DMARC reporting configured — `rua=` aggregate report destination and `ruf=` forensic report destination defined
- DMARC record syntax validated — no errors
- DMARC record documented in replit.md with the exact record value
- If DMARC does not exist or is too permissive (`p=none`) — provide the exact record to add and document it

---

**STEP 5 — SENDING DOMAIN REPUTATION CHECK**

The sending domain must not be on any blocklists:
- Check domain reputation against major blocklists: Spamhaus, Barracuda, MX Toolbox, SURBL
- Check IP reputation of Resend's sending servers — confirm not blocklisted
- Check domain age — newly registered domains have lower trust; document the domain age
- Check if the domain has any prior spam history
- Resend deliverability score reviewed if accessible
- Document the current reputation status for each check

**STEP 6 — EMAIL TEMPLATE AUDIT**

Every email template must follow deliverability best practices:
- All templates have a plain text version alongside HTML — not HTML-only
- All templates have a clear unsubscribe link for non-transactional emails
- All templates have a physical mailing address in the footer (CAN-SPAM requirement)
- Subject lines do not contain spam trigger words — review all subject lines
- HTML templates use inline CSS — not external stylesheets (blocked by many email clients)
- Images in templates have alt text — renders correctly even when images are blocked
- Templates tested across major email clients: Gmail, Outlook, Apple Mail — confirm correct rendering
- Templates confirmed rendering in Spanish as well as English (Phase 32 integration point — flag any English-only templates)
- Email-to-text ratio acceptable — not image-heavy with minimal text
- No deceptive subject lines — subject matches email content

**STEP 7 — BOUNCE AND COMPLAINT HANDLING**

Bounced emails and spam complaints must be handled automatically:
- Resend webhook for bounce events configured — hard bounces automatically suppress the address
- Resend webhook for complaint events configured — spam complaints automatically suppress the address
- Suppression list integrated with NDS — NDS checks suppression list before attempting delivery
- Soft bounce retry logic confirmed — soft bounces retry with backoff, hard bounces do not retry
- Bounce rate monitoring — alert fires if bounce rate exceeds 2% threshold
- Complaint rate monitoring — alert fires if complaint rate exceeds 0.1% threshold
- Suppression list management UI — platform staff can view and manage suppressed addresses
- NDS delivery record updated when bounce or complaint webhook fires — status reflects actual delivery outcome
