**SECTION 1 — UNIVERSAL DOCUMENT TEMPLATE ARCHITECTURE**

Create a single source of truth for all document templates.

**Create `server/services/documents/templateRegistry.ts`**

This is the canonical registry for every document template in the platform. Every form in every workflow reads from this registry. Editing a template here updates it everywhere it is used.

```typescript
export interface DocumentTemplate {
  id: string
  documentType: string
  title: string
  version: string
  sections: TemplateSection[]
  universalHeader: boolean  // use the platform universal header
  universalFooter: boolean  // use the platform universal footer
  requiresSignature: boolean
  requiresInitials: boolean[]  // one per section that needs initials
  requiresDateStamp: boolean
  requiresGpsCapture: boolean
  requiresIpCapture: boolean
  captureDeviceInfo: boolean
  allowSaveForLater: boolean
  trinityValidationRules: ValidationRule[]
  completionWebhook: string  // event type to publish on completion
}

export interface TemplateSection {
  id: string
  title: string
  order: number
  fields: TemplateField[]
  requiresAcknowledgment: boolean
  acknowledgmentText?: string
  requiresInitials: boolean
  requiresSignature: boolean
}

export interface TemplateField {
  id: string
  label: string
  type: 'text' | 'email' | 'phone' | 'date' | 'select' | 'checkbox' | 'signature' | 'initials' | 'upload' | 'textarea' | 'ssn' | 'address_block' | 'acknowledgment_check'
  required: boolean
  placeholder?: string
  options?: string[]  // for select fields
  validationPattern?: string
  helpText?: string
  mobileFullWidth: boolean  // always true for mobile
  sensitiveData: boolean  // mask in display, encrypt in storage
}
```

**SECTION 3 — UNIVERSAL FORM RENDERER**

Create `client/src/components/documents/UniversalFormRenderer.tsx`

This single component renders any document template from the registry. It receives a `templateId` and renders the complete form with all sections, fields, navigation, and controls.

**Universal controls that appear on every form:**

Top bar:
- Back button (with confirmation if data entered)
- Document title
- Progress indicator (Section N of N)
- Save for Later button (if template allows it)

Section navigation:
- Previous Section button
- Section title and description
- Field rendering (see Section 4)
- Section acknowledgment if required
- Section initials field if required
- Next Section button (disabled until section is complete)

Bottom action bar (fixed at bottom — never scrolls away):
- Save for Later (saves current progress, sends reminder)
- Back to Previous Section
- Continue to Next Section / Submit

Final submission screen:
- Review summary — all sections with entered data
- Edit button per section
- Signature capture (full signature — not just typed name)
- Date stamp (auto-populated, not editable)
- GPS capture if required (with user permission request)
- IP address capture (automatic, no user action)
- Device info capture (automatic)
- "I certify that all information provided is accurate and complete" acknowledgment checkbox
- Submit button (disabled until all required fields complete and acknowledgment checked)
- After submit: loading state → Trinity validation → success or error with specific missing items

**Progress auto-save:**
- Save form progress to localStorage AND to DB (draft document record) every 30 seconds
- On page load — check for saved draft and offer to continue
- Draft expires after 7 days

**SECTION 4 — FIELD RENDERING COMPONENTS**

Create `client/src/components/documents/fields/` directory with these field components:

**`TextField.tsx`** — standard text input
- Full width on mobile always
- Label above field always (not placeholder-only)
- Error message below field in red
- Character counter if maxLength set
- Minimum height 48px on mobile

**`SignatureField.tsx`** — captures actual drawn signature
- Canvas-based signature pad
- Works with mouse on desktop, touch on mobile
- "Clear" and "Done" buttons
- On mobile — full screen signature capture modal for easier signing
- Stores as base64 PNG
- Displays miniature of captured signature after completion

**`InitialsField.tsx`** — smaller signature for section acknowledgments
- Same canvas approach but compact
- 3-letter initials box
- Tappable on mobile

**`DateField.tsx`** — date input
- Date picker on desktop
- Native date picker on mobile
- Auto-populate with today for stamp fields (not editable when auto-stamped)

**`CheckboxField.tsx`** — acknowledgment checkboxes
- Large tap target (minimum 44px) on mobile
- Clear checked/unchecked visual state
- Label text fully visible — never truncated

**`AcknowledgmentField.tsx`** — scrollable text with acknowledgment checkbox
- Full text displayed in scrollable box
- User must scroll to bottom before checkbox enables
- Checkbox text: "I have read and understand the above"
- Scroll position tracked — checkbox disabled until bottom reached

**`UploadField.tsx`** — document/image upload
- Drag and drop on desktop
- Camera capture option on mobile (for ID photos, guard card photos)
- Front/Back upload pair option for IDs
- Preview thumbnail after upload
- File type and size validation with clear error messages
- Progress indicator during upload

**`AddressBlock.tsx`** — structured address entry
- Street address line 1
- Street address line 2 (optional)
- City, State (dropdown), ZIP
- All fields in one logical group

**`SSNField.tsx`** — sensitive data field
- Masked input — shows XXX-XX-XXXX as typed
- Only last 4 visible after completion
- Never logged in plain text
- Encrypted before transmission

**`SelectField.tsx`** — dropdown selection
- Native select on mobile for best UX
- Custom styled select on desktop
- Search/filter for long option lists

**`GpsStampField.tsx`** — automatic GPS capture
- Requests location permission if not granted
- Shows "Capturing location..." spinner
- Displays city/state after capture (not full coordinates in UI)
- Full coordinates stored in document metadata
- Graceful fallback if location denied — records denial

**`VerificationStamp.tsx`** — auto-populated on submission
- Submission timestamp (UTC and local timezone)
- IP address
- GPS coordinates (if captured)
- Device type (mobile/desktop, browser)
- Session ID for audit trail
- Displayed on final submitted document

**SECTION 6 — TRINITY VALIDATION ON SUBMISSION**

Create `server/services/documents/trinityDocumentValidator.ts`

When any document is submitted — before it is accepted and saved — Trinity runs completeness validation:

```typescript
// 7-Step Validation Pipeline:
// 1. TRIGGER: Document submitted by user
// 2. FETCH: Load document data, load template requirements, load employee/client record
// 3. VALIDATE:
//    - All required fields present and non-empty
//    - Signature fields contain actual signature data (not empty canvas)
//    - Date fields are valid dates
//    - Upload fields have file URLs (uploads completed)
//    - SSN/EIN fields match expected format
//    - Acknowledgment fields confirm scroll completion before check
//    - GPS captured if required
//    - IP address captured
// 4. PROCESS:
//    - Check each required field against template requirements
//    - Build list of missing or invalid fields
//    - Calculate completion percentage
// 5. MUTATE:
//    IF validation passes (completion === 100%):
//      - Set document status to 'completed'
//      - Append to employee/org document safe
//      - Generate final PDF with verification stamps
//      - Update employee onboarding status
//      - Emit document_completed event
//    IF validation fails:
//      - Do NOT save as completed
//      - Return specific list of missing/invalid fields
//      - Highlight each issue for the user
// 6. CONFIRM: Return validation result to frontend
// 7. NOTIFY:
//    IF passed: notify org owner, notify employee, emit to event bus
//    IF failed: show user specific actionable error list
```

**Validation response format:**
```typescript
{
  valid: boolean,
  completionPercentage: number,
  missingFields: Array<{
    sectionTitle: string,
    fieldLabel: string,
    reason: string,
    fieldId: string  // for UI to scroll to and highlight
  }>,
  warnings: Array<{
    message: string,
    fieldId: string
  }>
}
```

Frontend behavior on validation failure:
- Modal showing exact list of missing items
- Click any item to scroll directly to that field
- Each missing field highlighted with red border and error message
- Submit button remains disabled until all issues resolved

**SECTION 9 — MOBILE AND DESKTOP RENDERING**

**Mobile rules (max-width 768px):**
- Every field full width — no side-by-side fields on mobile
- All text minimum 16px — prevents auto-zoom on iOS
- All buttons minimum 48px height
- All tap targets minimum 44x44px
- Signature pad full screen modal on mobile
- Bottom action bar fixed — always visible
- No horizontal scroll anywhere
- Upload area shows camera icon prominently for mobile
- Date fields use native mobile date picker
- GPS capture triggered automatically with permission prompt on mobile
- Back/forward navigation uses full-width buttons

**Desktop rules:**
- Two-column layout where logical (name + last name side by side)
- Sidebar section navigation for long forms
- Signature pad inline (not modal)
- Keyboard shortcuts for navigation

**Full-sheet mobile view:**
- Forms render as full-screen pages on mobile — no sidebars, no modals except signature
- Each section takes full screen — swipe or button to advance
- Progress bar at top showing section completion
- All content scrolls vertically — no tabs or accordions
