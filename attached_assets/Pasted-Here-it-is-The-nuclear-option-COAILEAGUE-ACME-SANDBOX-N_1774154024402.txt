**AGENT 1 — CRUD COMPLETIONIST**
*Specialty: Every create, read, update, and delete operation across every entity in the system*

This agent does not trust that a feature works because it rendered. It executes every CRUD operation directly and traces the result from API call to database write to UI reflection.

Loop through every entity in the system:

Employees — create a new employee record with full profile, pay rate, pay type, position, and role. Read it back from the API. Update every editable field one at a time. Confirm each update writes to the correct DB column and reflects in the UI immediately. Soft delete and confirm the record is hidden from active rosters but retrievable in archived state. Clients — create a new client with address, billing schedule, and client-specific pay rate. Read it back. Update billing schedule and rate. Confirm changes propagate to any open invoices or scheduled shifts associated with that client. Delete and confirm cascade behavior on associated records.

Shifts — create a shift with a specific employee, client, position, start time, and end time. Read it back from the schedule view and the API. Update the shift time and confirm the overlap check fires correctly. Assign a different employee and confirm the original employee's schedule clears. Invoices — create an invoice manually. Add line items. Edit a line item amount and confirm the total recalculates server-side. Change the due date. Update the status through every stage: draft → sent → paid → overdue. Confirm each status transition persists and reflects correctly in the invoice list, aging buckets, and client record.

Payroll entries — create a manual payroll entry. Edit hours. Edit rate. Confirm gross pay recalculates correctly. Approve the entry. Confirm it locks and cannot be edited post-approval without an override. Mark as paid. Confirm pay stub generates and attaches to the employee record.

Documents — upload a document to an employee record. Read it back. Update the document metadata. Delete it and confirm it is removed from storage and the record.

For every CRUD operation above: confirm the write hits the DB, confirm the read returns the written value (not cached state), confirm the UI reflects the DB value after hard refresh, and confirm no other workspace can read the record.

Loop until two consecutive clean passes.

**AGENT 2 — FINANCIAL ACCURACY SPECIALIST**
*Specialty: Every number the platform produces must be mathematically provable*

This agent does not accept "looks right." It proves every figure against independent calculation.

Run the following financial stress tests against ACME sandbox data:

Payroll calculation suite — pull 10 real employee records with varying pay types. For each, manually calculate expected gross pay based on their hours, rate, and pay type. Run the platform's payroll calculation for the same period. Compare output line by line. Flag any discrepancy at any decimal place. Test overtime threshold behavior: employees at exactly 40 hours, at 40.5 hours, and at 60 hours. Confirm overtime multiplier applies at the correct threshold and to the correct hours only.

Pay type matrix — confirm hourly, salary, and contract pay types each calculate correctly. Confirm a salary employee's per-period amount does not fluctuate based on hours logged unless overtime rules apply. Confirm a contract employee's amount is based on the agreed rate, not an hourly derivation.

Invoice calculation suite — pull 5 real client invoices. For each, independently calculate the expected total based on shifts worked, client pay rate, and any applied adjustments. Compare to platform output. Confirm rounding is consistent (never rounds in the platform's favor). Confirm line item totals sum to invoice total with no floating-point drift.

Edge case battery — zero-hour week for an active employee: confirm $0 payroll entry generates cleanly with no errors. Mid-period pay rate change: confirm hours before the change use old rate, hours after use new rate, not blended. Retroactive shift addition to a closed payroll period: confirm the platform either blocks it cleanly or processes a correction entry, never silently modifies a closed run. Duplicate invoice prevention: submit the same invoice generation request twice within 500ms and confirm exactly one invoice is created.

All financial documents — every pay stub, payroll summary, and invoice PDF generated during this pass must be opened and verified: figures match the records, employee/client names are correct, org branding is present, no placeholder text remains, and the document saves to the correct record and persists after session reset.

Loop until two consecutive clean passes.

**LOOP EXECUTION PROTOCOL**

Every agent runs its full domain pass. At the end of each pass, each agent reports its finding count to the shared findings bus. If finding count is zero, the agent resets and runs a second full pass from a different entry point. If the second pass also returns zero findings, the agent is certified clean and stands down. If any pass produces findings, the agent fixes, notifies affected peer agents, and restarts its loop from the beginning. No agent is certified until two consecutive zero-finding passes are confirmed.

---

**FINDINGS BUS SCHEMA**

Every issue logged must include:

- **Issue ID** — sequential, prefixed by agent number
- **Severity** — P0 (financial/security/data loss), P1 (functional break), P2 (data inconsistency or sync gap), P3 (UX/cosmetic), P4 (legal/compliance)
- **Domain** — which agent owns it
- **Location** — exact file, endpoint, component, DB table, or UI surface
- **Description** — what is broken, missing, or inconsistent and the precise failure condition
- **Reproduction steps** — exact sequence that produces the failure in the ACME sandbox
- **Cascading notifications** — which peer agents were triggered and what they found
- **Fix applied** — exact change made with file and line reference, or architectural prescription if a larger change is required
- **Peer verification** — each notified agent confirms their domain is clean after the fix before the issue is closed
