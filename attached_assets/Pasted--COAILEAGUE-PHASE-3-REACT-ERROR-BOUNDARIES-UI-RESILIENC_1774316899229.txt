**SHORTCUT PATTERNS THAT WILL BE REJECTED — READ EVERY ONE**

These exact shortcuts have been attempted in prior phases and rejected every time. They will be rejected again here without discussion.

**REJECTED — Wrapping instead of fixing.** If a component crashes when data is null the fix is an ErrorBoundary at the surface level combined with proper null-safe data access throughout the component. A try-catch wrapper that swallows the crash and renders nothing is not a fix. It is a hidden crash with a clean exterior.

**REJECTED — Annotating instead of converting.** If a useEffect is missing a cleanup function the fix is adding the cleanup function. A comment saying cleanup is not needed must be supported by reading the effect, understanding what persistent resource it creates, and proving that resource does not need to be torn down. Generic annotations on effects that create intervals, WebSocket connections, or event listeners without cleanup are not acceptable.

**REJECTED — Partial surface coverage.** If ErrorBoundary wrapping is required on every major UI surface, wrapping five and noting the other twelve is not Phase 3 complete. Every surface in the required list is wrapped before Phase 3 closes. The list is the complete work order not a sample.

**REJECTED — Single root-level ErrorBoundary.** Wrapping the entire application in one ErrorBoundary at the root of App.tsx is not this phase's requirement. A root-level boundary means a crash in the schedule board takes down the payroll dashboard, the Trinity chat, and everything else simultaneously. Surface-level boundaries contain crashes to the affected surface only. The rest of the application remains functional.

**REJECTED — Happy path only components.** A component that only renders when data is present with no loading state, no empty state, and no error state is not production ready. Every component that receives async data must handle all four states. Missing states are added now. Not noted for later.

**REJECTED — Subagent batch delegation without verification.** Subagents introduced phantom tables, broken imports, and circular references in Phase 1. If subagents are used they receive one file at a time and their output is verified against the live server before proceeding. Batch delegation across multiple files without verification is not permitted.

**REJECTED — Script-generated fixes.** No script automatically adds ErrorBoundary wrappers. No script auto-generates cleanup functions. No script auto-generates loading and error states. These require reading the component, understanding what it renders, and writing the correct implementation for that specific component.

**REJECTED — Hardcoded fallback values.** Phase 2 RC6 caught Trinity returning hardcoded `{score: 100, complaints: 0, praise: 0}` as fake data. The same standard applies here. A component that returns a hardcoded value instead of handling a null or error state is producing fabricated UI. Return null, render an empty state, or surface the error. Never fake it.

**STEP 1 — CREATE THE CANONICAL ERRORBOUNDARY COMPONENT**

Create this component once. Every surface in this phase imports from this one location. It is never duplicated. Never reimplemented elsewhere.

Create `client/src/components/shared/ErrorBoundary.tsx`:

```typescript
import React, { Component, ErrorInfo, ReactNode } from 'react';

interface Props {
  children: ReactNode;
  fallback?: ReactNode;
  componentName?: string;
  onError?: (error: Error, errorInfo: ErrorInfo) => void;
}

interface State {
  hasError: boolean;
  error: Error | null;
}

export class ErrorBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    console.error(
      `[ErrorBoundary] ${this.props.componentName ?? 'Unknown'} crashed:`,
      error,
      errorInfo
    );
    this.props.onError?.(error, errorInfo);
  }

  handleRetry = () => {
    this.setState({ hasError: false, error: null });
  };

  render() {
    if (this.state.hasError) {
      if (this.props.fallback) {
        return this.props.fallback;
      }
      return (
        <div className="flex flex-col items-center justify-center p-8 
          rounded-lg border border-red-200 bg-red-50 min-h-[200px]">
          <div className="text-red-600 font-semibold text-lg mb-2">
            Something went wrong
          </div>
          <div className="text-red-500 text-sm mb-4 text-center max-w-md">
            {this.props.componentName
              ? `The ${this.props.componentName} encountered an error 
                 and could not display.`
              : 'This section encountered an error and could not display.'
            }
          </div>
          <button
            onClick={this.handleRetry}
            className="px-4 py-2 bg-red-600 text-white rounded 
              hover:bg-red-700 text-sm"
          >
            Try Again
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}
```

Verify this component compiles with zero TypeScript errors. Import it in one test component and confirm it renders correctly in the ACME sandbox before proceeding to Step 2. This is the foundation every subsequent step uses. Do not proceed if it does not compile and render cleanly.

**STEP 2 — CREATE THE CANONICAL USEASYNCDATA HOOK**

Create `client/src/hooks/useAsyncData.ts`. This hook wraps every React Query result and provides a consistent interface for all four states every async data component must handle. Every component that currently accesses `data?.something` directly from a React Query result must use this hook after its surface is wrapped in Step 3.

```typescript
import { UseQueryResult } from '@tanstack/react-query';

export interface AsyncDataState<T> {
  data: T | undefined;
  isLoading: boolean;
  isEmpty: boolean;
  isError: boolean;
  error: Error | null;
  refetch: () => void;
}

export function useAsyncData<T>(
  queryResult: UseQueryResult<T>,
  isEmpty?: (data: T) => boolean
): AsyncDataState<T> {
  const { data, isLoading, isError, error, refetch } = queryResult;

  const emptyCheck = isEmpty
    ? (data !== undefined && isEmpty(data))
    : (Array.isArray(data) && data.length === 0);

  return {
    data,
    isLoading,
    isEmpty: !isLoading && !isError && emptyCheck,
    isError,
    error: isError ? (error as Error) : null,
    refetch
  };
}
```

Verify this hook compiles with zero TypeScript errors. This is the canonical data access pattern going forward. It is imported from this one location everywhere it is used.

**STEP 4 — FIX EVERY USEEFFECT MISSING A CLEANUP FUNCTION**

Scan every component file in `client/src`. Find every `useEffect` call. For each one read the effect body completely. Identify every persistent resource it creates.

These effect patterns require a cleanup function — no exceptions:

```typescript
// setInterval — must clear
useEffect(() => {
  const interval = setInterval(() => refetch(), 30000);
  return () => clearInterval(interval);
}, []);

// setTimeout — must clear
useEffect(() => {
  const timeout = setTimeout(() => setVisible(false), 3000);
  return () => clearTimeout(timeout);
}, []);

// addEventListener — must remove
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

  ws.onmessage = handleMessage;
  return () => ws.close();

  const sub = eventBus.subscribe('shift-update', handleUpdate);
  return () => sub.unsubscribe();
}, []);
```

```typescript
const registeredIntervals: NodeJS.Timeout[] = [];

// When registering any background interval store the reference
const payrollScanInterval = setInterval(runPayrollScan, 60000);
registeredIntervals.push(payrollScanInterval);

// Graceful shutdown clears everything
const shutdown = async () => {
  console.log('[Shutdown] Graceful shutdown initiated');
  
  // Clear all background intervals
  registeredIntervals.forEach(interval => clearInterval(interval));
  
  // Close database pool
  await pool.end();
  
  console.log('[Shutdown] Clean exit');
  process.exit(0);
};

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```
