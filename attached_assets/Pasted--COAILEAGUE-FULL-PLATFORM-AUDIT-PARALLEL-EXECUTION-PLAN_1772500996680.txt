# COAILEAGUE FULL PLATFORM AUDIT — PARALLEL EXECUTION PLAN

## INSTRUCTIONS FOR REPLIT AGENT

This is a comprehensive platform-wide audit. Every system, every workflow, every pipeline, every feature must be verified. Nothing is assumed working. Everything is proven or flagged.

**METHODOLOGY:**
1. For each audit section, investigate FIRST — read code, query database, trace workflows
2. Map ALL issues before fixing ANY of them
3. Present findings in the gap report format: GAP-ID, Severity, Location, Impact, Fix, Dependencies
4. Do NOT fix anything until the full audit is presented and confirmed
5. Severity levels: CRITICAL (blocks launch), HIGH (degrades core feature), MEDIUM (impacts quality), LOW (tech debt)

**PARALLEL EXECUTION:** Run these audit sections simultaneously. Each section is independent and can be investigated in parallel. Report findings per section as they complete — don't wait for all sections to finish.

# SECTION 1: DATABASE FOUNDATION AUDIT
# ═══════════════════════════════════════════════

## 1A: Schema-to-Code Alignment

For EVERY table in shared/schema.ts:
- Does the Drizzle schema definition match the actual Postgres table? (columns, types, defaults, constraints)
- Run: `SELECT column_name, data_type, is_nullable, column_default FROM information_schema.columns WHERE table_name = '<table>'` for every table
- Compare against the Drizzle schema definition
- Flag ANY mismatch: missing columns, wrong types, wrong defaults, columns in DB not in schema, columns in schema not in DB
- Pay special attention to enum types — does every Postgres enum match the Drizzle enum definition exactly? List each enum and compare values.

## 1B: Foreign Key Integrity

- List every foreign key constraint in the database
- For each FK: query for orphaned records (child records pointing to non-existent parent records)
- Critical tables to check: 
  - shifts.employee_id → employees.id
  - shifts.client_id → clients.id  
  - time_entries.shift_id → shifts.id
  - time_entries.employee_id → employees.id
  - time_entries.invoice_id → invoices.id
  - invoice_line_items.invoice_id → invoices.id
  - payroll_entries.payroll_run_id → payroll_runs.id
  - payroll_entries.employee_id → employees.id
  - org_ledger entries referencing invoices/payroll that exist
  - approval_gates.workspace_id → workspaces.id
  - All chat messages → chatroom references
  - All notifications → user/workspace references
- Flag every orphaned record with table name, count, and example IDs

## 1C: NULL Fields That Should Never Be NULL

Query every table for NULL values in fields that should be required:
- employees: hourly_rate, overtime_rate, name, email, workspace_id, employment_type
- clients: name, workspace_id, contract_rate (for active clients)
- shifts: start_time, end_time, client_id, workspace_id, status
- time_entries: captured_pay_rate, captured_bill_rate (for approved/billed entries)
- invoices: total, subtotal, workspace_id, client_id, status
- payroll_runs: workspace_id, status
- payroll_entries: gross_pay, net_pay, employee_id (for processed entries)
- org_ledger: amount, entry_type, workspace_id
- workspaces: workspace_type, name

## 1D: Duplicate Record Detection

- Check for duplicate employees (same name + same workspace)
- Check for duplicate clients (same name + same workspace)
- Check for duplicate shifts (same client + same employee + same start_time + same workspace)
- Check for duplicate time entries (same shift_id + same employee_id)
- Check for duplicate invoice line items (same invoice_id + same time_entry_id)
- Check for duplicate ledger entries (same related_entity_id + same entry_type + same amount + created within 1 second)

## 1E: Enum Consistency

For every enum in the schema:
- List all values defined in Drizzle
- List all values in Postgres
- Search the ENTIRE codebase for string literals that should be enum values
- Flag any code using a string value that doesn't exist in the enum (like the 'open' status bug that was found earlier)
- Key enums to check: shift_status, invoice_status, payroll_status, employment_type, notification_type, coverage_type, workspace_type

# SECTION 2: TRINITY ORCHESTRATION AUDIT
# ═══════════════════════════════════════════════

## 2A: Scheduling Pipeline

- Trace the COMPLETE path from client demand → shift generation → employee assignment
- Verify shift generator reads actual client fields (minimumStaffing, coverageType, coverageStart, coverageEnd, scheduleDays)
- Verify the 12-hour max shift duration cap works for ALL coverage types (custom, 24_7, business_hours)
- Verify demand cap: shifts created per slot ≤ client's minimumStaffing
- Verify dedup: running the generator twice produces zero new shifts if nothing changed
- Verify employee scoring: distance, rate, availability, OT awareness, rest compliance, client complaints
- Verify availability constraints are enforced (employee with Mon-Fri availability never assigned to Saturday)
- Verify OT threshold: no employee assigned beyond 40 hours without explicit override
- Verify rest compliance: minimum 8-hour gap between shifts
- Verify the daemon cron schedule (when does it run? does it run reliably?)

## 2B: ShiftCompletionBridge

- Trace the COMPLETE path from shift end_time → bridge detection → time entry creation → auto-approve → event emission
- Verify the bridge only processes active, non-suspended, non-frozen workspaces
- Verify the bridge creates time entries with captured_pay_rate and captured_bill_rate (never NULL)
- Verify auto-approve respects the workspace's autoInvoicingEnabled setting
- Verify the time_entries_approved event fires with correct payload
- Verify the bridge handles edge cases: employee didn't clock in (no-show), shift was cancelled after starting, shift spans midnight

## 2C: Billing Pipeline

- Trace the COMPLETE path from approved time entries → invoice generation → auto-send → payment tracking → ledger
- Verify atomic claim pattern: UPDATE billed_at THEN create invoice (never reverse order)
- Verify no double-billing: run billing twice on same entries, second run should find zero unbilled entries
- Verify tax calculation: correct state tax rate applied, tax_exempt clients get zero tax
- Verify invoice auto-send: when auto_send is enabled, invoices transition to 'sent' not stuck in 'draft'
- Verify invoice line items: each line references the correct time entry, hours, and rate
- Verify payment recording: mark-as-paid updates invoice status, creates ledger entry, records reference number
- Verify partial payments: can an invoice be partially paid? How is the balance tracked?
- Verify credit memos / adjustments: if an invoice needs to be corrected after sending, what happens?

## 2D: Payroll Pipeline

- Trace the COMPLETE path from approved time entries → payroll run creation → approval gate → processing → disbursement
- Verify W2 calculations: regular hours (≤40) at base rate + OT hours (>40) at 1.5× rate
- Verify 1099 calculations: all hours at flat rate, no withholding
- Verify tax withholdings for W2: federal, state, FICA, Medicare (are the rates correct and current?)
- Verify the approval gate is MANDATORY: there must be no code path to process payroll without an approved gate
- Verify gate persistence: create a gate, restart the server, verify the gate still exists and can be approved
- Verify disbursement: after approval, does the Stripe Connect payout code execute for 1099 contractors?
- Verify W2 payroll export: is there integration with ADP/Gusto/Paychex?
- Verify the payroll-to-ledger write: every processed payroll creates a ledger entry

## 2E: Manual Edit Propagation

- Edit a time entry's hours → verify the linked invoice line item recalculates
- Edit a time entry's hours → verify the linked payroll entry recalculates
- Edit a shift's employee assignment → verify downstream records update
- Edit a client's contract_rate → verify future shifts use the new rate (existing shifts keep captured rate)
- Edit an employee's hourly_rate → verify future shifts use the new rate (existing shifts keep captured rate)
- For every edit: verify manually_edited flag is set, pre_edit_snapshot is saved, audit trail is complete

## 2F: Autonomous Settings Persistence

- Check where EACH of these settings is stored:
  - Auto-fill shifts toggle
  - Auto-send invoices toggle
  - Auto-approve time entries toggle
  - Auto-process billing toggle
  - Trinity autonomous mode master toggle
- For each: is it in React state (BUG — lost on refresh), localStorage (BUG — lost on device change), or database (CORRECT)?
- Fix any that aren't database-persisted
- Verify: toggle ON → refresh page → still ON. Toggle OFF → refresh → still OFF.
- Verify: Trinity's daemon reads these settings from DB, not from any frontend state

# ═══════════════════════════════════════════════

## 7A: Race Conditions

- Double-click "Fill Shifts" button rapidly → does it create duplicate shifts?
- Submit the same time entry twice quickly → does it create duplicates?
- Two managers approve the same payroll gate simultaneously → does it process twice?
- Mark an invoice as paid twice quickly → does it create duplicate payment records or ledger entries?
- Two employees claim the same open shift simultaneously → does it assign both?
- Rapid WebSocket events → do they process in order or can they interleave?

For each race condition: what is the current protection? (DB constraint, mutex, optimistic locking, none?)

## 7B: Authorization & Access Control

- Can an employee see another org's data?
- Can an employee access manager-only endpoints?
- Can a manager in Org A see Org B's invoices/payroll?
- Are all API endpoints checking workspace membership?
- Are financial endpoints (payroll, billing) restricted to owner/manager roles?
- Can a suspended workspace still process financial operations?

## 7C: Data Validation

- Create a shift with end_time before start_time → does the system reject it?
- Create a time entry with negative hours → rejected?
- Create an invoice with negative amount → rejected?
- Set an employee rate to $0 → rejected or allowed?
- Set a client contract_rate to $0 → rejected or allowed?
- Enter a pay_rate higher than bill_rate → warning? (Marcus Rivera issue)
