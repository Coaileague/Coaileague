THE CORRECT CIRCUIT BREAKER DESIGN:
A circuit breaker for a database connection should have three states:
CLOSED (normal operation): All queries pass through. Failures are counted. If failures exceed threshold within a time window the circuit opens.
OPEN (DB unavailable): Queries that can be served from cache are served from cache. Queries that cannot are rejected with a specific error code — not a generic 500. The circuit attempts a probe query every 10 seconds (not 30) to check if DB has recovered.
HALF-OPEN (testing recovery): One probe query allowed through. If it succeeds the circuit closes. If it fails the circuit stays open and resets the probe timer.
Current problems to fix:
Problem 1 — Failure threshold too low:
The circuit is opening after too few failures. At Replit startup the DB connection pool initializes and the first 1-2 queries may fail as the pool warms up. This should not open the circuit. Set failure threshold to 5 failures within a 60-second window before opening.
Problem 2 — Reset timeout too long:
30 seconds of circuit-open state means 30 seconds of white screen for every user. Set probe interval to 10 seconds.
Problem 3 — Auth queries not handled specially:
/api/auth/me must never return a 500 when the circuit is open. If the circuit is open and a valid session exists in the session store return the session data without hitting the DB. The session store (Redis or memory) should be the source of truth for auth status when DB is temporarily unavailable.
Problem 4 — WebSocket auth fails when circuit is open:
The ws_authenticate handler queries the DB for user validation. When the circuit is open this throws and disconnects the WebSocket. Fix: if circuit is open and session token is valid return the cached session user without DB query.
Problem 5 — No exponential backoff on reconnection:
The connection pool should use exponential backoff on reconnection attempts: 1s, 2s, 4s, 8s, 16s, max 30s. Not a fixed 30-second wait.
