1. Automated Error Recovery Logic
Action: Program the Orchestrator to monitor all tool outputs. If a non-LLM tool returns an error (e.g., calculate_distance fails due to a Maps API connection error), Lomu must have predefined Recovery Protocols.

Protocol Example: If the Maps API fails, Lomu's logic should execute a Fallback Protocol: "Temporarily assign based only on ZIP code distance and flag the shifts with a lower Confidence Score, alerting the human for review." The process continues without halting the entire schedule.

Why: Ensures high uptime and reliability by preventing external API failures from crashing the entire scheduling run.

2. Transactional Integrity and Rollback
Action: The entire schedule run must be treated as a database transaction. If the final database write or commit fails, the system must trigger a compensating transaction that clears all temporary data and reverts any partial changes.

Why: Guarantees that the system is always in a known, clean state, preventing data corruption and "ghost" assignments.
