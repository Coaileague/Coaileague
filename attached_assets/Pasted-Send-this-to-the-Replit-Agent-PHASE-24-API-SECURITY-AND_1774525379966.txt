**STEP 2 — AUTHENTICATION HARDENING**

Every authentication mechanism must be airtight:
- Session tokens: secure, httpOnly, sameSite=strict cookies confirmed — no tokens in localStorage
- Session expiry enforced — idle sessions expire after defined timeout
- Session fixation prevented — new session ID generated on login
- Magic link tokens: single-use confirmed, expiry enforced, workspace-scoped
- Portal tokens: expiry enforced, workspace-scoped (regression check from Phase 18)
- Auditor tokens: expiry enforced, workspace-scoped (regression check from Phase 20)
- Trinity bot token: never in logs or API responses (regression check from Phase 19)
- Password requirements enforced: minimum length, complexity, breach database check
- Password reset flow: token single-use, expiry enforced, old password invalidated on reset
- Account lockout after N failed login attempts — brute force protection confirmed
- Multi-factor authentication available for org_owner and platform staff roles
- OAuth tokens (QuickBooks): encrypted at rest (regression check from Phase 16)

---

**STEP 3 — RATE LIMITING AUDIT**

Every endpoint must have appropriate rate limiting:
- Auth endpoints (login, register, password reset, magic link): strictest limits — maximum 5 attempts per minute per IP
- Financial endpoints (invoice, payroll, billing): strict limits — maximum 30 requests per minute per workspace
- Portal endpoints (client portal, officer portal): moderate limits — maximum 60 requests per minute per token
- General API endpoints: standard limits — maximum 100 requests per minute per workspace
- Trinity/AI endpoints: limits per workspace per minute — prevent runaway AI costs
- Inbound webhook endpoints: rate limited per source IP
- Export endpoints (CSV, PDF): limits preventing bulk data scraping
- Rate limit responses return 429 with Retry-After header — not 403 or 500
- Rate limit state stored in Redis or equivalent — not in-memory only
- Rate limits configurable per workspace tier — enterprise workspaces get higher limits
- Rate limit violations logged — repeated violations trigger admin alert

---

**STEP 4 — INPUT VALIDATION AND SANITIZATION**

Every input must be validated before processing:
- All route handlers have Zod schema validation on request body — no raw `req.body` access
- All URL parameters validated — no raw `req.params.id` passed directly to DB queries
- All query parameters validated — no raw `req.query` used in DB queries without validation
- SQL injection surface reviewed — all DB queries use parameterized queries via Drizzle ORM — no raw SQL string concatenation
- XSS prevention confirmed — all user-supplied content escaped before rendering
- HTML injection prevented — no `dangerouslySetInnerHTML` with user content
- File upload validation: file type whitelist enforced, file size limits enforced, filename sanitized
- JSON payload size limits enforced — no unbounded request body sizes
- Numeric inputs validated — no NaN, no negative values where prohibited, no values exceeding business logic limits
- Date inputs validated — no invalid dates, no dates outside acceptable ranges

---

**STEP 5 — SECURITY HEADERS AUDIT**

Every HTTP response must include correct security headers:
- `Strict-Transport-Security` (HSTS): max-age minimum 1 year, includeSubDomains confirmed
- `Content-Security-Policy` (CSP): defined and enforced — no wildcard sources
- `X-Frame-Options`: DENY or SAMEORIGIN — clickjacking prevented
- `X-Content-Type-Options`: nosniff confirmed
- `Referrer-Policy`: strict-origin-when-cross-origin or stricter
- `Permissions-Policy`: camera, microphone, geolocation permissions scoped correctly for field operations use
- `Cache-Control`: sensitive endpoints return no-store, no-cache
- No server version disclosure — `X-Powered-By` header removed
- No unnecessary headers exposing internal infrastructure details
- HTTPS enforced — HTTP requests redirected to HTTPS

---

**STEP 6 — CORS CONFIGURATION AUDIT**

CORS must be restrictive and correct:
- Allowed origins explicitly defined — no wildcard `*` on authenticated endpoints
- Allowed methods restricted to what each endpoint needs — not blanket allow all
- Credentials flag correct — `Access-Control-Allow-Credentials: true` only where explicitly needed
- Preflight caching defined — `Access-Control-Max-Age` set appropriately
- CORS configuration not bypassed by any endpoint — confirm no manual CORS header overrides
- Inbound webhook endpoints have CORS appropriate to their source

**STEP 8 — AUTHORIZATION AUDIT**

Authorization must be enforced at every layer:
- Every route confirmed requiring authentication — no accidentally public endpoints
- Every route confirmed having correct role guard — no under-permissioned or over-permissioned endpoints
- Workspace isolation confirmed on every query — no missing workspaceId scopes (regression check from sync audit)

**STEP 9 — PENETRATION TEST SIMULATION**

Run targeted security tests against the platform:

**Test 1 — Brute force protection:** Attempt 10 rapid login requests with wrong password. Confirm account lockout or rate limit fires after defined threshold.

**Test 2 — SQL injection attempt:** Submit a crafted payload `'; DROP TABLE users; --` in a text input field. Confirm Drizzle ORM parameterization prevents execution.

**Test 3 — XSS attempt:** Submit `<script>alert('xss')</script>` in a text field. Confirm it is escaped and not executed.

**Test 4 — IDOR attempt:** Authenticate as user A, then attempt to access a resource belonging to user B by manipulating the ID in the URL. Confirm 403 or 404 returned.

**Test 5 — Role escalation attempt:** As an officer-role user, attempt to call a manager-only endpoint directly. Confirm 403 returned.

**Test 6 — Token reuse attempt:** Use an expired portal token or magic link. Confirm 401 or 403 returned with clear message.

**Test 7 — Rate limit test:** Send 100 rapid requests to a rate-limited endpoint. Confirm 429 returned with Retry-After header after threshold exceeded.

**Test 8 — Webhook replay attack:** Submit the same Stripe webhook payload twice. Confirm idempotency prevents double-processing (regression check from Phase 11).

All 8 tests must pass before Phase 24 closes.
