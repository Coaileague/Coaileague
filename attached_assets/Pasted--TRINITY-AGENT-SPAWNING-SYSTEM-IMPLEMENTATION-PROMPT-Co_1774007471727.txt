PHASE 3 — TRINITY AGENT SPAWNING ENGINE
═══════════════════════════════════════════════════════════════

Create ONE new file: lib/trinity/agent-spawner.ts
(or match the exact directory and naming convention found 
in the scan for Trinity's orchestration layer)

This file contains the complete spawn/recall/evaluate cycle.
Do not split across multiple files unless existing patterns 
require it.

Export the following functions:

─────────────────────────────────────────────
function spawnAgent(params: {
  workspaceId: string
  agentKey: string
  taskType: string
  inputPayload: Record<string, unknown>
  relatedEntityType?: string
  relatedEntityId?: string
  spawnedBy?: string
}): Promise<AgentTask>
─────────────────────────────────────────────
Logic:
1. Look up agent from agent_registry by workspaceId + agentKey
   — if not found, throw error and log to agent_task_logs
2. Create agent_tasks record with status = 'pending'
3. Log spawn event to agent_task_logs
4. Build the scoped API call:
   - System prompt = agent.system_prompt
   - User message = JSON.stringify(inputPayload) with instruction 
     to return ONLY valid JSON matching the output_schema, 
     no preamble, no markdown
5. Call the AI completion (use the same AI client already used 
   in Trinity's existing orchestration — do not introduce a 
   new client or new model)
6. Parse the response — strip any markdown fences if present
7. Validate response matches output_schema structure 
   — if invalid, mark task failed, log error, return task
8. Update agent_tasks: 
   status = 'complete', output_payload = parsed response, 
   completed_at = now
9. Log return event to agent_task_logs
10. Immediately call evaluateAgentPayload(task.id)
11. Return updated task record

─────────────────────────────────────────────
function evaluateAgentPayload(
  taskId: string
): Promise<AgentTask>
─────────────────────────────────────────────
Logic:
1. Fetch agent_tasks record by taskId with related agent_registry
2. Extract completion_score from output_payload 
   (agent is required to self-report score in output schema)
3. Extract confidence_level from output_payload
4. Extract flags array from output_payload
5. Compare completion_score against agent.completion_criteria.min_score

   IF score >= min_score AND no critical flags:
   - evaluation_result = 'approved'
   - Log approval to agent_task_logs
   - Call notifyTrinityApproval(task) to return payload 
     to Trinity's main orchestration thread

   IF score >= min_score BUT critical flags present:
   - evaluation_result = 'escalated_to_helpai'
   - Pass payload to HelpAI for secondary evaluation 
     using existing HelpAI interface pattern found in scan
   - Log escalation to agent_task_logs

   IF score < min_score AND retry_count < max_retries:
   - Increment retry_count
   - status = 're_tasked'
   - Log retry to agent_task_logs
   - Call spawnAgent again with same params 
     (agent re-tasks automatically)

   IF score < min_score AND retry_count >= max_retries:
   - evaluation_result = 'escalated_to_management'
   - Send platform notification to workspace management 
     using existing notification pattern found in scan:
     "Trinity was unable to complete [task_type] automatically. 
     Manual review required. [View Task]"
   - Log escalation to agent_task_logs

6. Write trinity_evaluation: short plain-English summary 
   of what the agent returned and why it was approved/denied
7. evaluated_at = now
8. Return updated task record

─────────────────────────────────────────────
function spawnParallelAgents(params: {
  workspaceId: string
  tasks: Array<{
    agentKey: string
    taskType: string
    inputPayload: Record<string, unknown>
    relatedEntityType?: string
    relatedEntityId?: string
  }>
  spawnedBy?: string
}): Promise<AgentTask[]>
─────────────────────────────────────────────
Logic:
1. Run all spawnAgent calls via Promise.all (true parallel)
2. Return array of all completed and evaluated task records
3. Log parallel spawn event noting how many agents were spawned

─────────────────────────────────────────────
function notifyTrinityApproval(task: AgentTask): void
─────────────────────────────────────────────
Logic:
1. This function is called when a payload is approved
2. It emits an internal event or calls back into Trinity's 
   orchestration dispatch with:
   - task.agent_key
   - task.output_payload
   - task.related_entity_type
   - task.related_entity_id
3. Trinity's orchestration layer receives it and continues 
   its main flow with the approved payload
4. Use whatever event/callback pattern already exists in 
   Trinity's orchestration — do not introduce new patterns

  taskId: string
  agentKey: string
  outputPayload: Record<string, unknown>
  completionScore: number
  flags: unknown[]
}): Promise<HelpAIEvaluation>

Logic:
1. HelpAI reviews the payload independently
2. Returns: { verdict: 'approve' | 'deny' | 'escalate_to_management', 
   reasoning: string, adjusted_score: number }
3. If approve → call notifyTrinityApproval with the task

6. Log all decisions to agent_task_logs

This makes HelpAI the QA layer between agent output and 
management escalation — reducing noise to management by 
catching borderline cases before they become human tickets.
