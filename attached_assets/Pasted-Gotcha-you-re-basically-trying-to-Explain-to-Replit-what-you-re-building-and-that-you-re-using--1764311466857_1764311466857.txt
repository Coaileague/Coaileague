> Project Overview
This project is an automation platform that uses Google Gemini as the primary AI engine for:

Workflow automation

Guard rails and safety guidelines

Tool use orchestration

Dynamic content generation (emails, onboarding flows, etc.)


Core Use Case
We have a Sales/Marketing onboarding pipeline for organizations. The pipeline:

1. Invites new organizations to sign up.


2. Helps migrate their info and configure a new workspace for their org.


3. Gets them “out-of-the-box ready” with minimal setup; end users can start using automation quickly.



AI-Generated Marketing & Emails

The platform uses Gemini to generate HTML marketing emails and campaigns.

AI can design unique content per org, including multiple variants for A/B testing.

Emails are sent automatically as part of the onboarding / invite flow.


Trial & Billing Model

Organizations are invited to a 14-day free trial.

During the 14 days, they can try any tier or automation feature.

Credits used during the trial are tracked, but not billed immediately.

At the end of the trial, if the org chooses a paid tier and activates a subscription (monthly or yearly), then:

They can pay by credit card or, if available, bank ACH.

Credits consumed during the trial are deducted from their new tier’s balance.

This makes the trial fair: whatever they “spent” in credits during the trial simply comes out of their first paid credit allocation rather than being lost.



RBAC & Universal Dashboard Page

We need a single universal page in the workspace that is shown based on RBAC (Role-Based Access Control).

This page should surface all the important onboarding and billing info depending on the user’s role (org leader, standard user, support staff, or my own admin role).

This page will live inside the org’s workspace and control access to: onboarding status, trial status, credits usage, billing, and invites.

2. RBAC Design for the “Universal Page”

Let’s define some roles and what the universal page shows for each.

Roles

Owner / Org Leader (ORG_OWNER)
The main decision-maker; can see everything for their organization.

Billing Admin (BILLING_ADMIN)
Can manage payment methods, subscriptions, and invoices.

Support / Internal Admin (You) (PLATFORM_SUPPORT)
You + your support team; can see cross-org details for troubleshooting.

Org Member / End User (ORG_USER)
Regular user in the workspace; minimal access.

Invited but not yet activated (INVITED_USER)
Someone who got an invite but hasn’t finished onboarding.

3. Trial & Credits Logic (you can use this as dev notes)

Basics:

trial_days = 14

trial_start set on org’s first activation.

trial_end = trial_start + 14 days

credits_used_during_trial tracked in a separate bucket.


When trial ends OR user activates early:

1. User selects a tier: Starter, Pro, etc.


2. Each tier has monthly_credit_allocation (or yearly).


3. On first billing cycle:

allocated_credits = tier.monthly_credit_allocation

available_credits = allocated_credits - credits_used_during_trial

If credits_used_during_trial > allocated_credits, system can:

Either allow a negative starting balance and require purchase of additional credits, or

Force them into a higher tier / additional pack before completing checkout (your choice).





You could model this in pseudo-code as:

availableCredits = planMonthlyCredits - trialCreditsUsed;
if (availableCredits < 0) {
  // handle overage: prompt user to buy extra credits or auto-add an add-on pack
}
